Topic 11

1 Application Insights & Azure Monitor

① Key Concepts

What is Application Insights?

Application Insights (AI) is an Application Performance Management (APM) service built on top of Azure Monitor. It collects telemetry from your application, stores it in a Log Analytics Workspace (workspace-based mode) or its own proprietary data store (classic mode), and provides querying, dashboards, alerting, and smart detection on top of that data.

💡
Workspace-based vs Classic — always choose workspace-based for new resources
Workspace-based AI (released 2020) stores all telemetry in a Log Analytics Workspace you own, so you can cross-query AI data with other Azure logs. Classic AI uses an isolated data store that you cannot cross-query and is being deprecated. The exam increasingly tests workspace-based behaviour.

Distributed Tracing & Correlation

When a request spans multiple services (e.g. API Gateway → Microservice A → Microservice B), Application Insights correlates all telemetry items using an Operation Id (a trace ID). This is propagated via HTTP headers:

  • W3C Trace Context (default, recommended): traceparent header. Format: 00-{traceId}-{parentId}-{flags}
  • AI Correlation headers (legacy): Request-Id and Correlation-Context headers. Used by older AI SDK versions.

The SDK automatically reads and writes these headers for outgoing HTTP and Azure SDK calls. For custom dependencies (e.g. a message queue consumer), you must manually start/stop an Activity or call TelemetryClient.StartOperation() to maintain the correlation chain.

🔗
Application Map requires dependency tracking
The Application Map is built automatically from dependency telemetry. Every call from Service A to Service B that is captured as a dependency item with the correct Operation Id appears as an edge in the map. If dependency tracking is disabled (e.g. you set EnableDependencyTrackingTelemetryModule = false), the Application Map will not show inter-service connections.

📈
Sampled counts are estimated — this is heavily tested
When adaptive or fixed-rate sampling is active, the SDK sends a itemCount property with each telemetry item. A requests row with itemCount = 10 and success = false represents 10 actual failures, not 1. KQL aggregation functions like sum(itemCount) give you the estimated real count. Using count() alone gives you the sampled count, which is lower than reality.

Live Metrics Stream

Live Metrics provides a real-time (1-second) stream of telemetry directly from your running application. Key characteristics the exam tests:

  • No sampling applied — every item is streamed (but not persisted to storage).
  • No storage — data is not written to the Log Analytics Workspace. It disappears when you close the Live Metrics blade.
  • Minimal overhead — the stream only activates when someone is watching; SDK polls the AI endpoint for a listener first.
  • Use case: immediate validation after a deployment, not historical analysis.

Smart Detection

Smart Detection proactively analyses your telemetry for anomalies and auto-creates alert rules. Detection algorithms include:

  • Failure anomalies: unusual spike in failed requests or dependency calls.
  • Performance degradation: response time for a specific operation slows significantly vs historical baseline.
  • Memory leak: gradual increase in memory usage correlated with requests.
  • Security detection: potential issues like insecure URL patterns.

Smart Detection does not require any configuration — it learns from your telemetry automatically. It sends email alerts and creates entries in the Azure Monitor Alerts blade.

Snapshot Debugger

When an exception is thrown, the Snapshot Debugger automatically captures a debug snapshot: the call stack and the values of local variables and parameters at the moment of the exception. Snapshots are stored in an Azure Storage account you configure. You can inspect them in Visual Studio or the Azure portal without needing to reproduce the issue. The Snapshot Debugger requires the Microsoft.ApplicationInsights.SnapshotCollector NuGet package and a small performance overhead (it uses a forked process to capture snapshots without blocking the main thread).

Key KQL Queries for the Exam

// Failed requests in the last hour, grouped by hour
requests
| where success == false
| summarize count() by bin(timestamp, 1h)

// Slow dependency calls (over 1 second)
dependencies
| where duration > 1000  // duration is in milliseconds
| order by duration desc

// Exception counts grouped by type
exceptions
| summarize count() by type

// Custom events by name
customEvents
| where name == "OrderPlaced"
| summarize count() by bin(timestamp, 1d)

// Estimated failure count accounting for sampling
requests
| where success == false
| summarize estimatedCount = sum(itemCount) by bin(timestamp, 1h)

// p95 response time per operation
requests
| summarize percentile(duration, 95) by name
| order by percentile_duration_95 desc

// Availability test failures by location
availabilityResults
| where success == false
| summarize count() by location, name
⚠
Connection string vs instrumentation key
Microsoft recommends using the connection string (APPLICATIONINSIGHTS_CONNECTION_STRING) rather than the instrumentation key for new applications. The connection string includes the endpoint override, enabling sovereign cloud support and future endpoint changes. The instrumentation key is still supported but flagged as legacy in newer SDK versions. The exam is starting to test this distinction.
② Exam Gotchas
🚨
Gotcha: Adaptive sampling is ON by default — your counts in the portal are estimates
The AZ-204 exam frequently presents a scenario where a developer queries Application Insights and gets a count of, say, 50 failed requests, but the actual failure count is much higher. This is because adaptive sampling is enabled by default in the ASP.NET Core Application Insights SDK. When the request rate exceeds roughly 5 requests per second, the SDK begins dropping telemetry items and attaches an itemCount property to each retained item indicating how many real events it represents. A developer who uses requests | where success == false | count() will see the sampled count. To see the estimated real count, use sum(itemCount). To get exact counts: either disable sampling entirely (not recommended for high-traffic apps) or use fixed-rate sampling with a known rate and account for it mathematically.
🚨
Gotcha: SQL query text is NOT captured by default — you must opt in
Application Insights dependency tracking will automatically track SQL calls (via SqlClient instrumentation) and record duration, success, and target database. However, the actual SQL command text (the query string) is NOT captured by default. This is a deliberate privacy decision to avoid logging sensitive data like PII in query parameters. To enable SQL text capture, you must explicitly set EnableSqlCommandTextInstrumentation = true on the DependencyTrackingTelemetryModule. The exam tests this — a scenario where you see dependency entries for SQL but the "data" field is blank or shows only the stored procedure name, and asks why the full query is missing.
🚨
Gotcha: Live Metrics does NOT persist data — it cannot be used for historical analysis
Live Metrics streams telemetry directly from the app in real-time with no storage and no sampling. This means: (1) if you close the Live Metrics blade, that data is gone forever — it was never written to the workspace; (2) you cannot query Live Metrics data with KQL; (3) you cannot set alerts based on Live Metrics. The exam presents scenarios like "your team needs to monitor CPU and request rate during a load test and review the data afterwards" — Live Metrics is wrong here because there is no "afterwards". Use Azure Monitor Metrics or Application Insights telemetry (requests table) for post-hoc analysis.

④ CLI / SDK Patterns

Registering Application Insights in ASP.NET Core

// Program.cs — recommended pattern (connection string, not ikey)
builder.Services.AddApplicationInsightsTelemetry(options =>
{
    // Use connection string from config (env var: APPLICATIONINSIGHTS_CONNECTION_STRING)
    options.ConnectionString = builder.Configuration["APPLICATIONINSIGHTS_CONNECTION_STRING"];

    // Disable adaptive sampling to get exact counts (use with caution on high-traffic apps)
    // options.EnableAdaptiveSampling = false;
});

// For worker services / background jobs
builder.Services.AddApplicationInsightsTelemetryWorkerService(options =>
{
    options.ConnectionString = builder.Configuration["APPLICATIONINSIGHTS_CONNECTION_STRING"];
});

TelemetryClient — Core Track Methods

// Inject via DI
public class OrderService
{
    private readonly TelemetryClient _telemetry;

    public OrderService(TelemetryClient telemetry)
    {
        _telemetry = telemetry;
    }

    public async Task PlaceOrder(Order order)
    {
        // Custom event — appears in customEvents table
        _telemetry.TrackEvent("OrderPlaced",
            properties: new Dictionary<string, string>
            {
                ["OrderId"] = order.Id.ToString(),
                ["CustomerId"] = order.CustomerId
            },
            metrics: new Dictionary<string, double>
            {
                ["OrderValue"] = order.TotalAmount
            });

        // Custom metric — appears in customMetrics table
        _telemetry.TrackMetric("OrderValue", order.TotalAmount);

        // Preferred: GetMetric() pre-aggregates locally, reduces telemetry volume
        _telemetry.GetMetric("OrderValue").TrackValue(order.TotalAmount);

        // Custom trace (structured log message) — appears in traces table
        _telemetry.TrackTrace($"Processing order {order.Id}",
            SeverityLevel.Information,
            new Dictionary<string, string> { ["OrderId"] = order.Id.ToString() });

        // Manual dependency tracking (e.g. calling a third-party payment API)
        var startTime = DateTimeOffset.UtcNow;
        var timer = System.Diagnostics.Stopwatch.StartNew();
        bool success = false;
        try
        {
            await _paymentGateway.ChargeAsync(order);
            success = true;
        }
        finally
        {
            timer.Stop();
            // Appears in dependencies table
            _telemetry.TrackDependency(
                dependencyTypeName: "HTTP",     // shown in Application Map
                target: "payment-gateway.example.com",
                dependencyName: "POST /charge",
                data: "/charge",               // SQL text / URL path — not captured by default for SQL
                startTime: startTime,
                duration: timer.Elapsed,
                resultCode: success ? "200" : "500",
                success: success);
        }
    }
}

Configuring Sampling

// Fixed-rate sampling — both client and server must use the same percentage
// This ensures correlated requests are consistently included or excluded
builder.Services.Configure<TelemetryConfiguration>(config =>
{
    var fixedSampling = new FixedRateSamplingTelemetryProcessor(config.DefaultTelemetrySink.TelemetryProcessorChain.Last);
    fixedSampling.SamplingPercentage = 25; // keep 25% of telemetry
    config.TelemetryProcessorChainBuilder
        .Use(next => fixedSampling)
        .Build();
});

// Adaptive sampling — fine-tune the target (default is 5/sec)
builder.Services.Configure<TelemetryConfiguration>(config =>
{
    var adaptive = new AdaptiveSamplingTelemetryProcessor
    {
        MaxTelemetryItemsPerSecond = 10, // allow up to 10/sec before sampling kicks in
        ExcludedTypes = "Exception"      // never sample exceptions — always capture all
    };
});

Custom TrackAvailability Test (from Azure Function)

// Azure Function running on a timer — sends availability result to AI
[FunctionName("AvailabilityTest")]
public async Task Run([TimerTrigger("0 */5 * * * *")] TimerInfo timer)
{
    var testName = "MyAPIHealthCheck";
    var runLocation = "East US";
    var startTime = DateTimeOffset.UtcNow;
    var stopwatch = Stopwatch.StartNew();
    bool isSuccess = false;
    string message = "";

    try
    {
        var response = await _httpClient.GetAsync("https://myapi.example.com/health");
        isSuccess = response.IsSuccessStatusCode;
        message = response.StatusCode.ToString();
    }
    catch (Exception ex)
    {
        message = ex.Message;
    }
    finally
    {
        stopwatch.Stop();
        // This appears in availabilityResults table in Application Insights
        _telemetry.TrackAvailability(
            name: testName,
            timeStamp: startTime,
            duration: stopwatch.Elapsed,
            runLocation: runLocation,
            success: isSuccess,
            message: message);
        _telemetry.Flush(); // important in serverless — ensure data is sent before function exits
    }
}
⑤ Wrong-Answer Traps
⚠
Trap: "Application Insights stores telemetry in the same storage account as the application"
FALSE. Application Insights has its own backend storage, completely separate from any storage account your application uses. In workspace-based mode, data is stored in a Log Analytics Workspace. In classic mode, it is stored in AI's proprietary data store. Your app's Azure Storage account (for blobs, queues, tables) has nothing to do with where AI telemetry is persisted. Choosing "configure the same storage account for Application Insights" on an exam question is always wrong.
⚠
Trap: "Setting a daily cap prevents data loss because all telemetry is buffered and sent the next day"
FALSE. When the daily data cap is reached, Application Insights immediately stops accepting NEW telemetry for the remainder of that UTC calendar day. The telemetry that was not sent is NOT buffered anywhere — it is dropped at the client SDK or at the AI ingestion endpoint (for ingestion sampling). There is no catch-up the next day. This creates a monitoring blind spot. Microsoft recommends: set the cap well above your expected daily volume, and configure a cap alert so you are notified when you are approaching the limit (e.g. at 80% of cap). Do not rely on the cap as the primary cost control — use sampling instead.
⚠
Trap: "Use ingestion sampling to ensure correlated telemetry items are always sampled together"
FALSE. Ingestion sampling is the worst sampling type for correlation consistency. It runs at the Application Insights ingestion endpoint, after the data has already been transmitted from your app. At that point, items from the same operation that were sent in separate HTTP batches may be independently sampled, potentially dropping the Request item but keeping the Dependency items, or vice versa. The result is orphaned telemetry — dependency calls with no parent request. For consistent cross-service correlation, use fixed-rate sampling in the SDK (same percentage, same Operation Id–based hash on both client and server).
⑥ Practice Q&A
1
Your ASP.NET Core application uses the Application Insights SDK with default configuration. A KQL query against the requests table shows 120 failed requests in the last hour. Your on-call engineer insists there were over 1,000 failures — the error rate was clearly visible in server logs. What is the most likely explanation?
▼
✓ Adaptive sampling is dropping telemetry items
The default Application Insights SDK enables adaptive sampling, which automatically reduces the volume of telemetry sent when request rates are high. With adaptive sampling active:

  • Each retained telemetry item carries an itemCount property representing how many real events it represents (e.g. itemCount = 10 means one row = 10 real requests).
  • Using requests | where success == false | count() counts sampled rows — not real events.
  • The correct query is requests | where success == false | summarize sum(itemCount) to get the estimated real count.
  • To get exact counts, disable adaptive sampling (EnableAdaptiveSampling = false) or switch to fixed-rate sampling and document the rate.
Other options like "the instrumentation key is wrong" or "the AI resource has a daily cap" would result in zero data or a cutoff, not a fraction of the real count.
2
A development team wants to trace requests across three microservices: API Gateway, Order Service, and Payment Service. All three use Application Insights. After deployment, the Application Map shows the API Gateway but does NOT show any connections to the Order Service or Payment Service. What is the most likely cause?
▼
✓ Dependency tracking is not enabled in the downstream services
The Application Map is built from dependency telemetry. For a connection to appear between two services:

  • The calling service must record an outgoing dependency call (either via auto-instrumentation or TrackDependency()).
  • The called service must record an incoming request with the same operationId (trace ID), which is propagated via the traceparent W3C header.
  • Both services must send telemetry to the same Application Insights resource (or to linked resources in the same workspace).
Possible fixes: ensure the DependencyTrackingTelemetryModule is not disabled; verify the SDK is initialized in all three services with a valid connection string; check that HTTP headers are not being stripped by a proxy or load balancer. Note: if the services report to different AI resources in different workspaces with no cross-workspace queries configured, they will appear as separate maps.
3
You need to monitor your application's availability from 8 geographic regions and receive an alert only when at least 3 locations simultaneously report failures — to avoid false positives from transient regional outages. You also need to validate an HTTPS SSL certificate expiry alongside the HTTP response. Which availability test type should you use?
▼
✓ Standard test with a multi-location alert rule configured for 3+ failures
The Standard availability test is the correct choice because:

  • It supports SSL/TLS certificate validity checking as a built-in feature (checks certificate expiry, not just HTTPS connectivity).
  • It supports content match and custom request headers in a single-step check.
  • You can configure it to probe from up to 16 geographic locations.
  • The associated alert rule can be set to trigger when X out of N locations report failure — set threshold to 3 to avoid single-region false positives.
URL ping tests do not include SSL certificate validation. Multi-step web tests are deprecated and use Visual Studio .webtest files. Custom TrackAvailability() via Azure Functions would work but requires more code and cannot directly use the built-in SSL check. Standard test is the simplest fit here.
4
Your team is investigating a production bug where an unhandled exception is thrown on roughly 0.1% of requests. The exception is intermittent and cannot be reproduced in dev. You want to automatically capture the local variable values and call stack at the moment the exception occurs, without modifying the core business logic. What should you configure?
▼
✓ Application Insights Snapshot Debugger
The Snapshot Debugger is designed exactly for this scenario:

  • Install Microsoft.ApplicationInsights.SnapshotCollector NuGet package and configure the storage account in appsettings.json.
  • When an exception is thrown, the SDK captures a debug snapshot (mini-dump) containing the call stack and local variable values at that instant.
  • Snapshots are uploaded to Azure Blob Storage and linked to the exception telemetry in Application Insights.
  • View snapshots in the Azure portal's Application Insights exceptions blade or download and open in Visual Studio.
  • Limit: 5 snapshots per exception type per day. After that, the exception continues to be logged but no new snapshots are captured until the next UTC day.
This does not require modifying business logic — only adding the package and configuring it. It does NOT use Live Metrics (which has no storage) and is NOT the same as adding TrackException() manually (which captures the stack trace but not local variable values).
5
A developer creates an Application Insights resource via the Azure portal without specifying a Log Analytics Workspace. They then try to run a cross-workspace KQL query joining Application Insights data with Azure Firewall logs. The query fails. What is the most direct fix?
▼
✓ Migrate the Application Insights resource to workspace-based mode by linking it to a Log Analytics Workspace
Classic Application Insights uses an isolated proprietary data store that cannot participate in cross-workspace KQL queries. To fix this:

  • In the Azure portal, open the Application Insights resource and go to Properties. Use the "Migrate to workspace-based" option to link it to a Log Analytics Workspace.
  • After migration, all new telemetry is written to the workspace and can be joined with any other data source in the same workspace (Firewall, VM, Container Insights, etc.).
  • Historical data from the classic store is NOT automatically migrated — it remains in the classic store with a read-only retention window.
  • Cross-workspace queries use the syntax: workspace("myWorkspace").requests | ... or simply query the workspace directly.
Creating a second Application Insights resource would not fix the problem. Changing the query syntax alone cannot cross the data store boundary of classic AI.
6
You are reviewing Application Insights telemetry for a service that calls an Azure SQL Database. The dependencies table shows entries for SQL calls with correct durations and success values, but the data column is empty for all SQL dependency items. Your manager wants the full SQL query text in the telemetry for query performance analysis. What must you do?
▼
✓ Set EnableSqlCommandTextInstrumentation = true on the DependencyTrackingTelemetryModule
SQL command text is intentionally not captured by default to protect potentially sensitive data (PII in WHERE clauses, etc.):

  • Enable it explicitly in Program.cs: configure the DependencyTrackingTelemetryModule and set EnableSqlCommandTextInstrumentation = true.
  • Once enabled, the data field in the dependencies table will contain the full SQL text for SqlClient calls.
  • This applies to parameterized queries — the parameter VALUES are still not captured (you see @param1, not the actual value). This is by design.
  • Be aware of compliance implications: if queries contain PII patterns, capturing SQL text may violate GDPR or data residency requirements.
This is distinct from stored procedure calls, where the procedure name IS captured in the name field by default — it is the inline query text that requires opt-in.
7
Your high-traffic application uses Application Insights with adaptive sampling. You need distributed tracing to work correctly across your frontend (React SPA) and backend (ASP.NET Core API). Both use the same Application Insights resource. After enabling fixed-rate sampling at 10% on the backend, traces from the frontend sometimes do not show up linked to the backend trace for the same user action. What is the most likely cause?
▼
✓ The JavaScript SDK is still using adaptive sampling — both client and server must use the same fixed-rate sampling percentage and key
Fixed-rate sampling only works for correlation if both ends sample the same items. Fixed-rate sampling is deterministic — it uses the Operation Id (trace ID) as the hash key, so a given trace is either always included or always excluded. But this only works if:

  • Both the JavaScript snippet (frontend) and the ASP.NET Core SDK (backend) use fixed-rate sampling at the same percentage (10%).
  • Both are keyed on the same Operation Id, which is propagated via the traceparent header.
  • If the frontend uses adaptive sampling (or no sampling configuration), it may send traces that the backend discards, or vice versa, breaking the correlation chain.
Fix: in the JavaScript snippet configuration, explicitly set samplingPercentage: 10 to match the backend. The hash-based algorithm ensures that the same Operation Id maps to the same sampling decision on both sides. Never mix adaptive sampling on one end with fixed-rate on the other if correlation matters.
8
An operator sets a daily data cap of 2 GB on an Application Insights resource. At 14:30 UTC, the cap is reached. What happens to telemetry generated by the application between 14:30 UTC and midnight UTC?
▼
✓ All new telemetry is dropped and not stored — there is a monitoring blind spot until midnight UTC
When the daily cap is hit, the AI endpoint immediately stops accepting new data until the cap resets at midnight UTC:

  • Telemetry generated after the cap is reached is NOT buffered or queued — it is permanently lost.
  • The Application Insights resource sends a "DailyCapReached" alert (if configured) and shows a warning in the portal.
  • The cap resets at midnight UTC — NOT midnight in the resource's region or the user's time zone.
  • Best practice: configure a metric alert on the DailyDataCapExceeded metric to be notified when approaching the cap (e.g. 80% threshold), not after it has been hit.
  • Better cost control: use adaptive or fixed-rate sampling in the SDK to reduce volume before data reaches the ingestion endpoint, rather than relying on the daily cap as the primary cost lever.
The cap does NOT cause a graceful degradation where some telemetry is kept — it is a hard stop. Any critical monitoring (e.g. error rate alerts, availability alerts) will stop firing for the rest of that UTC day if the cap is hit.
