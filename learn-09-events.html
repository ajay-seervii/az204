01 Azure Event Grid

① Key Concepts

Event-driven Architecture: Publishers → Topics → Subscriptions → Handlers

Event Grid is a fully managed event routing service that uses a publish-subscribe model. Publishers emit events to topics; Event Grid routes those events to subscribers based on filtering rules. It is designed for reactive workloads (something happened — react to it), not high-throughput data streaming.

Topic TypeWho Creates ItUse CaseExam Note
System Topic Azure (built-in for Azure services) Blob created, resource group changed, VM started — events from Azure resources you already own You subscribe to these; you do not manage the publisher. Enable via portal or CLI per resource.
Custom Topic You (your app publishes events) Your own application events: "order placed", "user registered", "sensor reading received" You get an endpoint + access key. POST events to that endpoint in EventGrid or CloudEvents schema.
Domain Topic You (part of an Event Domain) Multi-tenant SaaS: one domain manages thousands of topics — one per tenant, for example An Event Domain can host up to 100,000 topics. Subscribers can subscribe to all or individual domain topics.

Filtering on Subscriptions

Filters are applied per event subscription, not at the topic level. Three filter types can be combined:

  • Event type filter — include or exclude specific eventType values (e.g., only Microsoft.Storage.BlobCreated).
  • Subject prefix/suffix filter — match the subject field by prefix (e.g., /blobServices/default/containers/images/) or suffix (e.g., .jpg).
  • Advanced filter — JSON path expressions on any field in the event envelope or data object. Supports operators: NumberIn, StringContains, BoolEquals, IsNullOrUndefined, etc.
Advanced filter limit
Each event subscription supports a maximum of 25 advanced filter values per subscription. If you need more, split the logic across multiple subscriptions or pre-filter in a Function.

Dead-letter and Retry Policy

When Event Grid cannot deliver an event (handler returns 4xx/5xx, or the endpoint is unreachable), it enters a retry cycle. Events that exhaust retries or exceed the TTL are either dropped or sent to a dead-letter destination (a Blob Storage container you configure on the subscription).

Dead-letter is opt-in per subscription
Dead-lettering is not enabled by default. You must specify a storage container when creating the subscription. Without it, events that fail delivery are silently discarded after the retry window.
② Exam Gotchas
Webhook Validation Handshake — Custom Endpoints Must Handle It
When you create an event subscription pointing to a custom Webhook URL, Event Grid sends a SubscriptionValidationEvent (POST with eventType: "Microsoft.EventGrid.SubscriptionValidationEvent") to that endpoint. Your handler must extract the validationCode from the event data and respond with {"validationResponse": "<validationCode>"} within 30 seconds. If it does not, subscription creation fails with a validation error. Azure-native handlers (Azure Functions with EventGrid trigger, Logic Apps) handle this automatically — the exam distinguishes these from plain Webhooks. For newer endpoints, there is also an asynchronous validation path using a validationUrl that the endpoint owner visits in a browser.
At-least-once Delivery Does NOT Mean Ordered Delivery
Event Grid guarantees at-least-once delivery — not exactly-once, and not ordered. A single event may be delivered more than once (your handler must be idempotent). Events within the same topic can arrive at different subscribers in different orders, and even the same subscriber can receive events out of publication order. If your exam scenario says "events must be processed in strict order", Event Grid is the wrong answer — use Event Hubs with a single consumer per partition, or Service Bus sessions.
System Topics Are Per-Resource, Not Per-Subscription
A system topic is tied to a specific Azure resource (e.g., a specific Storage account). To receive events from all storage accounts in a subscription, you need to subscribe to the Azure subscription-level system topic or create separate subscriptions per account. The exam may present a scenario where a developer creates one system topic and wonders why they're not seeing events from a second storage account — the answer is that system topics are scoped per source resource.

⑤ Wrong-Answer Traps
Trap: "Event Grid guarantees ordered, exactly-once event delivery"
Both parts are wrong. Event Grid guarantees at-least-once delivery (not exactly-once — duplicate delivery is possible) and makes no ordering guarantee. Exam distractors often frame this as "Event Grid ensures events are processed in the order they were published" — this is false. For ordered processing, use Event Hubs (partition-level ordering) or Service Bus (sessions for ordered message groups).
Trap: "My Azure Function didn't receive the subscription validation event — I need to handle it in code"
When you use an Azure Functions EventGrid trigger binding (not an HTTP trigger), the Functions runtime handles the SubscriptionValidationEvent response automatically. You only need to implement the validation handshake if you are using a plain HTTP endpoint / Webhook as the destination. The exam frequently tests whether candidates know this distinction — choosing the wrong trigger type (HTTP trigger instead of EventGrid trigger) for an Azure Function Webhook would require manual validation logic.
Trap: "Event Grid is the right service for processing 100,000 sensor telemetry events per second"
Event Grid is designed for reactive event routing at low-to-moderate volumes where you need fan-out to multiple handlers. For high-throughput streaming telemetry (IoT sensors, clickstreams, log aggregation), Event Hubs is the correct answer. Event Grid does not provide consumer groups, checkpointing, event replay, or the sustained ingestion throughput that Event Hubs does.
⑥ Practice Q&A
1
A developer creates an event subscription on a custom Event Grid topic with the endpoint pointing to an ASP.NET Core Web API endpoint. The subscription creation fails with a validation error. What is the most likely cause and how should it be fixed?
▼
✓ Correct Answer
The webhook endpoint did not respond to the SubscriptionValidationEvent with the correct validationResponse.
  • When a subscription is created pointing to a Webhook (non-native handler), Event Grid POSTs a SubscriptionValidationEvent to the endpoint immediately.
  • The endpoint must extract data.validationCode from the event and return {"validationResponse": "<code>"} as a 200 response within 30 seconds.
  • If the endpoint is not implemented to handle this event type (e.g., it returns 404 or ignores the body), subscription creation fails.
  • Fix: add a handler in the ASP.NET controller that checks for eventType == "Microsoft.EventGrid.SubscriptionValidationEvent" and returns the echo response.
  • Alternative fix: switch the handler to an Azure Functions EventGrid trigger, which handles validation automatically.
2
Your application publishes events to an Event Grid custom topic. Some downstream consumers are failing to process events and you want failed events to be preserved for investigation. What must you configure, and where?
▼
✓ Correct Answer
Configure a dead-letter destination on each event subscription, pointing to a Blob Storage container.
  • Dead-lettering is configured at the event subscription level, not on the topic itself.
  • You specify a storage account and container. Events that exhaust all retries (up to 30 over 24 hours) are written there as JSON blobs.
  • Dead-lettering is off by default — without it, failed events are silently dropped after the retry window.
  • The storage account must be in the same subscription as the event subscription (or you must grant Event Grid the Storage Blob Data Contributor role on the container).
  • CLI flag: --deadletter-endpoint on az eventgrid event-subscription create.
3
A SaaS platform hosts 5,000 customers and wants each customer to receive only their own events via Event Grid. Creating 5,000 custom topics separately would be operationally unmanageable. What Event Grid feature addresses this?
▼
✓ Correct Answer
Event Domains, which host up to 100,000 domain topics under a single management endpoint.
  • An Event Domain is a single resource that acts as a container for thousands of topics.
  • Each customer gets their own domain topic (e.g., myDomain.eventgrid.azure.net/topics/customer-1234).
  • The domain owner can subscribe to all topics at once or to individual topics per customer.
  • Publishing uses a single endpoint with a per-topic path, or the domain endpoint with a topic specified in the event subject.
  • Limit: up to 100,000 topics per domain. Beyond that, use multiple domains.
4
You need to filter Event Grid events so that only events where the custom data field data.priority equals "high" are delivered to your handler. Which filter type do you use?
▼
✓ Correct Answer
Advanced filter using a StringIn or StringContains operator on the JSON path data.priority.
  • Subject prefix/suffix filters only work on the top-level subject field — they cannot inspect data.
  • Advanced filters use JSON path expressions to reference any field, including nested fields inside the data object.
  • Example: {"operatorType": "StringIn", "key": "data.priority", "values": ["high"]}.
  • Up to 25 advanced filter conditions per subscription; up to 5 values per StringIn condition.
  • Can combine multiple advanced filters — all conditions are ANDed together.
5
An Event Grid subscription has a retry policy of 30 attempts and a 24-hour TTL. The target webhook goes offline for 36 hours. What happens to events published during the outage?
▼
✓ Correct Answer
Events are dead-lettered (if configured) or permanently lost after 24 hours, regardless of the 30-retry limit.
  • The retry policy has two stop conditions: exhausting all 30 retries, OR exceeding the event TTL — whichever occurs first.
  • Events published within the 24-hour window before the outage ends will have their TTL expire before delivery can succeed (since the endpoint is down for 36 hours).
  • Events are not "held" indefinitely — once the TTL of 24 hours expires, they are moved to the dead-letter destination or dropped.
  • This means that even if the endpoint comes back after 36 hours, the events are gone from Event Grid's retry queue.
  • For scenarios requiring longer durable storage, use Service Bus (up to 14-day message lock + durable queue) or write to Event Hubs Capture.
6
A developer says: "I need multiple independent services to each receive every event from my Event Grid topic. I'll create a single subscription and have all services share it." What is wrong with this design?
▼
✓ Correct Answer
A single subscription delivers each event to exactly one endpoint. Each independent service needs its own subscription to receive all events independently.
  • Event Grid subscriptions route each event to the one handler configured on that subscription — it is not a broadcast to multiple handlers on the same subscription.
  • To achieve fan-out (every service gets every event), create one subscription per service handler. Event Grid delivers the event to all matching subscriptions independently.
  • This is unlike a Service Bus queue (competing consumers) — Event Grid is designed for fan-out via multiple subscriptions.
  • Up to 500 subscriptions per topic (default limit), so the pattern scales to hundreds of independent consumers.
7
You want to receive events when any blob is created anywhere in your Azure subscription (across all storage accounts). What Event Grid topic type and configuration should you use?
▼
✓ Correct Answer
Create a system topic at the Azure subscription or resource group scope, not at the individual storage account scope.
  • Event Grid system topics can be scoped to an Azure subscription or resource group, which will surface events from all matching resources within that scope.
  • Example: subscribe to the Azure subscription-level system topic and filter for eventType: Microsoft.Storage.BlobCreated — this catches blobs from any storage account in the subscription.
  • Alternatively, use Azure Policy or Event Grid event subscriptions at the resource group level to capture events from all storage accounts in that group.
  • A system topic per individual storage account is the wrong approach when coverage needs to span multiple accounts.
8
Your Event Grid custom topic is configured with CloudEvents 1.0 schema. A developer posts events using the old EventGrid schema format and reports that events are silently dropped. Why?
▼
✓ Correct Answer
The topic's input schema is set to CloudEvents 1.0; events submitted in EventGrid schema format fail schema validation and are rejected (returning a 400 error), not silently dropped.
  • Each custom topic has a fixed --input-schema set at creation time: eventgridschema, cloudeventschemav1_0, or customeventschema.
  • Events that do not match the configured schema return an HTTP 400 Bad Request to the publisher — the publisher should be logging this error, not reporting silent drops.
  • The schema is immutable after topic creation — you cannot change a topic from EventGrid schema to CloudEvents schema later. You would need to create a new topic.
  • The fix: update the publishing code to use CloudEvents 1.0 format (required fields: specversion, type, source, id), and set the Content-Type header to application/cloudevents+json.

02 Azure Event Hubs

① Key Concepts

High-throughput Event Streaming Architecture

Event Hubs is a distributed streaming platform built for ingesting millions of events per second. Think of it as a durable, append-only log split into partitions. Unlike Event Grid (routing/reaction), Event Hubs is designed for streaming pipelines: telemetry ingestion, log aggregation, real-time analytics, and event replay.

ConceptDefinitionExam Relevance
Partition An ordered, append-only sequence of events. Each event in a partition has a monotonically increasing offset. Partitions are set at namespace/hub creation. Partition count determines max parallelism. Cannot decrease after creation; can increase only in Premium/Dedicated.
Consumer Group A named view of the entire event stream. Each consumer group maintains its own offset position independently of all others. Multiple consumer groups allow multiple independent applications to read the same data without interfering. The $Default consumer group always exists.
Offset / Checkpoint Offset is the byte position of an event within a partition. A checkpoint is when a consumer saves its current offset so it can resume from that point after a restart. Without checkpointing, a consumer restarts from the beginning of the retention window (or latest, depending on configuration). EventProcessorClient does checkpointing automatically to Blob Storage.
Throughput Unit (TU) Standard tier capacity unit. 1 TU = 1 MB/s ingress, 2 MB/s egress, up to 1,000 events/s ingress. You can scale TUs manually or use auto-inflate (Standard) to auto-scale up to a configured maximum.
Processing Unit (PU) Premium tier capacity unit. More powerful than TUs; provides isolation and higher limits. Premium PUs support dynamic partition count increases (unlike Standard TUs).
Capacity Unit (CU) Dedicated tier. A single-tenant cluster for maximum throughput, longest retention, and full isolation. Dedicated = single-tenant. If a question says "tenant isolation" or "predictable performance at scale", dedicated is the answer.
Event Hubs Capture Automatic, continuous snapshot of the event stream written to Azure Blob Storage or ADLS Gen2 in Apache Avro format. Enabled per Event Hub (not namespace). Configurable time/size window (minimum 60 seconds or 10 MB). Avro format means consumers must deserialize using an Avro library.

Partition key determines which partition an event goes to
If you specify a partition key when publishing, Event Grid hashes it to assign the event to a consistent partition. All events with the same partition key land in the same partition, guaranteeing relative ordering for that key. If you omit the partition key (or use round-robin), events are distributed across all partitions — higher throughput but no key-based ordering.
EventProcessorClient is the modern SDK for consuming
The EventProcessorClient (in Azure.Messaging.EventHubs.Processor) handles partition ownership, load balancing across multiple consumer instances, and checkpointing to a Blob Storage container. The older EventProcessorHost is legacy. The exam may reference both — prefer EventProcessorClient in modern scenarios.
② Exam Gotchas
Partition Count Is Effectively Immutable for Standard Tier
For Standard tier, partition count is set at Event Hub creation and cannot be changed (neither increased nor decreased) after the fact. If you need more parallelism later, you must delete and recreate the Event Hub (losing all retained events) or migrate to Premium/Dedicated where dynamic partition increases are supported. The exam tests this directly: "your throughput is insufficient and you need more parallelism — what do you do with a Standard namespace?" The answer is: you cannot simply increase partitions; you must add more TUs for throughput, and for true partition parallelism increases, you must recreate the hub or upgrade to Premium.
Consumer Groups Grant Independent Views — Consumers Within a Group Split Partitions
This is the most frequently misunderstood Event Hubs concept on the exam. A consumer group as a whole reads the entire event stream (all partitions). But within a consumer group, each active consumer instance owns a subset of partitions — no partition is read by two consumers in the same group simultaneously. With 4 partitions and 2 consumers in the same group, each consumer reads 2 partitions. If you add a 5th consumer when there are only 4 partitions, the 5th consumer is idle (no partitions to own). Adding more consumers than partitions does NOT increase throughput — you need more partitions first.
Event Hubs Capture Uses Avro Format — Downstream Consumers Must Deserialize
Event Hubs Capture writes event data to Blob Storage or ADLS Gen2 in Apache Avro binary format, not plain JSON. Files are organized by a configurable naming convention: {Namespace}/{EventHub}/{PartitionId}/{Year}/{Month}/{Day}/{Hour}/{Minute}/{Second}.avro. A scenario where a developer claims they can directly read Capture files as JSON is incorrect — they need an Avro reader (e.g., the Apache.Avro NuGet package, or Azure Data Factory, or Azure Synapse). Also: Capture is enabled per Event Hub, not at the namespace level.

④ CLI / SDK Patterns

Create a Namespace, Event Hub, and Enable Capture

# Create a Standard namespace (choose tier carefully — Basic lacks consumer groups/Capture)
az eventhubs namespace create \
  --name myEHNamespace \
  --resource-group myRG \
  --location eastus \
  --sku Standard \               # Basic | Standard | Premium
  --capacity 2 \                  # Number of TUs; auto-inflate can increase this automatically
  --enable-auto-inflate true \
  --maximum-throughput-units 10   # Auto-inflate ceiling

# Create an Event Hub with 4 partitions and 7-day retention
az eventhubs eventhub create \
  --name myTelemetryHub \
  --namespace-name myEHNamespace \
  --resource-group myRG \
  --partition-count 4 \           # Cannot change for Standard after creation
  --message-retention 7           # Days; max 7 for Standard, 90 for Premium/Dedicated

# Enable Capture to Blob Storage (must pre-create the container)
az eventhubs eventhub update \
  --name myTelemetryHub \
  --namespace-name myEHNamespace \
  --resource-group myRG \
  --enable-capture true \
  --capture-interval 300 \        # Seconds; min 60, max 900
  --capture-size-limit 314572800 \ # Bytes; 300 MB here; min 10 MB
  --destination-name EventHubArchive.AzureBlockBlob \
  --storage-account /subscriptions/{subId}/resourceGroups/myRG/providers/Microsoft.Storage/storageAccounts/mycapturestorage \
  --blob-container capturecontainer \
  --archive-name-format "{Namespace}/{EventHub}/{PartitionId}/{Year}/{Month}/{Day}/{Hour}/{Minute}/{Second}.avro"

# Create a consumer group (Standard tier allows multiple)
az eventhubs eventhub consumer-group create \
  --name analyticsConsumerGroup \
  --eventhub-name myTelemetryHub \
  --namespace-name myEHNamespace \
  --resource-group myRG

SDK — Send and Receive Events (.NET)

// NuGet: Azure.Messaging.EventHubs
//        Azure.Messaging.EventHubs.Processor  (for EventProcessorClient)

using Azure.Messaging.EventHubs;
using Azure.Messaging.EventHubs.Producer;
using Azure.Messaging.EventHubs.Processor;
using Azure.Storage.Blobs;

// ── PRODUCER: Send events with a partition key ──────────────────────────
var producerClient = new EventHubProducerClient(connectionString, "myTelemetryHub");

// Events with the same partition key always go to the same partition (ordering guarantee)
using EventDataBatch batch = await producerClient.CreateBatchAsync(
    new CreateBatchOptions { PartitionKey = "device-sensor-42" }
);
batch.TryAdd(new EventData(Encoding.UTF8.GetBytes("{\"temp\":72.4}")));
batch.TryAdd(new EventData(Encoding.UTF8.GetBytes("{\"temp\":72.6}")));
await producerClient.SendAsync(batch);

// ── CONSUMER: EventProcessorClient with checkpointing ───────────────────
// Checkpoint store must be a Blob Storage container
var storageClient = new BlobContainerClient(storageConnectionString, "checkpoints");
var processor = new EventProcessorClient(
    storageClient,
    consumerGroup: "$Default",     // Use named consumer group for isolation
    connectionString: connectionString,
    eventHubName: "myTelemetryHub"
);

processor.ProcessEventAsync += async (args) => {
    string body = Encoding.UTF8.GetString(args.Data.Body.ToArray());
    Console.WriteLine($"Partition {args.Partition.PartitionId}: {body}");
    // Checkpoint saves offset so restart continues from here, not the beginning
    await args.UpdateCheckpointAsync();
};

processor.ProcessErrorAsync += (args) => {
    Console.WriteLine($"Error on partition {args.PartitionId}: {args.Exception.Message}");
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
// ... keep running ...
await processor.StopProcessingAsync();

⑤ Wrong-Answer Traps
Trap: "Adding more consumers to a consumer group increases read throughput"
Not beyond the partition count. Each partition can only be owned by one consumer instance per consumer group. If you have 4 partitions and spin up 6 consumer instances in the same consumer group, 2 of those instances will be idle with no partitions assigned. To increase read parallelism, you need more partitions — and for Standard tier you cannot add partitions after creation, so you would need to recreate the hub (or use Premium/Dedicated and increase partitions dynamically).
Trap: "Event Hubs Basic supports multiple consumer groups"
Basic tier only supports the single built-in $Default consumer group. You cannot create additional consumer groups on a Basic namespace. The exam frequently presents a scenario where you need multiple independent applications to read the same stream (requiring separate consumer groups) and asks which tier is required — the answer is Standard or higher. If the question says "Basic tier is already deployed", the fix is to upgrade to Standard, not add consumer groups.
Trap: "Event Hubs guarantees exactly-once delivery"
Event Hubs guarantees at-least-once delivery. A consumer that crashes between receiving an event and checkpointing will re-read that event on restart. To achieve exactly-once semantics you must implement idempotency in your processing logic (e.g., track event IDs in a database and skip duplicates). The Kafka surface on Event Hubs supports idempotent producer settings, but the AMQP/SDK surface does not provide exactly-once out of the box.
⑥ Practice Q&A
1
An IoT application sends 500,000 telemetry events per second to Event Hubs. The processing team needs three independent applications to each consume and analyze the full stream without interfering with each other. How should the Event Hubs namespace be configured?
▼
✓ Correct Answer
Standard tier (or higher) with three separate consumer groups — one per application.
  • Each consumer group maintains its own independent offset cursor across all partitions. All three groups receive all events from the hub.
  • Basic tier supports only $Default — it cannot provide three independent groups. Standard tier supports up to 20 consumer groups per Event Hub.
  • The three applications do not share offsets. One application falling behind does not affect the others.
  • For 500K events/second at ~200 bytes each, that is roughly 100 MB/s ingress, requiring approximately 100 TUs. Use Standard with auto-inflate or upgrade to Premium.
  • Ensure partition count is high enough (e.g., 32) to allow each application to scale consumer instances for parallel processing.
2
A developer notices that after their event processor restarts, it begins re-reading events from an hour ago instead of continuing where it left off. What is the most likely cause?
▼
✓ Correct Answer
The processor is not checkpointing, or checkpoints are being written to a location that does not persist across restarts.
  • Without a valid checkpoint, EventProcessorClient defaults to reading from the beginning of the retention window (or from EventPosition.Earliest for that partition).
  • If using EventProcessorClient, a Blob Storage container must be provided as the checkpoint store, and await args.UpdateCheckpointAsync() must be called after each event (or after processing a batch) to persist the current offset.
  • Common mistake: checkpointing to a local file or in-memory store that is lost on restart, rather than Blob Storage.
  • Another cause: the checkpoint container is being cleared between restarts, or the connection string points to a different storage account in production than in development.
  • Fix: verify the BlobContainerClient points to a persistent container, ensure UpdateCheckpointAsync is called, and verify the container exists before starting the processor.

A team needs to archive all incoming Event Hubs events to long-term storage for compliance. They want to query the archived data with Azure Synapse Analytics. Which Event Hubs feature should they use, and what is the output format?
▼
✓ Correct Answer
Event Hubs Capture to Azure Data Lake Storage Gen2 in Apache Avro format.
  • Capture automatically writes events to either Azure Blob Storage or ADLS Gen2 in Avro format. ADLS Gen2 is preferred for Synapse Analytics integration (native connector support).
  • Files are organized by a time-based naming pattern: {Namespace}/{EventHub}/{PartitionId}/{Year}/{Month}/{Day}/{Hour}/{Minute}/{Second}.avro.
  • Avro files can be queried from Azure Synapse via external tables using the built-in Avro reader, or processed with Azure Data Factory mapping data flows.
  • Capture is enabled per Event Hub, not per namespace. It requires Standard tier or higher (Basic does not support Capture).
  • The capture window is configurable: write when either the time interval (min 60 sec) or the size threshold (min 10 MB) is reached.
4
An Event Hubs Standard namespace has an Event Hub with 8 partitions. A developer deploys 12 instances of a consumer application all using the same consumer group. How many instances will actively process events?
▼
✓ Correct Answer
Only 8 instances will actively process events; the remaining 4 will be idle.
  • Each partition can be owned by exactly one consumer instance within a consumer group at any point in time.
  • With 8 partitions and 12 consumers in the same group, the EventProcessorClient load balancer assigns one partition per active consumer until all 8 are assigned.
  • The remaining 4 consumer instances will enter a standby/idle state. They will take over partitions only if one of the 8 active instances fails or is removed.
  • This means running more consumers than partitions wastes compute resources but provides faster failover (idle consumers pick up orphaned partitions quickly).
  • To make all 12 instances active, increase partition count to 12 — but on Standard tier this requires recreating the Event Hub. On Premium/Dedicated, you can increase partitions dynamically.
5
You are designing a solution where all events from the same device must be processed in order by a single consumer. How should you configure event publishing to Event Hubs to guarantee this?
▼
✓ Correct Answer
Publish events with the device ID as the partition key. All events sharing the same partition key are routed to the same partition and are read in order within that partition.
  • Event Hubs hashes the partition key to determine the target partition. All events with the same key consistently land on the same partition.
  • Within a partition, events are stored in append-only order by offset — a consumer reading that partition will see events in the exact order they were ingested.
  • The consumer owning that partition (within a consumer group) processes its events serially, guaranteeing per-device ordering.
  • Do NOT use round-robin publishing (no partition key) if ordering per device matters — events from the same device could land on different partitions.
  • Tradeoff: if one device generates far more events than others, its partition becomes a hot partition. Consider hashing device IDs into a fixed number of buckets to distribute load.

A company is evaluating whether to use Azure Event Grid or Azure Event Hubs for their new system that receives 200,000 log entries per second from distributed servers and needs to replay the last 3 days of logs on demand. Which service is correct and why?
▼
✓ Correct Answer
Azure Event Hubs — it supports high-throughput ingestion, event retention for replay, and consumer groups for independent readers.
  • Event Grid is for discrete event routing (something happened → react). It has no event retention or replay capability; once an event is delivered, it's gone.
  • Event Hubs is a streaming log — events are retained for a configurable window (1–7 days Standard, up to 90 days Premium/Dedicated). Consumers can replay from any offset within the retention window.
  • 3-day replay requires Standard tier with --message-retention 3 (or higher). Basic supports only 1 day and no extra consumer groups.
  • 200,000 log entries/second is well within Event Hubs Standard capacity (with enough TUs and partitions).
  • For even longer retention beyond 90 days, combine Event Hubs with Capture → ADLS Gen2, and replay from there using Azure Synapse or custom consumers.

An existing Event Hubs Standard namespace was created with 4 partitions. After six months, throughput has grown and the team needs 16 partitions for more consumer parallelism. What are the options?
▼
✓ Correct Answer
For Standard tier, you cannot increase partition count in-place. Options are: (1) recreate the Event Hub with 16 partitions (accepting data loss from the old hub), or (2) upgrade the namespace to Premium tier, which supports dynamic partition increases.
  • Standard tier partition count is immutable after Event Hub creation — it can be neither increased nor decreased.
  • Option 1 — Recreate: create a new Event Hub with 16 partitions, redirect producers to it, and drain the old hub. Any events still in the old hub's retention window must be re-processed before decommissioning it.
  • Option 2 — Upgrade to Premium: Premium tier supports az eventhubs eventhub update --partition-count to increase (not decrease) partitions on an existing hub without data loss. Premium costs more but provides isolation, customer-managed keys, and longer retention.
  • Adding more TUs alone will improve ingress/egress throughput but will NOT increase consumer parallelism beyond the current partition count.
8
A team uses Event Hubs Capture and reports that they cannot read the captured files as JSON using a standard text editor or az storage blob download. What is the correct explanation?
▼
✓ Correct Answer
Event Hubs Capture writes files in Apache Avro binary format, not plain JSON. They must be deserialized using an Avro reader.
  • Avro is a compact binary serialization format. Opening a .avro file in a text editor shows binary garbage — it is not human-readable.
  • The Avro file contains a schema header followed by compressed binary data blocks. The schema is embedded in the file header.
  • To read: use the Apache.Avro NuGet package in .NET, avro-tools CLI, Azure Data Factory, Azure Synapse Analytics (built-in Avro support), or Azure Databricks (spark.read.format("avro")).
  • There is no option to change Capture output to JSON — Avro is the only supported format for Event Hubs Capture.
  • The file naming convention embeds partition ID and timestamp, making it easy to find events from a specific partition and time range.
