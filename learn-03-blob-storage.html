Topic 03

M1 Explore Azure Blob Storage

① Key Concepts

Storage Account Types

Account Type Supported Services Redundancy Options Key Use Case
General-purpose v2 (GPv2) Blob, File, Queue, Table, Disk LRS, ZRS, GRS, GZRS, RA-GRS, RA-GZRS Recommended for most scenarios; supports all access tiers including Archive
Premium Block Blob Block blobs and append blobs only LRS, ZRS only High-throughput, low-latency workloads; no GRS/GZRS support
Premium Page Blob Page blobs only LRS only Azure VM unmanaged disks; random read/write access
Premium File Share Azure Files only LRS, ZRS only High-performance SMB/NFS file shares for enterprise workloads
GPv2 is always the recommended default
If an exam question asks which account type to use and does not specify performance or pricing constraints, choose General-purpose v2. It supports every feature including lifecycle management, all redundancy tiers, and all access tiers.

Blob Types

Blob Type Max Size Block Size / Unit Allowed Operations Typical Use Case
Block Blob 4.77 TB Up to 100 MB per block; max 50,000 blocks Put Block, Put Block List, Get Block List; full read/write/delete Files, images, videos, documents, backups
Append Blob 195 GB Up to 4 MB per append block; max 50,000 blocks Append Block only — no overwrite, no update Log files, audit trails, telemetry streams
Page Blob 8 TB 512-byte pages (aligned I/O) Random read/write to any 512-byte range Azure VM unmanaged OS/data disks, virtual hard drives
Append blobs cannot be overwritten — only appended
This is the defining exam trap for append blobs. Once a block is committed, it cannot be modified. You can append new blocks (each up to 4 MB), but you cannot update bytes in an existing block. This makes append blobs ideal for log ingestion where you never need to correct past entries.

Access Tiers

Tier Access Pattern Min Storage Duration Storage Cost Access Cost Availability SLA
Hot Frequent None Highest Lowest 99.9% (RA: 99.99%)
Cool Infrequent (>30 days) 30 days (early deletion fee) Lower than Hot Higher than Hot 99% (RA: 99.9%)
Cold Rare (>90 days) 90 days (early deletion fee) Lower than Cool Higher than Cool 99% (RA: 99.9%)
Archive Offline / long-term (>180 days) 180 days (early deletion fee) Lowest Highest + rehydration fee N/A (offline — must rehydrate)
Account-level vs. blob-level tiering
The account-level default tier (Hot or Cool) applies to blobs with no explicit tier set. Cold and Archive tiers can only be set at the individual blob level — they cannot be set as the account default. This distinction appears on exams.

Redundancy Options

Redundancy Copies Failure Protection Read from Secondary?
LRS (Locally Redundant) 3 copies — same datacenter Drive/rack failure within one datacenter No
ZRS (Zone Redundant) 3 copies — 3 availability zones, same region Zone-level failure within one region No
GRS (Geo Redundant) 6 copies — LRS primary + LRS secondary region (async) Regional disaster (secondary is read-only after failover) Only with RA-GRS
GZRS (Geo-Zone Redundant) 6 copies — ZRS primary + LRS secondary region (async) Zone failure + regional disaster Only with RA-GZRS
RA-GRS / RA-GZRS Same as GRS/GZRS Same as GRS/GZRS Yes — always, even before failover
② Exam Gotchas
Archive blobs cannot be read directly — rehydration is mandatory
A blob in the Archive tier is stored offline. Any attempt to read, copy, or download it returns a 409 BlobArchived error until the blob is rehydrated. Rehydration means either: (a) changing the blob's tier back to Hot or Cool using Set Blob Tier, or (b) copying the archived blob to a new blob in a Hot or Cool tier. Standard rehydration priority takes up to 15 hours. High priority takes under 1 hour for blobs smaller than 10 GB and costs more.
Premium storage accounts do NOT support Archive tier or lifecycle management
Premium Block Blob, Premium Page Blob, and Premium File Share accounts are optimized for low latency — they only support Hot tier. You cannot tier blobs to Cool, Cold, or Archive in a Premium account, and lifecycle management policies do not apply to Premium accounts at all. If an exam scenario says the app needs lifecycle tiering, the answer is GPv2, not Premium.
ZRS does NOT protect against regional disasters
ZRS replicates across three availability zones in the same Azure region. If the entire region experiences a disaster (power, network, or natural disaster affecting all zones), ZRS data is at risk. Only GRS, GZRS, RA-GRS, or RA-GZRS replicate data to a geographically distant paired region. Exam distractors often describe ZRS as "protecting against regional outages" — that is false.

④ CLI / SDK Patterns
# Create a GPv2 storage account with GRS redundancy
az storage account create \
  --name mystorageacct \
  --resource-group myRG \
  --location eastus \
  --sku Standard_GRS \          # SKU encodes both type and redundancy
  --kind StorageV2 \            # StorageV2 = GPv2; BlobStorage = legacy Blob account
  --access-tier Hot             # Default account tier: Hot or Cool (not Cold/Archive)

# Create a Premium Block Blob account (high-throughput, no Archive support)
az storage account create \
  --name mypremiumacct \
  --resource-group myRG \
  --location eastus \
  --sku Premium_LRS \           # Premium only supports LRS or ZRS
  --kind BlockBlobStorage       # BlockBlobStorage = Premium Block Blob account

# Upload a blob and immediately set its access tier
az storage blob upload \
  --account-name mystorageacct \
  --container-name mycontainer \
  --name myblob.txt \
  --file ./myblob.txt \
  --tier Cool                   # Sets tier at upload time (GPv2 only)

# Change an existing blob tier to Archive
az storage blob set-tier \
  --account-name mystorageacct \
  --container-name mycontainer \
  --name myblob.txt \
  --tier Archive                # Blob becomes offline; reads will fail until rehydrated

# Rehydrate an archived blob by copying to a new blob in Hot tier
az storage blob copy start \
  --account-name mystorageacct \
  --destination-blob rehydrated-myblob.txt \
  --destination-container mycontainer \
  --source-uri "https://mystorageacct.blob.core.windows.net/mycontainer/myblob.txt" \
  --tier Hot                    # Destination tier — sets the rehydrated copy to Hot
  --rehydrate-priority High     # Standard (up to 15h) or High (under 1h, costs more)
// C# SDK — BlobServiceClient hierarchy
using Azure.Storage.Blobs;
using Azure.Storage.Blobs.Models;

// Top-level: BlobServiceClient (storage account)
var serviceClient = new BlobServiceClient(connectionString);

// Container level
var containerClient = serviceClient.GetBlobContainerClient("mycontainer");
await containerClient.CreateIfNotExistsAsync(PublicAccessType.None); // No public access by default

// Blob level — upload
var blobClient = containerClient.GetBlobClient("myblob.txt");
using var stream = File.OpenRead("./myblob.txt");
await blobClient.UploadAsync(stream, overwrite: true);  // overwrite: true silently replaces

// Set blob access tier programmatically
await blobClient.SetAccessTierAsync(AccessTier.Cool);

// Change tier to Archive
await blobClient.SetAccessTierAsync(AccessTier.Archive);
⑤ Wrong-Answer Traps
Trap: "Use a Premium Block Blob account for infrequently accessed data to save costs"
Premium Block Blob accounts only support Hot tier. They have higher storage costs than GPv2 in exchange for sub-millisecond latency. They do not support Cool, Cold, or Archive tiers. Premium accounts are only appropriate for latency-sensitive workloads, not cost-optimized cold storage.
Trap: "RA-GRS provides two independent write endpoints for load balancing writes"
RA-GRS (Read-Access Geo-Redundant Storage) provides a read-only secondary endpoint. You can read from the secondary endpoint at any time using the -secondary URL suffix (e.g., mystorageacct-secondary.blob.core.windows.net), but all writes still go to the primary region only. RA-GRS is for read scale-out and disaster recovery reads, not write load balancing.
Trap: "You can update bytes within an existing append blob block to fix a log entry"
Append blobs are write-once per block. Once a block is appended and committed, its content is immutable. The only operation allowed after creation is appending new blocks. To "correct" data in an append blob, you would need to create a new blob. This is by design — it makes append blobs suitable for tamper-evident audit logs.
⑥ Practice Q&A
1
A company stores compliance logs that must be retained for 7 years but accessed only once or twice per year. The logs are written sequentially and must never be overwritten. Which blob type and access tier combination is most appropriate?
▼
✓ Append Blob + Archive tier
Append blob is the correct blob type because log files are written sequentially and must not be overwritten — append blobs enforce this by only allowing the Append Block operation.

Archive tier is correct because: access is rare (<2 times/year), it has the lowest storage cost, and the 180-day minimum is easily met (7-year retention). The early-deletion fee is irrelevant here since the data stays far beyond 180 days.
  • Block blob would also hold the data, but does not enforce the no-overwrite constraint
  • Page blob is for random-access scenarios like VM disks — not log files
  • Cool tier would work but costs more than Archive for 7-year retention with rare access
2
Your application reads from Azure Blob Storage in a different Azure region when the primary region is unavailable. You need the secondary endpoint to be always readable, not just during failover. Which redundancy option should you configure?
▼
✓ RA-GRS (Read-Access Geo-Redundant Storage)
RA-GRS provides a persistent read-only secondary endpoint that is always available, regardless of whether the primary region is experiencing issues. The secondary URL is accountname-secondary.blob.core.windows.net.
  • GRS replicates to a secondary region but does NOT expose a readable secondary endpoint — you can only read from secondary after Microsoft initiates a failover
  • ZRS does not replicate to another region at all — it stays within the same region across zones
  • GZRS combines zone redundancy with geo-replication, but still does not expose a secondary read endpoint — only RA-GZRS does
3
A developer uploads a 6 GB video file to Azure Blob Storage using the REST API. The upload fails with a 413 error. What is the most likely cause, and how should the upload be redesigned?
▼
✓ File exceeds single Put Blob limit (5 GB); use Put Block + Put Block List
The Put Blob operation has a maximum size limit of 5,000 MB (5 GB). A 6 GB file exceeds this limit. The correct approach for large blobs is chunked upload:
  • Put Block: upload the file in chunks (each up to 100 MB with newer API versions)
  • Put Block List: commit the list of block IDs to assemble the final blob
  • The Azure SDK handles this automatically via UploadAsync — it detects the file size and automatically switches to multipart upload when the file exceeds the single-request threshold
  • Manually staging blocks gives you parallelism (upload multiple blocks concurrently) and resume capability (retry failed blocks only)
4
A blob was moved to Archive tier 45 days ago. A business requirement now requires the data to be accessible within 30 minutes. A developer submits a rehydration request. Approximately how long will this take, and what priority should be selected?
▼
✓ Select High Priority rehydration — under 1 hour for blobs under 10 GB
The 30-minute SLA rules out Standard priority rehydration (up to 15 hours). High priority rehydration completes in under 1 hour for blobs smaller than 10 GB.
  • The blob has been in Archive for 45 days — well past the 180-day minimum, so there is no early deletion fee
  • High priority rehydration costs more per operation but is the only option that meets the 30-minute window
  • Rehydration can be done by: (a) calling Set Blob Tier with rehydratePriority=High to move the blob back to Hot/Cool, or (b) Copy Blob to a new Hot/Cool destination blob
  • The original blob remains in Archive until the operation completes — the blob's status will show as rehydrate-pending-to-hot or rehydrate-pending-to-cool during the process
5
Which storage account type should you choose to host Azure VM unmanaged disks that require random read/write access at the block level?
▼
✓ Premium Page Blob storage account (or GPv2 with page blobs)
Page blobs are the only blob type that supports random 512-byte aligned read/write operations, making them the correct blob type for VM disks.
  • Premium Page Blob account (kind: StorageV2 with Premium_LRS or, historically, Storage with Premium_LRS) provides the low latency required for OS and data disks
  • GPv2 with Standard_LRS can also host page blobs (unmanaged disks) but with higher latency — acceptable for non-critical or dev/test VMs
  • Block blobs and append blobs do not support random-access writes — they do not work for VM disk scenarios
  • Modern Azure guidance is to use managed disks instead of unmanaged page blobs — but page blobs still appear on the AZ-204 exam
6
A storage account uses GRS redundancy. The primary region becomes unavailable. A developer immediately attempts to read from the secondary endpoint. What happens?
▼
✓ The read fails — GRS secondary is not readable; only RA-GRS secondary is always readable
With standard GRS, the secondary region receives asynchronous copies of data, but the secondary endpoint is not exposed for reads unless Microsoft initiates a storage account failover (which promotes the secondary to primary). Until failover completes, reads from the secondary URL will fail.
  • To read from the secondary at any time without waiting for failover, you need RA-GRS
  • With RA-GRS, the secondary endpoint is always at: accountname-secondary.blob.core.windows.net
  • Note: data in the secondary may be slightly stale due to asynchronous replication (the RPO/recovery point objective is typically under 15 minutes, but Microsoft does not guarantee a specific RPO value)
7
A developer needs a storage account for a high-throughput ETL pipeline that ingests millions of small files per second with sub-millisecond latency. The data never needs tiering. Which account type and redundancy should they choose?
▼
✓ Premium Block Blob account with LRS or ZRS
Premium Block Blob (kind: BlockBlobStorage, SKU: Premium_LRS or Premium_ZRS) is purpose-built for high-throughput, low-latency block blob workloads.
  • Sub-millisecond first-byte latency is only available on Premium accounts — GPv2 with Standard SKU has higher latency (single-digit milliseconds)
  • Premium Block Blob only supports LRS and ZRS — GRS/GZRS are not available, which is acceptable since the scenario doesn't mention geo-disaster recovery
  • No need for tiering rules out the only significant limitation vs. GPv2
  • The ETL pipeline would typically connect via the Azure SDK or ADLS Gen2 compatible API if hierarchical namespace is needed

M2 Manage the Blob Storage Lifecycle

① Key Concepts

Lifecycle Management Policy Structure

A lifecycle policy is a JSON document containing an array of rules. Each rule has a name, a filter set (what blobs it applies to) and an action set (what to do). Policies run once per day.

Policy Component Property Description Example Value
Rule — Filter prefixMatch Array of container/prefix strings to match blob names ["logs/2023", "archive/"]
Rule — Filter blobTypes Blob types this rule applies to: blockBlob, appendBlob ["blockBlob"]
Rule — Filter blobIndexMatch Match blobs by their blob index tags (key-value pairs) [{"name": "project", "op": "==", "value": "alpha"}]
Rule — Action tierToCool Move blob to Cool tier after N days daysAfterModificationGreaterThan: 30
Rule — Action tierToCold Move blob to Cold tier after N days daysAfterModificationGreaterThan: 90
Rule — Action tierToArchive Move blob to Archive tier after N days daysAfterModificationGreaterThan: 180
Rule — Action delete Delete the blob after N days daysAfterModificationGreaterThan: 2555
Rule — Action (snapshot) snapshots: { tierToArchive } Apply action to snapshots (must be explicitly declared) daysAfterCreationGreaterThan: 90

Condition Types

Condition Measures From Requires
daysAfterModificationGreaterThan Last modified timestamp Always available; default condition
daysAfterLastAccessTimeGreaterThan Last access timestamp Access tracking must be enabled on the storage account
daysAfterCreationGreaterThan Creation timestamp Used for snapshots and blob versions
Multiple actions can exist in one rule — the most cost-effective wins
If a blob qualifies for multiple actions in the same rule (e.g., both tierToCool and tierToArchive), the lifecycle engine applies the action that results in the lowest storage cost — typically the deepest tier. This is important: you don't need separate rules to cascade through Hot → Cool → Archive; a single rule with multiple daysAfterModificationGreaterThan thresholds works.
Blob index tags can filter lifecycle rules without relying on naming conventions
Instead of filtering by prefix (which requires you to organize blobs by name), you can attach blob index tags (e.g., retention=7years) and write lifecycle rules that match on those tags. This decouples storage organization from lifecycle behavior and is more flexible when blobs from different prefixes share the same retention policy.
② Exam Gotchas
Lifecycle policies do NOT apply to Premium storage accounts
This is one of the most commonly tested limitations. Premium Block Blob, Premium Page Blob, and Premium File Share accounts do not support lifecycle management policies at all. If a question says the customer is using a Premium account and asks how to automatically tier blobs to Archive, the correct answer is that this is not possible — the account type must be changed to GPv2.
Snapshots are NOT included in lifecycle actions unless explicitly declared
Lifecycle rules for baseBlob do not automatically cascade to snapshots or blob versions. To apply a lifecycle action (tier or delete) to snapshots, you must include a "snapshot": {} block within the rule's actions. Similarly, blob versions require a "version": {} block. Forgetting this is a common exam trap — if a question asks why snapshots are not being archived by a lifecycle policy, the answer is the missing snapshot block.
Early deletion fees apply even when lifecycle policy moves the blob
When a lifecycle rule automatically moves a blob from Cool to Archive, the early deletion fee still applies if the blob has been in Cool for fewer than 30 days. The policy engine does not waive fees based on the fact that the movement was policy-driven. Similarly, deleting a Cool blob via a lifecycle policy before 30 days incurs a prorated fee. Planning policies must account for minimum durations to avoid unexpected charges.

④ CLI / SDK Patterns
# Create or replace a lifecycle management policy from a JSON file
az storage account management-policy create \
  --account-name mystorageacct \
  --resource-group myRG \
  --policy @lifecycle-policy.json   # Must be a JSON file path prefixed with @

# View current lifecycle policy
az storage account management-policy show \
  --account-name mystorageacct \
  --resource-group myRG

# Delete lifecycle policy
az storage account management-policy delete \
  --account-name mystorageacct \
  --resource-group myRG
// lifecycle-policy.json — full example policy
{
  "rules": [
    {
      "name": "tiering-rule",          // Rule name must be unique within the policy
      "enabled": true,
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],   // Only applies to block blobs
          "prefixMatch": ["logs/"]      // Only blobs with names starting with logs/
        },
        "actions": {
          "baseBlob": {
            "tierToCool": {
              "daysAfterModificationGreaterThan": 30   // Move to Cool 30+ days after last modify
            },
            "tierToCold": {
              "daysAfterModificationGreaterThan": 90   // Move to Cold 90+ days after last modify
            },
            "tierToArchive": {
              "daysAfterModificationGreaterThan": 180  // Move to Archive 180+ days after last modify
            },
            "delete": {
              "daysAfterModificationGreaterThan": 2555 // Delete after 7 years (2555 days)
            }
          },
          "snapshot": {
            "delete": {
              "daysAfterCreationGreaterThan": 90       // Delete snapshots older than 90 days
            }
          },
          "version": {
            "tierToArchive": {
              "daysAfterCreationGreaterThan": 60       // Archive old versions after 60 days
            }
          }
        }
      }
    },
    {
      "name": "tag-based-rule",
      "enabled": true,
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "blobIndexMatch": [
            {
              "name": "retention",     // Blob index tag key
              "op": "==",
              "value": "short-term"    // Blob index tag value
            }
          ]
        },
        "actions": {
          "baseBlob": {
            "delete": {
              "daysAfterLastAccessTimeGreaterThan": 7  // Requires access tracking enabled on account
            }
          }
        }
      }
    }
  ]
}
⑤ Wrong-Answer Traps
Trap: "Creating a lifecycle policy immediately moves blobs to Archive"
Lifecycle policies run once per day. Even if a policy is configured with a condition of daysAfterModificationGreaterThan: 0, the policy engine will not process blobs until its next scheduled run (up to 24 hours after creation). Additionally, conditions must actually evaluate to true — if no blobs have been modified more than N days ago, no blobs move. There is no way to trigger a manual immediate execution of a lifecycle policy.
Trap: "You can pass lifecycle policy rules as --rule arguments on the CLI"
The az storage account management-policy create command requires the policy to be a complete JSON document passed via --policy @filename.json (file reference) or --policy '{...}' (inline JSON string). There are no individual --rule, --action, or --filter flag arguments. Attempting to build the policy from CLI flags is not supported — the entire policy must be expressed as JSON.
Trap: "A lifecycle rule that moves blobs from Hot to Cool at 30 days will also tier snapshots"
Actions defined under "baseBlob" do not affect snapshots or blob versions. Snapshots require their own "snapshot": {} action block, and blob versions require a "version": {} block. This is a frequent source of confusion — customers configure rules and then discover their snapshots are consuming storage in Hot tier indefinitely because the snapshot section was omitted from the policy.
⑥ Practice Q&A
1
A developer creates a lifecycle management policy for a GPv2 storage account. The policy has a rule that tiers blobs to Archive after 180 days. The developer then notices that blob snapshots are still in Hot tier after 200 days. What is the most likely cause?
▼
✓ The policy's rule does not include a "snapshot" action block
Lifecycle actions under baseBlob only apply to the base blob — they do not cascade to snapshots or versions. To tier snapshots, the rule must include a separate "snapshot" key within the actions object:
  • Add "snapshot": { "tierToArchive": { "daysAfterCreationGreaterThan": 180 } } to the rule's actions block
  • Snapshots use daysAfterCreationGreaterThan (not daysAfterModificationGreaterThan) because snapshots are immutable and have no modification date
  • The same principle applies to blob versions — they need a separate "version" block
2
You are designing a lifecycle policy for blobs that should be accessible by the blob access pattern rather than their age. Blobs not accessed in 14 days should be moved to Cool tier. What must be enabled on the storage account first?
▼
✓ Last access time tracking must be enabled on the storage account
The condition daysAfterLastAccessTimeGreaterThan requires Azure to track the last time each blob was accessed (read). This tracking is not enabled by default because it adds a small overhead to every read operation.
  • Enable via portal: Storage account → Lifecycle management → Enable access tracking
  • Enable via CLI: az storage account update --enable-last-access-tracking true
  • Without this setting enabled, the daysAfterLastAccessTimeGreaterThan condition is always evaluated as false and no blobs will be tiered by that condition
  • This is distinct from daysAfterModificationGreaterThan, which is always available without any extra configuration
3
A lifecycle policy moves a blob from Hot to Cool tier on day 10. The blob was originally uploaded 10 days ago. How many days of Cool tier storage will be charged?
▼
✓ 30 days — the full minimum retention period is charged
The Cool tier has a 30-day minimum storage duration. When a blob is moved to Cool, the billing clock starts. If the blob is deleted or moved before the 30-day minimum elapses, you are charged for the remaining days as an early deletion fee.
  • In this scenario: the blob enters Cool on day 10; if it's moved or deleted on day 20, you owe 10 more days of Cool storage as an early deletion fee
  • The same principle applies to Cold (90 days) and Archive (180 days)
  • The lifecycle policy itself doesn't waive these fees — the early deletion fee applies regardless of whether the movement was manual or policy-driven
4
An architect wants to apply different lifecycle rules to blobs based on their business context (e.g., "project=alpha" blobs should be archived sooner than "project=beta" blobs) without restructuring the container hierarchy. What feature enables this?
▼
✓ Blob index tags with blobIndexMatch filters in lifecycle rules
Blob index tags are key-value metadata stored at the blob level and indexed by the storage system (unlike user-defined metadata, which is not searchable). Lifecycle rules can filter on blob index tags using the blobIndexMatch filter:
  • Tag a blob at upload time: az storage blob upload --tags "project=alpha"
  • Update tags later: az storage blob tag set --tags "project=alpha"
  • Write separate lifecycle rules — one matching project == alpha with aggressive tiering, another matching project == beta with conservative tiering
  • This is more maintainable than prefix-based filtering when blobs from different projects are in the same container
5
A company uses a Premium Block Blob storage account for high-throughput ingestion. They want to configure a lifecycle policy to automatically delete blobs older than 90 days. What should the architect tell them?
▼
✓ This is not possible — lifecycle management is not supported on Premium storage accounts
Lifecycle management policies are only supported on GPv2 storage accounts with Standard performance. Premium accounts (Block Blob, Page Blob, File Share) do not support lifecycle policies at all.
  • Options to work around this: (a) migrate to a GPv2 account if lifecycle management is needed; (b) implement a scheduled Azure Function or Logic App that queries old blobs and deletes them via the SDK; (c) use tiered architecture — ingest to Premium, then copy completed data to GPv2 for archiving
  • The architect should clarify whether the 90-day deletion requirement can be handled by an external process, or whether the account type needs to be reconsidered
6
A lifecycle rule has two actions: tierToCool after 30 days and tierToArchive after 60 days. A blob was last modified 65 days ago. Which action does the lifecycle engine apply?
▼
✓ tierToArchive — the most cost-effective (deepest) eligible action wins
When a blob qualifies for multiple actions in the same rule, the lifecycle engine applies the action that results in the lowest storage cost. Since the blob is 65 days past its last modification, it satisfies both the 30-day Cool condition and the 60-day Archive condition. Archive has a lower per-GB storage cost than Cool, so tierToArchive is applied.
  • The blob moves directly from Hot to Archive — it does not first stop at Cool
  • This means the blob immediately becomes offline (Archive) even though it never passed through Cool or Cold
  • The early deletion fee for Cool (30 days) is not triggered because the blob was never actually placed in Cool tier

M3 Work with Azure Blob Storage

① Key Concepts

SDK Client Hierarchy

The Azure Blob Storage SDK uses a three-level client hierarchy. Each client is scoped to a specific resource level and exposes relevant operations.

Client Class Scope Key Operations
BlobServiceClient Storage account List containers, create/delete containers, get account properties, generate account SAS
BlobContainerClient Container Create/delete container, set access policy, list blobs, upload blob (shortcut), generate container SAS
BlobClient Individual blob Download, set tier, set metadata, set tags, acquire lease, create snapshot, delete blob, generate blob SAS
BlockBlobClient Block blob (specialized) Upload (UploadAsync, StageBlockAsync, CommitBlockListAsync), all BlobClient operations
AppendBlobClient Append blob (specialized) Create append blob, AppendBlockAsync — no overwrite operations
PageBlobClient Page blob (specialized) Create page blob, UploadPagesAsync, ClearPagesAsync (random access)

Metadata vs. Blob Index Tags

Feature Metadata Blob Index Tags
Storage HTTP headers on the blob Indexed key-value pairs in the storage system
Key case sensitivity Keys are case-insensitive Keys are case-sensitive
Value case sensitivity Values are case-sensitive Values are case-sensitive
Cross-account search Not searchable — cannot query across blobs Searchable — use Find Blobs By Tags API to query across entire account
Lifecycle policy filtering Not supported Supported via blobIndexMatch
Maximum entries 8 KB total per blob 10 tags per blob; keys ≤128 chars, values ≤256 chars
SDK method SetMetadataAsync() / GetPropertiesAsync() SetTagsAsync() / GetTagsAsync()

Service SAS Single service (Blob, Queue, Table, or File) Storage account key No limit (key-based) Rotate account key, or use Stored Access Policy
Account SAS Multiple services in one account Storage account key No limit (key-based) Rotate account key
User Delegation SAS Blob and Data Lake Storage Gen2 only Azure AD credentials (OAuth) 7 days Revoke the user delegation key; or wait for expiry
User Delegation SAS is the most secure SAS type — prefer it when possible
Because it is backed by Azure AD credentials rather than a storage account key, it avoids the risk of key exposure. The maximum validity of 7 days limits blast radius if a token is leaked. Use a Stored Access Policy with Service SAS if you need the ability to revoke a token before expiry without rotating the account key.

Blob Leases

A lease is an exclusive write lock on a blob or container. Any write or delete operation on a leased blob requires the current lease ID.

Lease State Description Operations Allowed
Available No lease exists on the blob All operations
Leased Active lease held by a client Write/delete only with correct lease ID; reads always allowed
Expired Fixed-duration lease elapsed without renewal All operations (another client can acquire a new lease)
Breaking Break requested; waiting for break period to elapse Reads allowed; writes blocked until break completes
Broken Break period elapsed or break was immediate (period=0) All operations; another client can now acquire
Lease duration: 15–60 seconds (fixed) or -1 (infinite)
Fixed leases auto-expire if the client crashes — preventing deadlocks. Infinite leases (-1) require an explicit ReleaseLease call. Use infinite leases only when the client lifecycle is well-managed (e.g., a distributed lock manager). In exam scenarios asking how to prevent concurrent writes from two processes, the answer is usually: acquire a fixed-duration lease, pass the lease ID on all writes, and renew the lease before it expires.

Data Protection Features

Feature What it Does Retention / Duration Reversible?
Soft Delete (blob) Marks deleted blobs as soft-deleted; recoverable within retention period 1–365 days (configurable) Yes — undelete within retention window
Versioning Auto-creates a new version on every overwrite or delete Until explicitly deleted or lifecycle policy removes old versions Yes — restore any version to be the current version
Immutability (time-based) WORM: no delete or overwrite allowed during retention interval User-specified days/years; locked policies cannot be shortened No — once a locked policy's interval starts, it cannot be reduced

② Exam Gotchas
UploadAsync silently overwrites existing blobs — no error is thrown
Unlike databases where an INSERT to an existing key fails, BlobClient.UploadAsync() by default overwrites any existing blob with the same name without throwing an exception. To prevent accidental overwrites, pass an overwrite: false parameter, or use an HTTP conditional header If-None-Match: * (which fails if the blob already exists). This is a classic exam trap: "What happens when UploadAsync is called on a blob that already exists?" — the answer is the existing blob is overwritten silently.
Metadata keys are case-insensitive; metadata values are case-sensitive
When you set metadata with key Author and retrieve it with key author, you get the same value — keys are case-insensitive. However, the value John is different from john — values are case-sensitive. This asymmetry is a subtle exam point. Also: metadata is not searchable — you cannot query "find all blobs where Author=John" using metadata alone. For cross-blob searching, you must use blob index tags.
User Delegation SAS maximum validity is 7 days — not unlimited
A User Delegation SAS token is backed by a user delegation key obtained from Azure AD. The user delegation key itself has a maximum validity of 7 days, which means the SAS token's expiry cannot exceed 7 days from the delegation key's start time. If an exam question asks for a SAS that must be valid for 30 days and must be backed by Azure AD, that is not currently possible with User Delegation SAS — you would need a Service SAS or Account SAS with a Stored Access Policy for revocation.

④ CLI / SDK Patterns
// C# — Full SDK pattern: upload with conditional check, metadata, tags, and lease

using Azure.Storage.Blobs;
using Azure.Storage.Blobs.Models;
using Azure.Storage.Blobs.Specialized;

var serviceClient = new BlobServiceClient(connectionString);
var containerClient = serviceClient.GetBlobContainerClient("mycontainer");
await containerClient.CreateIfNotExistsAsync(PublicAccessType.None);

var blobClient = containerClient.GetBlobClient("report-2024.pdf");

// Upload — prevent overwrite using IfNoneMatch: * condition
var uploadOptions = new BlobUploadOptions
{
    Conditions = new BlobRequestConditions { IfNoneMatch = ETag.All }, // Fail if blob exists
};
try
{
    using var fileStream = File.OpenRead("./report-2024.pdf");
    await blobClient.UploadAsync(fileStream, uploadOptions);
}
catch (Azure.RequestFailedException ex) when (ex.Status == 409)
{
    Console.WriteLine("Blob already exists — upload skipped.");
}

// Set custom metadata (keys are case-insensitive; values case-sensitive)
var metadata = new Dictionary<string, string>
{
    { "department", "Finance" },  // Stored; retrievable but NOT searchable
    { "author", "Jane Smith" }
};
await blobClient.SetMetadataAsync(metadata);

// Set blob index tags (searchable across the storage account)
var tags = new Dictionary<string, string>
{
    { "project", "Q4-Report" },   // Tags are searchable via FindBlobsByTagsAsync
    { "status", "final" }
};
await blobClient.SetTagsAsync(tags);

// Acquire a 30-second lease to prevent concurrent writes
var leaseClient = blobClient.GetBlobLeaseClient();
BlobLease lease = await leaseClient.AcquireAsync(TimeSpan.FromSeconds(30));
Console.WriteLine($"Lease acquired: {lease.LeaseId}");

// Write using the lease ID (any write without the correct lease ID fails with 412)
var writeOptions = new BlobUploadOptions
{
    Conditions = new BlobRequestConditions { LeaseId = lease.LeaseId }
};
using var updateStream = new MemoryStream(System.Text.Encoding.UTF8.GetBytes("updated content"));
await blobClient.UploadAsync(updateStream, writeOptions);

// Release the lease when done
await leaseClient.ReleaseAsync();
// C# — Generate a User Delegation SAS (Azure AD backed, max 7 days)
using Azure.Storage.Blobs;
using Azure.Storage.Sas;
using Azure.Identity;

// Authenticate with Azure AD — no storage account key needed
var serviceClient = new BlobServiceClient(
    new Uri("https://mystorageacct.blob.core.windows.net"),
    new DefaultAzureCredential()
);

// Get a user delegation key (max 7 days from now)
var startsOn = DateTimeOffset.UtcNow;
var expiresOn = DateTimeOffset.UtcNow.AddDays(7); // Max allowed
UserDelegationKey userDelegationKey = await serviceClient.GetUserDelegationKeyAsync(startsOn, expiresOn);

// Build the SAS builder
var sasBuilder = new BlobSasBuilder
{
    BlobContainerName = "mycontainer",
    BlobName = "report-2024.pdf",
    Resource = "b",               // "b" = blob; "c" = container
    StartsOn = startsOn,
    ExpiresOn = expiresOn
};
sasBuilder.SetPermissions(BlobSasPermissions.Read); // Read-only SAS

// Generate the SAS token
var uriBuilder = new BlobUriBuilder(blobClient.Uri)
{
    Sas = sasBuilder.ToSasQueryParameters(userDelegationKey, "mystorageacct")
};
Uri sasUri = uriBuilder.ToUri(); // Share this URI — no account key exposed
# CLI — Upload, set tier, manage metadata and tags
# Upload blob and set Cool tier atomically
az storage blob upload \
  --account-name mystorageacct \
  --container-name mycontainer \
  --name report-2024.pdf \
  --file ./report-2024.pdf \
  --tier Cool \                      # GPv2 only; not available on Premium accounts
  --metadata "department=Finance" "author=Jane Smith"  # Custom metadata at upload

# Set blob index tags (searchable — unlike metadata)
az storage blob tag set \
  --account-name mystorageacct \
  --container-name mycontainer \
  --name report-2024.pdf \
  --tags "project=Q4-Report" "status=final"

# Query blobs by tag across the entire storage account
az storage blob query \
  --account-name mystorageacct \
  --query-expression "\"project\" = 'Q4-Report'"   # Find all blobs tagged with project=Q4-Report

# Acquire lease (30 seconds)
az storage blob lease acquire \
  --account-name mystorageacct \
  --container-name mycontainer \
  --blob-name report-2024.pdf \
  --lease-duration 30               # 15-60 seconds for fixed; -1 for infinite
⑤ Wrong-Answer Traps
Trap: "To search blobs by custom attribute, store the attribute in blob metadata"
Blob metadata is not indexed and cannot be searched across blobs. There is no API to query "find all blobs where metadata['department'] equals 'Finance'". The correct answer for cross-blob attribute searching is blob index tags, which are indexed by Azure Storage and support the Find Blobs By Tags operation. Metadata is only useful when you already know which blob you want and need to read its properties.
Trap: "Soft delete prevents all data loss — blobs deleted during the retention period are always recoverable"
Soft delete protects against accidental deletion by standard delete operations, but it does NOT protect against: (a) overwriting a blob with UploadAsync (the new content replaces the old; if versioning is off, the old content is lost); (b) immutability policies that are locked (delete is blocked entirely regardless of soft delete); or (c) deleting the storage account itself. For full data protection against overwrites, enable blob versioning in addition to soft delete.
Trap: "A Stored Access Policy lets you revoke any SAS token before it expires"
A Stored Access Policy can revoke SAS tokens — but only Service SAS tokens that reference that policy. A Stored Access Policy cannot revoke: Account SAS tokens (they don't support stored policies), or ad-hoc Service SAS tokens that do not reference a stored policy. User Delegation SAS tokens are revoked by revoking or expiring the user delegation key. The exam often pairs "SAS revocation" with "Stored Access Policy" — the correct answer depends on which SAS type is in scope.
⑥ Practice Q&A
1
A developer calls BlobClient.UploadAsync() on a blob that already exists with important data. No special options are passed. What happens?
▼
✓ The existing blob is silently overwritten with no error
By default, UploadAsync overwrites existing blobs without throwing an exception. This is distinct from, for example, SQL Server's INSERT, which would fail on a duplicate key. To prevent overwriting:
  • Pass overwrite: false as a parameter — this will throw a RequestFailedException (HTTP 409 Conflict) if the blob already exists
  • Use an HTTP conditional header: set BlobRequestConditions { IfNoneMatch = ETag.All } in BlobUploadOptions — fails with 412 Precondition Failed if blob exists
  • Enable blob versioning — the previous version is preserved automatically on every overwrite, even if UploadAsync silently replaces the current version
2
A security team requires that all SAS tokens for accessing blob data must be revocable without rotating the storage account key. Which approach satisfies this requirement for short-lived tokens up to 7 days?
▼
✓ User Delegation SAS — revoke by deleting/expiring the user delegation key
User Delegation SAS is backed by Azure AD credentials and a user delegation key (not the storage account key). Revoking a user delegation key (via BlobServiceClient.RevokeUserDelegationKeysAsync()) immediately invalidates all SAS tokens derived from that key — without affecting the storage account key or other SAS tokens.
  • Maximum validity is 7 days — sufficient for the stated requirement
  • Alternatively, a Service SAS linked to a Stored Access Policy can be revoked by deleting or modifying the stored policy — this works without rotating the account key too
  • Account SAS does not support Stored Access Policies and can only be revoked by rotating the account key
3
Two microservices might simultaneously update the same blob, causing a race condition. Which Azure Blob Storage feature should be used to serialize these writes, and what happens if a service tries to write without the lock?
▼
✓ Blob lease — the write without the lease ID fails with HTTP 412 Precondition Failed
A blob lease provides an exclusive write lock. The process that holds the lease passes its LeaseId with every write operation via BlobRequestConditions.LeaseId.
  • Any write or delete operation on a leased blob that does NOT include the correct lease ID fails with HTTP 412 Precondition Failed
  • Read operations always succeed regardless of lease status — leases only gate writes and deletes
  • Use a fixed-duration lease (15–60 seconds) so the lock auto-expires if the holding service crashes, preventing indefinite deadlock
  • The holding service must renew the lease before it expires using leaseClient.RenewAsync() if the operation takes longer than the lease duration
4
A developer sets blob metadata with key "Source" and value "Azure". Later, another developer retrieves the metadata using key "source" (lowercase). What value is returned?
▼
✓ "Azure" — metadata keys are case-insensitive; the correct value is returned
Blob metadata key lookup is case-insensitive. Whether you retrieve using "Source", "source", or "SOURCE", you get the same value. However, metadata values are case-sensitive:
  • The value "Azure" is different from "azure" — if you store "Azure", you get back "Azure" exactly
  • This asymmetry (case-insensitive keys, case-sensitive values) is a specific exam point
  • Also remember: metadata is stored as HTTP headers on the blob and is not indexed or searchable — for searchable attributes, use blob index tags instead
5
A compliance officer requires that financial documents stored in Azure Blob Storage cannot be deleted or overwritten for exactly 7 years, and this policy must be enforced even by storage administrators. Which feature should be used?
▼
✓ Time-based retention immutability policy (WORM) — locked configuration
Time-based immutability with a locked policy prevents any user — including storage administrators and the account owner — from deleting or overwriting blobs during the retention interval.
  • Locking the policy is irreversible for shortening: once locked, the retention interval can only be extended, never shortened or disabled before expiry
  • Unlocked immutability policies are in a "testing" state and can be modified or deleted — they do NOT satisfy regulatory WORM requirements
  • Legal hold is an alternative when you don't know the exact duration — it blocks modifications indefinitely until an authorized user removes the hold tag
  • Soft delete and versioning alone do NOT satisfy WORM requirements because an administrator could still delete the soft-deleted blobs or purge versions
6
A developer needs to find all blobs across an entire storage account that have the tag "status=pending-review". Which SDK method or REST operation enables this cross-blob search?
▼
✓ BlobServiceClient.FindBlobsByTagsAsync() with a tag filter expression
Blob index tags are the only blob attribute type that is indexed for search. The FindBlobsByTagsAsync() method on BlobServiceClient accepts an OData-style filter expression:
  • Example: await serviceClient.FindBlobsByTagsAsync("\"status\" = 'pending-review'")
  • This searches across all containers in the storage account and returns a paginated list of matching blob items with their container names
  • Metadata cannot be used for this — there is no FindBlobsByMetadata API because metadata is not indexed
  • Tags support compound queries: "\"project\" = 'alpha' AND \"status\" = 'pending-review'"
  • Requires the caller to have the Storage Blob Data Reader role or equivalent RBAC permission
7
What is the difference between soft delete and blob versioning, and when should you enable both?
▼
✓ They protect against different failure modes — enable both for complete accidental-change protection
Soft delete protects against accidental deletion: when a blob is deleted, Azure keeps a soft-deleted copy for the configured retention period (1–365 days). You can undelete it during that window.
  • Does NOT protect against overwrites — if UploadAsync silently replaces the blob content, soft delete does not capture the previous content (the blob was never "deleted," it was just overwritten)
Blob versioning protects against both accidental deletions and overwrites: every time a blob is overwritten or deleted, Azure automatically creates a new version preserving the previous state.
  • Versions are stored permanently until explicitly deleted or a lifecycle policy removes them
  • You can restore any previous version to become the current version
Enable both together for complete protection: soft delete catches gaps if versioning is disabled for any reason, and versioning catches overwrites that soft delete misses. On the exam, if asked for complete point-in-time recovery for blob data including overwrites, the answer is blob versioning (not just soft delete).
8
A developer uses BlobContainerClient to list blobs. Some blobs have tags set, but GetProperties() on each blob returns no tags. What additional call is needed to retrieve blob index tags?
▼
✓ Call BlobClient.GetTagsAsync() — tags are not returned by GetPropertiesAsync()
Blob index tags and blob metadata/properties use separate API calls:
  • blobClient.GetPropertiesAsync() returns system properties (content type, length, ETag, tier) and user-defined metadata — but NOT blob index tags
  • blobClient.GetTagsAsync() returns the blob index tags dictionary separately
  • When listing blobs via containerClient.GetBlobsAsync(BlobTraits.Tags), pass BlobTraits.Tags to include tags in the listing response and avoid a separate round-trip per blob
  • This separation matters for performance at scale: if you are listing 10,000 blobs and need their tags, use BlobTraits.Tags in the list call rather than making 10,000 individual GetTagsAsync calls
