Topic 05

01 Azure Container Registry (ACR)

① Key Concepts

Registry SKUs

ACR comes in three service tiers. The tier you choose determines storage quota, webhook limits, and which enterprise features are available. This is a heavily tested area — know the limits cold.

Feature Basic Standard Premium
Storage included 10 GB 100 GB 500 GB
Webhooks 2 10 500
Geo-replication No No Yes
Private endpoints No No Yes
Customer-managed keys (CMK) No No Yes
Dedicated data endpoints No No Yes
Zone redundancy No No Yes
Content trust (image signing) No No Yes
💡
Premium = All the enterprise features
The exam reliably tests "which SKU supports X?" where X is one of: private endpoints, geo-replication, customer-managed keys, or dedicated data endpoints. The answer is always Premium. Basic and Standard are functionally similar, differing only in storage and webhook count.

Authentication Methods

ACR supports several authentication options. Understanding which is recommended for which scenario is exam-critical.

Method Use Case Recommended?
Managed Identity Azure services (ACI, AKS, App Service) pulling images Yes — preferred for Azure-hosted workloads
Service Principal CI/CD pipelines, non-Azure workloads, headless automation Yes — preferred for automated/headless scenarios
Admin user Legacy, quick testing, Docker login from local workstation No — single credential, not auditable, disabled by default
Individual Azure AD token Interactive developer login via az acr login Yes — for human developers, short-lived token

ACR Tasks

ACR Tasks allow you to build, test, and patch container images in the cloud without needing a local Docker installation. There are three categories of tasks:

  • Quick task: triggered on-demand via az acr build. Sends a build context to ACR and builds the image remotely. No local Docker daemon required.
  • Automatically triggered task: responds to a source code commit (GitHub/Azure Repos webhook) or base image update. Enables automated rebuild when a base image changes.
  • Scheduled task: runs on a defined cron schedule. Useful for periodic security scanning or nightly builds.
  • Multi-step task: defined in a YAML file. Chains build, push, test, and conditional steps. Supports parallel and dependent step execution.
💡
Geo-replication behaviour
With a geo-replicated Premium registry, a single docker push automatically replicates the image to all configured replica regions. Pull requests are served from the nearest replica — but this only works automatically when the client's DNS resolves to the regional endpoint. The global registry endpoint always works but may incur cross-region latency.
② Exam Gotchas
🚨
Geo-replication: push replicates automatically, pull does NOT guarantee closest replica
When you push an image to a geo-replicated ACR, Azure replicates the image to all replica regions automatically — you do not need to push to each region separately. However, when pulling, the client is served from the nearest replica only if DNS routing is configured. If the client always uses the global registry FQDN (e.g., myacr.azurecr.io), geo-replication still works but may not reduce pull latency. The exam distinguishes between "automatic replication on push" (always true) and "latency reduction on pull" (requires DNS/routing).
🚨
Admin user: disabled by default, not "enabled by default"
The ACR admin user is disabled by default when you create a registry. It provides a single username and two passwords (for rotation). Because it is a shared credential with no per-user audit trail, Microsoft explicitly recommends against it for production. Exam questions often frame scenarios where "you need ACI to pull from ACR" and list admin user as an option — the correct answer is managed identity or service principal, not admin user, even though admin user technically works.
🚨
ACR Tasks do not require a local Docker daemon
az acr build uploads the build context (Dockerfile and source) to ACR and builds the image entirely in the cloud. There is no requirement to have Docker Desktop or any local container runtime installed. This is a deliberate distractor — the exam may describe a developer scenario where Docker is unavailable and ask "which approach allows them to build and push an image?" The answer is ACR Tasks / az acr build.

④ CLI / SDK Patterns

Create a registry and build an image remotely

# Create a registry (Premium SKU for geo-replication + private endpoints)
az acr create \
  --resource-group myRG \
  --name myacr \
  --sku Premium           # Basic | Standard | Premium

# Build an image remotely in ACR — no local Docker needed
# The '.' is the build context (current directory containing Dockerfile)
az acr build \
  --image myimage:v1 \   # tag for the resulting image
  --registry myacr \     # target registry name or login server
  .                      # build context path

# Import an image from Docker Hub into ACR (no local pull/push needed)
az acr import \
  --name myacr \
  --source docker.io/library/nginx:latest \
  --image nginx:latest

# Enable the admin user (NOT recommended for production)
az acr update --name myacr --admin-enabled true

# Get admin credentials
az acr credential show --name myacr

# Log in interactively (developer workstation — Azure AD token, ~3h expiry)
az acr login --name myacr

# Create a task triggered on git commit
az acr task create \
  --registry myacr \
  --name buildOnCommit \
  --image myimage:{{.Run.ID}} \
  --context https://github.com/myorg/myrepo.git \
  --file Dockerfile \
  --git-access-token <PAT>

# Run a task manually
az acr task run --registry myacr --name buildOnCommit

# Run an ad-hoc command in ACR (useful for running scripts against images)
az acr run \
  --registry myacr \
  --cmd '$Registry/myimage:v1 echo hello' \
  /dev/null

# List repositories in a registry
az acr repository list --name myacr --output table

# Show tags for a repository
az acr repository show-tags --name myacr --repository myimage
💡
az acr build vs docker build + docker push
az acr build is a single command that uploads the build context, builds the image server-side in ACR, and pushes it to the registry — all without a local Docker daemon. The exam loves to present scenarios where Docker is unavailable or you need a "cloud-native" build process.
⑤ Wrong-Answer Traps
⚠️
Trap: "Admin user is required for ACI to pull images from ACR"
FALSE. ACI can authenticate to ACR using a managed identity (system-assigned or user-assigned) or a service principal. The admin user is a legacy option that works but is explicitly not recommended. When the exam offers "enable admin user" as an option alongside "assign managed identity", the managed identity answer is correct.
⚠️
Trap: "Standard SKU supports geo-replication for high availability"
FALSE. Geo-replication is a Premium-only feature. Standard does not support geo-replication, private endpoints, customer-managed keys, or dedicated data endpoints. If an exam scenario describes a requirement for low-latency pulls across multiple regions, the required SKU is always Premium.
⚠️
Trap: "You must push to each geo-replicated region separately"
FALSE. With a geo-replicated Premium ACR, a single push to the primary registry endpoint automatically replicates the image to all configured replica regions. There is no need to push to each region's endpoint individually — ACR handles replication transparently behind the scenes.
⑥ Practice Q&A
1
Your company requires that your Azure Container Registry be accessible only via a private network endpoint, with no public internet access. Your images must also be replicated to three Azure regions for low-latency pulls. Which ACR SKU must you select?
▼
✓ Premium
Both private endpoints and geo-replication require the Premium SKU.
  • Private endpoints (Azure Private Link integration) are available only on Premium.
  • Geo-replication (replicating the registry content to multiple Azure regions) is available only on Premium.
  • Standard supports neither feature. Basic supports neither feature.
  • The exam frequently combines two Premium-only requirements in a single scenario — the answer is still Premium either way.
2
A developer's workstation does not have Docker Desktop installed. They need to build a container image from a Dockerfile in a local directory and push it to Azure Container Registry. What is the most appropriate approach?
▼
✓ az acr build --image myimage:v1 --registry myacr .
ACR Tasks (Quick Task) builds the image server-side — no local Docker daemon required.
  • az acr build uploads the local directory (build context) to ACR and builds the image entirely in the cloud.
  • The resulting image is automatically pushed to the specified registry.
  • This is the key exam scenario: "no local Docker" points directly to ACR Tasks.
  • Using docker build followed by docker push would fail because Docker is not installed.
3
You need to configure an Azure Container Instance to pull a private image from Azure Container Registry without storing credentials in the ACI configuration. What is the recommended authentication approach?
▼
✓ Assign a managed identity to the ACI and grant it AcrPull role on the registry
Managed identity is the preferred, credential-free authentication method for Azure services.
  • A system-assigned or user-assigned managed identity on the ACI is granted the AcrPull built-in role on the ACR.
  • ACI automatically uses the identity token when pulling — no passwords, secrets, or admin credentials are needed.
  • The admin user works technically but requires storing the password, is not auditable per-user, and is explicitly not recommended by Microsoft.
  • Service principals also work but require managing and rotating credentials.
4
Your team wants ACR to automatically rebuild a container image whenever the base OS image (e.g., mcr.microsoft.com/dotnet/aspnet:8.0) is updated in the upstream registry. Which ACR Tasks feature supports this?
▼
✓ Automatically triggered task with base image update trigger
ACR Tasks can monitor a base image and trigger a rebuild when it changes.
  • When creating the ACR task, include --base-image-trigger-enabled true (or the equivalent YAML setting).
  • ACR tracks the referenced base image. When Microsoft pushes a new aspnet:8.0 patch, ACR automatically triggers the dependent task.
  • This ensures downstream images always contain the latest OS patches without manual intervention.
  • Quick tasks run on demand and do not monitor for base image changes — they are not the answer here.
5
You are evaluating ACR SKUs. Your requirements are: up to 50 GB of image storage, 8 webhooks for CI/CD integration, and no need for geo-replication or private endpoints. Which SKU is the most cost-effective choice?
▼
✓ Standard
Standard provides 100 GB storage and 10 webhooks — sufficient for these requirements at lower cost than Premium.
  • Basic only supports 2 webhooks, which is below the 8 required — Basic is eliminated.
  • Standard supports 100 GB storage (covers 50 GB requirement) and 10 webhooks (covers 8 requirement).
  • Premium would also work but adds unnecessary cost when geo-replication and private endpoints are not needed.
  • Always match SKU to requirements — don't over-provision when the exam asks for the "most cost-effective" or "minimum required" tier.
6
An ACR admin user has two passwords. What is the purpose of having two passwords rather than one?
▼
✓ To enable credential rotation without downtime
The two passwords allow you to rotate credentials while keeping services running.
  • To rotate: update all clients to use password2 while password1 is still active, then regenerate password1. No service interruption occurs.
  • This is the same pattern used for Azure Storage account keys (two keys for rotation).
  • The two passwords are NOT for two different users — the admin user has a single username only.
  • Despite supporting rotation, the admin user is still not recommended for production because it is a shared credential with no per-user audit trail.
7
You want to copy a public image from Docker Hub into your private Azure Container Registry without pulling the image to a local machine first. Which command accomplishes this?
▼
✓ az acr import --name myacr --source docker.io/library/nginx:latest --image nginx:latest
az acr import copies an image directly into ACR from a remote registry, server-to-server.
  • No local Docker daemon is required — the copy happens entirely in the cloud between registries.
  • Source can be Docker Hub (docker.io/), another ACR, or any OCI-compliant registry.
  • The --image flag specifies the tag to use in the destination registry.
  • This is different from az acr build which builds an image from source code; az acr import copies an existing image.
8
You want to sign container images stored in Azure Container Registry to ensure image integrity and provenance. Which ACR SKU and feature must you use?
▼
✓ Premium SKU with Content Trust enabled
Content Trust (Docker Notary-based image signing) is a Premium-only feature.
  • Content Trust allows publishers to sign images with a private key. Consumers can verify the signature before pulling.
  • To enable: az acr config content-trust update --registry myacr --status enabled
  • Only available on Premium SKU — Basic and Standard do not support Content Trust.
  • On the client side, set DOCKER_CONTENT_TRUST=1 to enforce signed-only pulls.

02 Azure Container Instances (ACI)

① Key Concepts

What is ACI?

Azure Container Instances is the fastest way to run a container in Azure. There is no cluster to provision or manage — you create a container group and Azure runs it directly on its infrastructure. ACI is optimised for short-lived, burst, or task-oriented workloads. It is not designed for long-running stateful applications (use AKS or App Service for those).

Container Groups

A container group is the top-level resource in ACI. It is analogous to a Kubernetes pod. Multiple containers in the same group share the same:

  • Network namespace — same IP address, same localhost. Inter-container communication uses localhost:port.
  • Storage volumes — mounted volumes are shared across containers in the group.
  • Lifecycle — the group starts and stops together. Restart policy applies at the group level.
💡
Container groups share an IP — port conflicts matter
Because all containers in a group share the same IP address, two containers in the same group cannot bind to the same port number. If container A listens on port 80 and container B also tries to listen on port 80, the group will fail to start. External traffic is routed to containers by the port number specified in the group's port configuration.

Restart Policies

Policy Behaviour Typical Use Case
Always Container restarts whenever it exits (any exit code). Default policy. Long-running web servers, daemons
OnFailure Container restarts only if it exits with a non-zero exit code. Batch jobs, data processing — run once successfully, do not restart on success
Never Container is never restarted, regardless of exit code. One-shot tasks — fire and forget, do not retry even on failure

Volume Mounts

Volume Type Persistence Notes
Azure Files share Persistent — survives container restart SMB-mounted Azure Storage file share. Shared across containers in the group.
emptyDir Temporary — lost on container restart Empty directory for scratch space. Shared within the group, gone on restart.
Secret volume In-memory — not written to disk Secrets delivered as files to the container. Stored in tmpfs.
gitRepo Cloned at startup only Deprecated. Clones a git repo into a volume at container start. Do not use in new designs.

Environment Variables

ACI supports two types of environment variables, and the distinction is exam-critical:

  • Plain environment variables (--environment-variables): visible in portal, CLI output (az container show), and logs. Use for non-sensitive configuration.
  • Secure environment variables (--secure-environment-variables): values are never returned by az container show, the portal, or any management API. The key name is visible but the value is masked. Use for secrets, connection strings, passwords.
② Exam Gotchas
🚨
Secure environment variables: key is visible, value is permanently hidden
When you set a secure environment variable with --secure-environment-variables KEY=SECRET_VALUE, the key name appears in az container show output under secureEnvironmentVariables, but the value is blank/masked and can never be retrieved through the management plane. This is by design — you cannot inspect the value after creation. The exam may ask: "What happens when you run az container show on a container with a secure environment variable?" The answer is that the key is shown but the value is not.
🚨
emptyDir volumes are lost on container restart — not on group restart
An emptyDir volume is shared between all containers in the group and survives if only one container in the group restarts (the volume persists as long as the group is alive). However, if the container group itself is stopped and restarted (e.g., deallocated and reallocated), the emptyDir content is lost. For durable scratch storage that survives group restarts, use an Azure Files volume mount instead.
🚨
ACI has no built-in autoscaling — it cannot scale out dynamically
ACI runs a fixed container group. There is no mechanism to automatically add more container groups in response to load — ACI has no equivalent of Kubernetes HPA or App Service autoscale. For scenarios that require scaling out to handle variable traffic, ACI is the wrong choice. Use Azure Container Apps (with KEDA-based autoscaling) or AKS. The exam may describe a traffic burst scenario and ask which service automatically scales — ACI is not the answer.
③ Key Numbers & Limits
Metric Value Exam Note
Max vCPUs per container group (Linux) 4 vCPU Cannot exceed 4 vCPUs for a Linux container group
Max RAM per container group (Linux) 16 GB Shared across all containers in the group
Max vCPUs per container group (Windows) 4 vCPU Same limit as Linux
Autoscaling Not supported ACI cannot scale out automatically; use Container Apps or AKS
Persistent storage Azure Files only emptyDir and secret volumes are ephemeral; only Azure Files persists
OS types supported Linux, Windows A container group is either all-Linux or all-Windows — cannot mix
Restart policy default Always If no restart policy is specified, containers restart indefinitely
Public IP One per container group Multiple containers share the group's single IP; use different ports
④ CLI / SDK Patterns

Creating container groups with various configurations

# Basic container creation — pulls from Docker Hub
az container create \
  --resource-group myRG \
  --name mycontainer \
  --image nginx:latest \
  --dns-name-label myapp-dns \  # creates myapp-dns.<region>.azurecontainer.io
  --ports 80

# Batch job with OnFailure restart policy — restarts only if it fails
az container create \
  --resource-group myRG \
  --name batchjob \
  --image myacr.azurecr.io/processor:v2 \
  --restart-policy OnFailure \  # Always | OnFailure | Never
  --environment-variables QUEUE_NAME=orders BATCH_SIZE=100

# Secure environment variables — values masked in portal and CLI output
az container create \
  --resource-group myRG \
  --name secureapp \
  --image myacr.azurecr.io/app:latest \
  --environment-variables APP_ENV=production \
  --secure-environment-variables DB_PASSWORD=s3cr3t API_KEY=abc123
  # NOTE: DB_PASSWORD and API_KEY values will NOT appear in az container show

# Mount an Azure Files share as a persistent volume
az container create \
  --resource-group myRG \
  --name fileapp \
  --image myacr.azurecr.io/app:latest \
  --azure-file-volume-account-name mystorageaccount \
  --azure-file-volume-account-key <storagekey> \
  --azure-file-volume-share-name myfileshare \
  --azure-file-volume-mount-path /app/data

# Pull from private ACR using managed identity
az container create \
  --resource-group myRG \
  --name privateapp \
  --image myacr.azurecr.io/myimage:latest \
  --acr-identity /subscriptions/<subId>/resourceGroups/myRG/providers/Microsoft.ManagedIdentity/userAssignedIdentities/myidentity \
  --assign-identity /subscriptions/<subId>/resourceGroups/myRG/providers/Microsoft.ManagedIdentity/userAssignedIdentities/myidentity

# Check container logs
az container logs --resource-group myRG --name mycontainer

# Show container details — note secure env var values are empty
az container show --resource-group myRG --name secureapp --query "containers[0].environmentVariables"

# Execute a command inside a running container
az container exec --resource-group myRG --name mycontainer --exec-command "/bin/sh"
⑤ Wrong-Answer Traps
⚠️
Trap: "ACI is suitable for long-running stateful applications"
FALSE. ACI is designed for short-lived, burst, or task workloads. While ACI containers can technically run indefinitely, there is no built-in autoscaling, no rolling update mechanism, and the storage model is not designed for stateful workloads (only Azure Files mounts provide durability). For stateful long-running applications, use AKS (for container orchestration) or App Service (for web apps with persistent storage). The exam will describe a stateful or highly-available workload and offer ACI as a distractor.
⚠️
Trap: "Two containers in the same ACI group can both listen on port 80"
FALSE. All containers in an ACI container group share a single IP address. Port numbers within the group must be unique — two containers cannot bind to the same port. The exam may present a multi-container group scenario where one sidecar conflicts on a port. The correct solution is to change one container's listening port so there is no conflict within the group.
⚠️
Trap: "You can mix Linux and Windows containers in the same ACI container group"
FALSE. An ACI container group must be entirely Linux or entirely Windows. You cannot mix OS types within a single group. This is different from AKS, which supports mixed node pools. If an exam scenario requires a Linux container and a Windows container working together under one IP, ACI cannot satisfy that requirement in a single group.
⑥ Practice Q&A
1
You are running a data processing batch job in ACI. The job should restart automatically if it crashes (non-zero exit), but should NOT restart once it completes successfully (exit code 0). Which restart policy should you configure?
▼
✓ OnFailure
OnFailure restarts the container only when it exits with a non-zero exit code.
  • Always (default) would restart the container even after a successful completion — wrong for a batch job.
  • OnFailure matches exactly: retry on crash, stop on success.
  • Never would not retry even on crash — wrong if you want crash recovery.
  • The exam pattern: "batch job, retry on failure, stop on success" = OnFailure.
2
You create an ACI container with a database connection string passed as a secure environment variable. A colleague runs az container show to inspect the container configuration. What will they see regarding the secure environment variable?
▼
✓ The key name is visible but the value is empty/masked
Secure environment variable values are permanently hidden from the management plane.
  • az container show returns the variable name under secureEnvironmentVariables but the value field is blank.
  • This applies to the portal, CLI, ARM API, and any management SDK — the value cannot be retrieved after creation.
  • The value IS available inside the running container as a normal environment variable — the container process can read it.
  • This is the correct design for secrets: the workload can use it, but management tooling cannot expose it.
3
An ACI container group contains two containers: a primary application container and a logging sidecar. The primary uses port 80 and the sidecar needs to expose a metrics endpoint. Which port can the sidecar NOT use?
▼
✓ Port 80 — it is already bound by the primary container
Containers in a group share a single IP and cannot bind the same port twice.
  • The group has one IP address. Port 80 is claimed by the primary container.
  • The sidecar must use a different port (e.g., 9090, 8080, 443).
  • This is the same constraint as two processes on the same Linux host competing for a port.
  • To expose the sidecar's metrics externally, include its port in the container group's ports definition.
4
You need temporary scratch storage shared between two containers in an ACI group during a single run. The data does not need to survive a group restart. Which volume type should you use?
▼
✓ emptyDir volume
emptyDir provides an ephemeral shared directory for the lifetime of the container group.
  • Both containers in the group can read and write to the same emptyDir mount path.
  • The data is stored in memory (tmpfs) and is lost when the group is stopped/restarted.
  • No Azure Storage account or SMB configuration is needed — it's purely in-memory.
  • If persistence across restarts is required, use Azure Files instead.
5
Your application running in ACI must persist data to a file share that survives container group restarts and can be accessed by other Azure services. Which volume mount type is appropriate?
▼
✓ Azure Files volume mount
Azure Files is the only ACI volume type that provides durable, persistent storage.
  • Azure Files mounts an SMB file share from Azure Storage into the container at a specified path.
  • Data written to the mount path persists after the container group is stopped, restarted, or deleted.
  • The same share can be mounted by other services (VMs, other ACI groups, App Service) simultaneously.
  • emptyDir and secret volumes are ephemeral and not suitable for durable persistence.
6
You need to run a containerised web application in Azure. The application must automatically scale from 0 to 50 container replicas based on HTTP request volume, with no cluster management. ACI has been suggested. Is ACI appropriate?
▼
✓ No — use Azure Container Apps instead
ACI does not support autoscaling. Container Apps does.
  • ACI runs a fixed container group with no scale-out mechanism. You cannot configure rules to add more groups automatically.
  • Azure Container Apps supports HTTP-based autoscaling (concurrent request threshold) and can scale to 0 and back up to 300 replicas.
  • Container Apps requires no Kubernetes expertise — it abstracts cluster management, satisfying the "no cluster management" requirement.
  • AKS would also satisfy scaling needs but requires cluster management, which the scenario explicitly excludes.
7
You deploy an ACI container group and specify no restart policy. Two hours later the main container process exits with code 0. What happens?
▼
✓ The container restarts because the default policy is Always
When no restart policy is specified, ACI defaults to Always — the container restarts on any exit.
  • The Always policy does not distinguish between success (exit 0) and failure (non-zero exit).
  • A batch job that completes successfully (exit 0) with the Always policy will be immediately restarted in a loop.
  • To prevent restart after success, the correct policy is OnFailure.
  • Always remember: if a question describes a "job that keeps restarting after finishing", the fix is to change the restart policy to OnFailure or Never.

03 Azure Container Apps

① Key Concepts

Architecture Overview

Azure Container Apps is a fully managed serverless container platform built on top of Kubernetes, KEDA (Kubernetes Event-driven Autoscaling), Envoy proxy, and DAPR (Distributed Application Runtime). You do not need Kubernetes knowledge to use it — the underlying K8s cluster is entirely managed by Microsoft. You interact with Container Apps through its own resource model.

Core Resource Hierarchy

Resource Description Key Point
Environment Logical grouping of container apps All apps in an environment share the same VNET and Log Analytics workspace
Container App A deployable unit — the application itself Has ingress config, scaling rules, DAPR config, and manages revisions
Revision An immutable snapshot of the app configuration Created on every update; used for traffic splitting and rollback
Replica A running instance of a revision Autoscaler creates/removes replicas based on scaling rules

Ingress

Ingress controls how your app receives external or internal traffic. There are two modes:

  • External ingress: app is accessible from the public internet (or from anywhere if the environment has no VNET restriction). Azure provides a public FQDN.
  • Internal ingress: app is accessible only from within the same Container Apps environment (VNET). No public endpoint is exposed.
  • No ingress: app runs as a background worker with no HTTP endpoint at all (e.g., queue processors).

Revisions and Traffic Splitting

Every time you update a container app (change the container image, environment variables, scaling rules, or DAPR configuration), Azure Container Apps creates a new, immutable revision. The previous revision continues running until you change traffic weights.

Revision Mode Behaviour Use Case
Single (default) Only one active revision at a time. New revision automatically receives 100% traffic. Standard deployments with instant cutover
Multiple Multiple revisions can be active simultaneously. Traffic is split by weight percentage. Canary deployments, A/B testing, blue/green rollouts
💡
Rollback = set traffic weight to the old revision
Container Apps does not have a "rollback" button. To roll back, switch to Multiple revision mode (if not already) and use az containerapp ingress traffic set to send 100% of traffic back to a previous revision. Old revisions are retained and remain available until you explicitly deactivate or delete them.

Scaling

Container Apps uses KEDA for event-driven autoscaling. You define scaling rules that specify when to add or remove replicas:

  • HTTP scaling: scale based on the number of concurrent HTTP requests. Most common for web apps.
  • CRON scaling: scale based on a schedule (e.g., scale up at 8 AM, down at 6 PM).
  • Custom KEDA scalers: scale based on queue length (Azure Storage Queue, Service Bus), Kafka topic lag, database row count, custom metrics — any KEDA-supported scaler.
  • Scale to zero: minimum replicas can be set to 0. The app will have zero running instances (and zero cost) when idle, and will scale up on the first incoming trigger.

DAPR Integration

DAPR (Distributed Application Runtime) is an optional sidecar that provides microservice building blocks: service-to-service invocation, pub/sub messaging, state management, secret management, and bindings. It is enabled per container app, not per environment.

DAPR Setting Value Description
--dapr-enabled true / false Enable or disable the DAPR sidecar for this app
--dapr-app-id string (e.g., orderservice) Unique identifier for this app in the DAPR service discovery mesh
--dapr-app-port integer (e.g., 3000) The port your app listens on so DAPR can forward requests to it
--dapr-app-protocol http / grpc Protocol DAPR uses to communicate with your app
② Exam Gotchas
🚨
Updates create new revisions — the current revision is never modified
When you run az containerapp update, Azure Container Apps creates a brand new revision with the updated configuration. The existing revision is not modified — it is immutable by definition. In Single revision mode, traffic automatically shifts to the new revision. In Multiple mode, you must manually adjust traffic weights. The exam will ask "what happens when you update a container app?" — the answer is always "a new revision is created".
🚨
Scale to zero means zero cost but also cold start latency
When minimum replicas = 0, the app scales down to zero running instances during idle periods. The first request after the scale-to-zero event will experience a cold start — there is latency while Container Apps spins up a new replica. If the scenario requires zero cold starts (e.g., a latency-sensitive API), set minimum replicas to 1 or more. The exam tests this trade-off: scale-to-zero saves cost but introduces cold start latency.
🚨
Container Apps abstracts Kubernetes — you do NOT use kubectl or K8s manifests
Although Container Apps runs on Kubernetes internally, you do not interact with the underlying cluster via kubectl, Helm charts, or Kubernetes YAML manifests. All management is done through az containerapp CLI commands, ARM/Bicep templates, or the Azure portal. The exam may describe a scenario where a developer with K8s experience tries to apply a Deployment manifest to Container Apps — this does not work. Container Apps is a PaaS abstraction on top of K8s.

④ CLI / SDK Patterns

Container Apps lifecycle management

# Create a Container Apps environment
az containerapp env create \
  --resource-group myRG \
  --name myenv \
  --location eastus \
  --logs-workspace-id <logAnalyticsWorkspaceId> \
  --logs-workspace-key <logAnalyticsKey>

# Create a container app with external ingress and autoscaling
az containerapp create \
  --resource-group myRG \
  --name myapp \
  --environment myenv \
  --image myacr.azurecr.io/myapp:v1 \
  --ingress external \               # external | internal
  --target-port 3000 \               # port your app container listens on
  --min-replicas 0 \                 # scale to zero when idle
  --max-replicas 10 \                # maximum replica count
  --cpu 0.5 \
  --memory 1.0Gi

# Update the app — creates a NEW revision automatically
az containerapp update \
  --resource-group myRG \
  --name myapp \
  --image myacr.azurecr.io/myapp:v2   # changing image tag creates new revision

# Switch to multiple-revision mode to enable traffic splitting
az containerapp revision set-mode \
  --resource-group myRG \
  --name myapp \
  --mode multiple                    # Single | Multiple (default: Single)

# Split traffic: send 80% to stable revision, 20% to canary
az containerapp ingress traffic set \
  --resource-group myRG \
  --name myapp \
  --revision-weight myapp--v1=80 myapp--v2=20

# Rollback: route 100% traffic back to previous revision
az containerapp ingress traffic set \
  --resource-group myRG \
  --name myapp \
  --revision-weight myapp--v1=100

# List all revisions (active and inactive)
az containerapp revision list \
  --resource-group myRG \
  --name myapp \
  --output table

# Enable DAPR on an existing container app
az containerapp dapr enable \
  --resource-group myRG \
  --name myapp \
  --dapr-app-id orderservice \        # unique name for service discovery
  --dapr-app-port 3000 \              # port DAPR proxies to your app
  --dapr-app-protocol http            # http | grpc

# Add HTTP scaling rule (scale on concurrent requests)
az containerapp update \
  --resource-group myRG \
  --name myapp \
  --scale-rule-name http-rule \
  --scale-rule-type http \
  --scale-rule-http-concurrency 50    # scale up when concurrent requests exceed 50

# Add a CRON scaling rule (scale on a schedule)
az containerapp update \
  --resource-group myRG \
  --name myapp \
  --scale-rule-name business-hours \
  --scale-rule-type cron \
  --scale-rule-cron-timezone "America/New_York" \
  --scale-rule-cron-desired-replicas 5 \
  --scale-rule-cron-start-schedule "0 8 * * 1-5" \   # 8 AM Mon-Fri
  --scale-rule-cron-end-schedule "0 18 * * 1-5"       # 6 PM Mon-Fri
💡
DAPR app-id is the service discovery key
The --dapr-app-id is how other DAPR-enabled services in the same environment find and call this service. For DAPR service-to-service invocation, service A calls http://localhost:3500/v1.0/invoke/<app-id>/method/<method> through the local DAPR sidecar. The DAPR sidecar handles routing to the correct container app by app-id.
⑤ Wrong-Answer Traps
⚠️
Trap: "Container Apps uses Kubernetes YAML manifests for deployment"
FALSE. Container Apps completely abstracts the underlying Kubernetes cluster. You deploy using az containerapp create/update CLI commands, ARM/Bicep templates, or Terraform. You cannot use kubectl apply, Helm charts, or raw Kubernetes Deployment/Service YAML files to manage Container Apps. The K8s cluster exists, but Microsoft owns it — you have no access to it. This is a major differentiator from AKS.
⚠️
Trap: "To roll back a Container Apps deployment, run az containerapp rollback"
FALSE. There is no az containerapp rollback command. Rollback is achieved by manipulating traffic weights on revisions. Switch to Multiple revision mode, then use az containerapp ingress traffic set to route 100% of traffic to a previous (still-retained) revision. Old revisions persist until you explicitly deactivate or delete them — they are always available for rollback by re-routing traffic.
⚠️
Trap: "DAPR must be enabled at the environment level to work"
FALSE. DAPR is enabled and configured at the individual container app level, not at the environment level. Each app independently opts in to DAPR with its own app-id, app-port, and app-protocol. An environment can contain a mix of DAPR-enabled and non-DAPR apps. The environment hosts the DAPR control plane components, but you enable DAPR on a per-app basis.
⑥ Practice Q&A
1
You have a Container App running revision v1. You update the container image to v2. The app is in the default revision mode. What happens to traffic after the update?
▼
✓ A new revision is created and receives 100% of traffic automatically
In Single revision mode (the default), the new revision automatically receives all traffic.
  • Every update to a container app creates a new immutable revision — the existing revision is never modified.
  • In Single mode, the new revision automatically becomes the active revision and takes 100% of traffic.
  • The old revision (v1) is deactivated but retained. Its revision record remains in the list.
  • In Multiple mode, you would need to manually set traffic weights to shift traffic to v2.
2
You want to perform a canary deployment for a Container App: 10% of traffic to the new version v2, 90% to the stable version v1. What configuration change is required before setting traffic weights?
▼
✓ Switch the revision mode from Single to Multiple
Traffic splitting across multiple revisions requires Multiple revision mode.
  • Single mode (default) keeps only one active revision — you cannot split traffic in Single mode.
  • Run: az containerapp revision set-mode --mode multiple
  • Then: az containerapp ingress traffic set --revision-weight v1=90 v2=10
  • Traffic weights must total 100%. You can adjust them incrementally as confidence in v2 grows.
3
A Container App is configured with --min-replicas 0. After a period of inactivity the app scales to zero. What is the consequence of the next incoming HTTP request?
▼
✓ The request triggers a scale-up, but experiences cold start latency while a replica starts
Scale-to-zero means zero running replicas = zero cost but a cold start on the first request.
  • When min replicas = 0 and no requests arrive, Container Apps scales the app to zero instances.
  • The first request after a scale-to-zero event must wait for a new replica to start before it can be served.
  • Cold start duration depends on image pull time and application startup time.
  • To eliminate cold starts, set --min-replicas 1 — always at least one warm instance running (at the cost of continuous resource usage).
4
You want two Container Apps (an order service and a payment service) to communicate with each other using DAPR service-to-service invocation. What must be configured on each app?
▼
✓ DAPR must be enabled on both apps, each with a unique dapr-app-id and the correct dapr-app-port
DAPR service-to-service invocation requires DAPR enabled on both calling and called services.
  • Enable DAPR on orderservice: --dapr-enabled true --dapr-app-id orderservice --dapr-app-port 3000
  • Enable DAPR on paymentservice: --dapr-enabled true --dapr-app-id paymentservice --dapr-app-port 8080
  • The order service calls payment via DAPR sidecar: http://localhost:3500/v1.0/invoke/paymentservice/method/pay
  • The DAPR sidecar handles service discovery, retries, and mTLS between the two apps automatically.
  • Both apps must be in the same Container Apps environment for DAPR service invocation to work.
5
After deploying a new revision of a Container App, you discover a critical bug. You want to immediately restore the previous version. What is the correct procedure?
▼
✓ Set revision mode to Multiple (if not already), then route 100% traffic to the previous revision using az containerapp ingress traffic set
Rollback in Container Apps = reroute traffic to a previous revision.
  • Step 1: az containerapp revision set-mode --mode multiple (if currently in Single mode)
  • Step 2: Identify the previous revision name with az containerapp revision list
  • Step 3: az containerapp ingress traffic set --revision-weight <previous-revision>=100
  • The buggy new revision drops to 0% traffic. Previous revision is immediately serving all requests.
  • There is no "rollback" command — it is always done by manipulating traffic weights on revisions.
6
You are building a background worker Container App that processes messages from an Azure Service Bus queue. It should scale up when queue depth is high and scale to zero when the queue is empty. Which scaling trigger type should you configure?
▼
✓ Custom KEDA scaler — Azure Service Bus queue scaler
KEDA-based custom scalers allow Container Apps to scale based on queue depth and other event sources.
  • The worker app has no HTTP ingress (it's a background processor), so HTTP scaling is not applicable.
  • The Azure Service Bus KEDA scaler monitors queue/topic message count and adjusts replicas accordingly.
  • Set --min-replicas 0 to enable scale-to-zero when the queue is empty (zero cost at idle).
  • Configure the scaler with the Service Bus connection string and queue name, plus a messageCount threshold per replica.
  • KEDA supports many event sources: Storage Queue, Event Hubs, Kafka, Redis, PostgreSQL, and more.
7
A developer experienced with Kubernetes tries to deploy a container to Azure Container Apps by writing a Kubernetes Deployment YAML file and applying it with kubectl apply. Will this approach work?
▼
✓ No — Container Apps abstracts Kubernetes; kubectl and K8s YAML manifests are not supported
Container Apps is a PaaS abstraction on top of Kubernetes. The underlying cluster is not directly accessible.
  • You cannot use kubectl, Helm, or Kubernetes manifests to manage Container Apps resources.
  • Microsoft owns and manages the underlying K8s cluster — customers have no direct access to it.
  • The correct tools are: az containerapp CLI, ARM/Bicep templates, Terraform Azure provider, or the Azure portal.
  • If the scenario requires direct K8s API access, Helm, or custom K8s operators, the correct service is AKS (Azure Kubernetes Service), not Container Apps.
8
Two Container Apps (App A and App B) are deployed in the same environment. App B needs to be accessible only from App A and not from the public internet. Which ingress configuration should App B have?
▼
✓ Internal ingress
Internal ingress makes an app accessible only within the Container Apps environment (same VNET).
  • With internal ingress, App B gets a DNS name resolvable only from within the environment's VNET — no public endpoint is created.
  • App A can reach App B via its internal FQDN: appb.internal.<env-unique-id>.<region>.azurecontainerapps.io
  • External ingress would expose App B to the public internet — wrong for a private internal service.
  • No ingress (disabled) means App B has no HTTP endpoint at all — wrong if App A needs to call it over HTTP.
