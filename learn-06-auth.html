01 Microsoft Identity Platform

① Key Concepts

Protocols: OAuth 2.0 and OpenID Connect

The Microsoft Identity Platform is built on OAuth 2.0 (authorisation — grants access to resources) and OpenID Connect (OIDC) (authentication — proves who the user is, adds ID token on top of OAuth). Every AZ-204 auth question eventually maps back to one of these two protocols.

OpenID ConnectAuthentication — verify user identityID token (JWT)/oauth2/v2.0/authorize + /token
OIDC DiscoveryRead platform metadata—/.well-known/openid-configuration

When you register an app in Azure AD you create two distinct objects — the exam tests whether you know the difference.

💡
Single-tenant vs Multi-tenant registration
Single-tenant: signInAudience = AzureADMyOrg — only users in your tenant can sign in. Multi-tenant: AzureADMultipleOrgs — any Azure AD org. Personal + work accounts: AzureADandPersonalMicrosoftAccount. The authority URL changes accordingly: https://login.microsoftonline.com/{tenantId} for single-tenant vs https://login.microsoftonline.com/common or /organizations for multi-tenant.

💡
The .default scope shortcut
Requesting https://graph.microsoft.com/.default tells the platform to issue an access token with all the permissions that have been pre-configured on the app registration — useful in client credentials flow where you cannot prompt for incremental consent. For delegated flows you can also request individual scopes like User.Read.
② Exam Gotchas
⚠
Application permissions always require admin consent — even low-privilege ones
A user can consent to delegated permissions like User.Read themselves (subject to tenant policy). But any application permission (roles claim) requires an admin to grant consent — even if the permission seems innocuous. Exam scenario: "A developer registers an app and requests User.Read.All as an application permission. A regular user tries to consent. What happens?" Answer: The consent fails — admin consent is required for all application permissions.
⚠
Service Principal ≠ Application Object — cross-tenant scenarios
When a multi-tenant app is used in a different tenant, only a Service Principal is created in that tenant — the Application Object stays in the home tenant. If an admin deletes the Service Principal in their tenant, users in that tenant lose access — but the app registration in the home tenant is unaffected. Exam tests: "You need to revoke access for all users in Contoso tenant. What do you delete?" Answer: the Service Principal in the Contoso tenant.
⚠
scp vs roles — which claim proves delegated vs application permission?
Access tokens carry scp for delegated permissions (there is a signed-in user) and roles for application permissions (no user). If your API receives a token and needs to decide whether a user or a daemon is calling, check which claim is present. A daemon app using client credentials will have roles but no scp. A signed-in user app will have scp. Both can be in the same token if the app has both types configured, but that is unusual.

⑤ Wrong-Answer Traps
⚠
Trap: "Use Implicit flow for SPAs — it's simpler and doesn't need PKCE"
Implicit flow is deprecated for SPAs since 2019. Microsoft's recommendation is Auth Code + PKCE for all public clients including SPAs and mobile apps. Implicit flow exposes tokens in URL fragments (browser history, referrer headers) and does not support refresh tokens in all scenarios. Exam will present this as a distractor — always choose Auth Code + PKCE for SPAs.
⚠
Trap: "The Application Object in Azure AD represents the app running in a specific tenant"
The Application Object lives only in the home tenant (where you registered the app) and is the global definition. What lives in each tenant where the app is used is the Service Principal. The exam may swap these definitions as distractors. Remember: Application Object = global blueprint in home tenant; Service Principal = local instance in each consuming tenant.
⚠
Trap: "A regular user can grant admin consent if they own the app registration"
App ownership in Azure AD gives you permission to manage the application object (add redirect URIs, manage credentials) but does not give you the ability to grant admin consent. Admin consent requires the Global Administrator, Application Administrator, or Cloud Application Administrator role. Being the app's owner is irrelevant to consent authority.
⑥ Practice Q&A
1
A background service needs to read all users in Azure AD without any user signing in. Which OAuth 2.0 flow and permission type should you use?
▼
✓ Client Credentials flow + Application permissions
Client Credentials flow is the correct choice for daemon/service scenarios with no user context.
  • Client Credentials uses a client secret or certificate — no user interaction at all.
  • Application permissions (not delegated) are required because there is no signed-in user to delegate on behalf of.
  • The token will contain the roles claim (e.g., User.Read.All), not scp.
  • Admin must pre-consent to all application permissions before the daemon can run.
  • Wrong answers: Authorization Code (requires user), Device Code (requires user interaction on another device), Implicit (deprecated, requires user).
2
Your app is registered in Tenant A. A user in Tenant B tries to sign in. Azure AD creates an object in Tenant B. What type of object is created?
▼
✓ Service Principal
A Service Principal is created in Tenant B when a user or admin from that tenant first consents to the application.
  • The Application Object remains only in the home tenant (Tenant A) — it is never replicated.
  • The Service Principal in Tenant B holds the consent records, RBAC assignments, and sign-in policies for that tenant.
  • Deleting the Service Principal in Tenant B revokes access for all users in Tenant B without affecting Tenant A.
  • The app must be registered as multi-tenant (AzureADMultipleOrgs) for this to work.
3
An access token for Microsoft Graph contains a scp claim with value "User.Read Mail.Send". What does this tell you about how the token was obtained?
▼
✓ Delegated permissions — a signed-in user is present
The scp (scopes) claim is only present in tokens obtained via delegated permission flows (Auth Code, Auth Code + PKCE, On-Behalf-Of, Device Code).
  • Presence of scp means there was a signed-in user who consented to those scopes.
  • Application permission tokens carry a roles claim instead of scp.
  • Your API can use this distinction to enforce whether a user or daemon is calling: check for scp vs roles.
  • Note: both scp and roles can theoretically coexist if the app has both types, but this is rare in practice.
4
A developer wants to shorten the access token lifetime to 15 minutes for a high-security API. What is the correct approach?
▼
✓ Create a TokenLifetimePolicy and assign it to the app or service principal
TokenLifetimePolicy is the Azure AD policy object that controls token lifetimes.
  • Create the policy: az ad policy token-issuance create or via Microsoft Graph API with a tokenLifetimePolicy resource.
  • Assign it to a service principal or application using a policy assignment.
  • Access token minimum is 10 minutes; maximum is 1 day for access tokens.
  • Wrong approach: You cannot change token lifetime in the MSAL configuration on the client side — the client can only accept what the server issues.
  • Refresh token lifetimes use a separate Continuous Access Evaluation (CAE) approach in modern tenants.
5
A single-page application (SPA) needs to authenticate users and call Microsoft Graph. Which OAuth 2.0 flow is currently recommended by Microsoft?
▼
✓ Authorization Code flow with PKCE
Auth Code + PKCE (Proof Key for Code Exchange) is the current Microsoft recommendation for SPAs and all public clients.
  • PKCE adds a code verifier/challenge to prevent authorization code interception — no client secret needed.
  • Unlike Implicit flow, Auth Code + PKCE returns tokens from the /token endpoint (not URL fragment) — not exposed in browser history.
  • Supports refresh tokens (Implicit flow in browsers does not get long-lived refresh tokens).
  • In MSAL.js v2+, PublicClientApplication uses Auth Code + PKCE by default for SPAs.
  • Implicit flow answer is always wrong for SPAs in modern scenarios — it was deprecated in 2019.
6
Your web API (API A) receives a request from a client with a delegated access token. API A needs to call another API (API B) on behalf of the original user. Which OAuth 2.0 flow should API A use?
▼
✓ On-Behalf-Of (OBO) flow
The On-Behalf-Of flow allows a middle-tier web API to propagate the user's identity downstream to another API.
  • API A presents its incoming user access token + its own client credentials to the identity platform.
  • The platform issues a new access token for API B, scoped to API B's permissions, but still representing the original user (scp present, user identity preserved).
  • API A must be a confidential client (has client secret or cert) to use OBO.
  • The user must have consented to the scopes that API A is requesting for API B.
  • Wrong answer: Client Credentials flow — this would lose the user context and produce an app-only token.
7
An Azure CLI tool runs on an IoT device with no browser available. The user must authenticate. Which OAuth 2.0 flow is appropriate?
▼
✓ Device Code flow
Device Code flow is designed for input-constrained devices (smart TVs, IoT devices, CLIs without a browser).
  • The device calls the /devicecode endpoint and receives a device_code and a short URL + user code.
  • It displays the URL and code to the user, who then goes to that URL on a browser-capable device and enters the code.
  • The device polls the token endpoint until the user completes sign-in.
  • This is a delegated flow — a real user authenticates, so the resulting token has scp claims.
  • Wrong answer: Client Credentials — that would not involve a real user and would need application permissions.
8
A developer calls az ad app permission add --api-permissions <guid>=Role. Does this grant the permission immediately?
▼
✓ No — it only adds the permission to the required permissions list; admin consent must be granted separately
Adding a permission via CLI (or portal "Add a permission") only declares that the app needs it — it does not grant it.
  • For application permissions (=Role): you must additionally run az ad app permission admin-consent --id <appId> or grant consent in the portal.
  • For delegated permissions (=Scope): consent is granted at runtime when the user (or admin) signs in and approves the permissions prompt.
  • Until consent is granted, the token will not include the permission in scp or roles.
  • This is a common cause of "insufficient_scope" or "403 Forbidden" errors in freshly configured apps.

① Key Concepts

PublicClientApplication vs ConfidentialClientApplication

MSAL's two root client types map directly to the OAuth 2.0 public vs confidential client distinction. Choosing the wrong one is a common exam trap.

💡
Always call AcquireTokenSilent first — the correct pattern
The canonical MSAL pattern is: (1) call AcquireTokenSilent, (2) catch MsalUiRequiredException, (3) fall back to AcquireTokenInteractive. Skipping the silent step wastes a round-trip and forces unnecessary user prompts. The token cache handles expiry and refresh token redemption automatically.

Token Cache

MSAL maintains an in-memory token cache that automatically handles access token refresh (using the refresh token) before expiry. The cache is per-application-instance by default.

Web apps (ASP.NET Core)In-memory (not shared across instances)IDistributedTokenCache backed by Redis or SQL
Daemon / functionsIn-memoryDistributed cache for multi-instance scale-out
SPAs (MSAL.js)sessionStorage (default) or localStoragesessionStorage for single-tab, localStorage for cross-tab persistence

② Exam Gotchas
⚠
AcquireTokenSilent can fail even when the user previously signed in
AcquireTokenSilent throws MsalUiRequiredException if the refresh token has expired or been revoked; the user's password changed; an admin revoked the session; or Conditional Access policies now require MFA. You must catch MsalUiRequiredException and fall back to interactive — do not assume silent always works because the user previously consented. The exam will present a broken scenario where the developer only calls silent and never handles the exception.
⚠
Web apps need distributed token cache — in-memory cache breaks on scale-out
If a web app is deployed to multiple instances (e.g., App Service with auto-scale), each instance has its own in-memory MSAL cache. When a request hits a different instance than the one that acquired the token, AcquireTokenSilent finds no token and forces a re-login. Fix: register services.AddDistributedTokenCaches() with a Redis or SQL backing store using Microsoft.Identity.Web. Exam scenario: "Users are intermittently forced to re-login after you scaled App Service to 3 instances. Root cause?"
⚠
AcquireTokenForClient requires .default scope and application permissions — not delegated scopes
AcquireTokenForClient (client credentials) fails with AADSTS70011 if you pass individual delegated scopes like User.Read. You must use https://graph.microsoft.com/.default. Additionally, the app must have application permissions (not delegated permissions) configured and admin-consented. The resulting token carries roles, not scp. An API that validates scp will return 403 to a daemon calling with application permissions.

④ CLI / SDK Patterns
// ConfidentialClientApplication — web app Authorization Code exchange
var confidentialApp = ConfidentialClientApplicationBuilder
    .Create(clientId)
    .WithClientSecret(clientSecret)           // or .WithCertificate(cert)
    .WithAuthority(AzureCloudInstance.AzurePublic, tenantId)
    .WithRedirectUri("https://myapp.example.com/signin-oidc")
    .Build();

// Exchange auth code for tokens after redirect
AuthenticationResult result = await confidentialApp
    .AcquireTokenByAuthorizationCode(
        scopes: new[] { "https://graph.microsoft.com/.default" },
        authorizationCode: Request.Query["code"])
    .ExecuteAsync();

string accessToken = result.AccessToken;
DateTimeOffset expiry = result.ExpiresOn;
// PublicClientApplication — desktop app with silent + interactive fallback
var publicApp = PublicClientApplicationBuilder
    .Create(clientId)
    .WithAuthority(AzureCloudInstance.AzurePublic, tenantId)
    .WithDefaultRedirectUri()                // http://localhost for desktop
    .Build();

var accounts = await publicApp.GetAccountsAsync();
AuthenticationResult result;

try {
    // ALWAYS try silent first
    result = await publicApp
        .AcquireTokenSilent(
            scopes: new[] { "User.Read" },
            account: accounts.FirstOrDefault())
        .ExecuteAsync();
}
catch (MsalUiRequiredException) {
    // Silent failed — must prompt user interactively
    result = await publicApp
        .AcquireTokenInteractive(scopes: new[] { "User.Read" })
        .WithPrompt(Prompt.SelectAccount)
        .ExecuteAsync();
}
// Daemon — ConfidentialClientApplication with client credentials
var daemonApp = ConfidentialClientApplicationBuilder
    .Create(clientId)
    .WithClientSecret(clientSecret)
    .WithAuthority($"https://login.microsoftonline.com/{tenantId}")
    .Build();

// Use .default scope — individual delegated scopes will fail here
var result = await daemonApp
    .AcquireTokenForClient(
        scopes: new[] { "https://graph.microsoft.com/.default" })
    .ExecuteAsync();
// Token has 'roles' claim (application permissions), NOT 'scp'
// ASP.NET Core — distributed token cache (Microsoft.Identity.Web)
// Program.cs
services.AddMicrosoftIdentityWebAppAuthentication(Configuration, "AzureAd")
    .EnableTokenAcquisitionToCallDownstreamApi()
    .AddDistributedTokenCaches();   // use IDistributedCache backing

// Configure Redis as the distributed cache
services.AddStackExchangeRedisCache(options => {
    options.Configuration = Configuration["Redis:ConnectionString"];
});
⑤ Wrong-Answer Traps
⚠
Trap: "PublicClientApplication stores tokens securely on disk by default"
False. MSAL stores tokens in memory only by default. Tokens are lost when the process exits. Desktop apps must implement token cache serialization using MsalCacheHelper (from Microsoft.Identity.Client.Extensions.Msal) or manual SetBeforeAccess / SetAfterAccess callbacks. MSAL's cache helper uses the OS credential store (Windows Credential Manager, macOS Keychain, Linux SecretService) for secure at-rest storage.
⚠
Trap: "Use AcquireTokenByUsernamePassword for automated testing — it avoids browser prompts"
ROPC (Resource Owner Password Credentials) should be avoided in all scenarios. It does not support MFA, Conditional Access, or federated identity — it simply fails if any are enforced. Use AcquireTokenForClient (daemon pattern) for automated/unattended scenarios. ROPC sends the user's plain-text password to the library, violating zero-trust principles. Microsoft explicitly recommends against ROPC even for testing.
⚠
Trap: "The .default scope requests only the minimum required permissions"
False. The .default scope requests all API permissions pre-configured on the app registration — it is a shortcut that says "give me everything this app is configured for." It does not select a minimal set. For client credentials flows you have no choice but to use .default because there is no user to prompt for incremental consent. For delegated flows you may request individual scopes for principle of least privilege.
⑥ Practice Q&A
1
Your ASP.NET Core web API calls Microsoft Graph using MSAL. After deploying to App Service with 4 instances, users intermittently get 401 errors from Graph. Logs show "No account or login hint was passed." What is the root cause and fix?
▼
✓ In-memory token cache not shared across instances — implement distributed token cache
Root cause: MSAL's default in-memory token cache is per-process. A follow-up request hitting a different App Service instance has no cached token and fails AcquireTokenSilent.
  • Fix: Use AddDistributedTokenCaches() in Microsoft.Identity.Web, backed by Azure Cache for Redis or Azure SQL.
  • "No account or login hint" confirms GetAccountsAsync() returned empty — confirming a cache miss on that instance.
  • All instances share the same external cache store, so any instance can serve a cached token.
  • Sticky sessions (ARR affinity) can mask the problem but is not the right fix at scale.
2
A daemon service uses AcquireTokenForClient with scope https://graph.microsoft.com/User.Read. The call fails with AADSTS70011. What is wrong?
▼
✓ Client credentials must use the .default scope — individual delegated scopes are not valid
AADSTS70011 = "The provided value for the input parameter 'scope' is not valid."
  • AcquireTokenForClient only accepts resource-level .default scopes: https://graph.microsoft.com/.default.
  • Individual scopes like User.Read are delegated — they require a user and are not valid in client credentials.
  • The app also needs User.Read.All configured as an application permission (Role type) with admin consent.
  • The token will carry roles: ["User.Read.All"], not scp.
3
Your middle-tier web API receives a user's delegated access token scoped for your API. It needs to call Azure Key Vault on behalf of the same user. Which MSAL method should it use?
▼
✓ AcquireTokenOnBehalfOf
On-Behalf-Of (OBO) flow propagates user identity from one API to another.
  • Your API (confidential client) uses the incoming access token as the "user assertion" plus its own credentials to request a new token for Key Vault.
  • Azure AD validates the chain and issues a Key Vault token still representing the original user.
  • Key Vault access is then subject to the user's RBAC roles on the vault — not the API's identity.
  • Wrong answer: AcquireTokenForClient would produce an app-only token, losing user context entirely.
4
A console tool uses PublicClientApplication. Users must not be re-prompted on every launch. What must the developer implement?
▼
✓ Token cache serialization using MsalCacheHelper or manual ITokenCache callbacks
MSAL's in-memory cache is lost when the process exits.
  • Use MsalCacheHelper from Microsoft.Identity.Client.Extensions.Msal — handles cross-platform encrypted storage.
  • Or manually implement app.UserTokenCache.SetBeforeAccess / SetAfterAccess to read/write an encrypted file.
  • On subsequent launches: call GetAccountsAsync(), find the saved account, call AcquireTokenSilent — no user prompt if refresh token is still valid.
  • Tokens must be encrypted at rest — never store in plain text.
5
AcquireTokenSilent throws MsalUiRequiredException with reason ConditionalAccessRequired. What should you do?
▼
✓ Call AcquireTokenInteractive and pass the Claims from the exception
Conditional Access requires user interaction even after a previous successful sign-in.
  • The exception's Claims property contains the CA challenge details (e.g., MFA required, device compliance needed).
  • Pass those claims: .AcquireTokenInteractive(scopes).WithClaims(ex.Claims).ExecuteAsync().
  • Without passing the claims challenge, the interactive flow may not communicate the CA requirement correctly to Azure AD.
  • After the user satisfies the CA requirement (e.g., completes MFA), a new token is issued.
6
Which MSAL application type and method should an Azure Function use to periodically sync data from Graph with no user involved?
▼
✓ ConfidentialClientApplication + AcquireTokenForClient (or use Managed Identity)
Unattended background process = daemon pattern.
  • Use ConfidentialClientApplication with client secret or certificate, AcquireTokenForClient, scope https://graph.microsoft.com/.default.
  • Application permissions (e.g., User.Read.All) must be pre-consented by an admin.
  • Preferred approach for Azure Functions: use Managed Identity + DefaultAzureCredential — avoids storing client secrets entirely.
  • Wrong: PublicClientApplication cannot use client credentials and cannot hold secrets.
7
A developer sets WithAuthority("https://login.microsoftonline.com/common"). Can a personal Microsoft account user sign in?
▼
✓ Only if the app registration has signInAudience = AzureADandPersonalMicrosoftAccount
The /common endpoint routes to the appropriate STS based on account type — but the app registration must permit personal accounts.
  • If signInAudience = AzureADMultipleOrgs, personal account sign-ins fail even with /common.
  • AzureADandPersonalMicrosoftAccount must be set on the app registration to allow MSA accounts.
  • Use /organizations to allow only work/school accounts across all tenants (excludes personal accounts).

03 Shared Access Signatures (SAS)

① Key Concepts

What a SAS Is — and What It Is Not

A Shared Access Signature is a signed URI that grants time-limited, permission-scoped access to storage resources without sharing the account key. It is a delegation mechanism for Azure Storage (Blob, Queue, Table, File) — it is not an Azure AD identity and does not appear in sign-in logs. The exam frequently pits SAS against Azure AD / Managed Identity RBAC as the "more secure" alternative.

SAS TypeScopeSigned withTypical use
User Delegation SASBlob storage onlyAzure AD (Entra ID) credentials — getUserDelegationKeyMost secure option; no account key ever generated or stored
Service SASA single service (blob, queue, table, or file) at container/blob levelStorage account keyDelegate access to one container/blob without an access policy
Account SASOne or more services, at the account levelStorage account keyBroadest scope — can grant access to multiple services, including account-level ops like List Containers

Anatomy of a SAS Token

A SAS URI is the resource URI plus a query string of parameters. The exam expects you to recognise these fields in a snippet and reason about what they restrict.

💡
Ad hoc SAS vs Stored Access Policy
An ad hoc SAS encodes all restrictions (permissions, expiry) directly in the token — the only way to revoke it early is to rotate the storage account key (which invalidates every SAS signed with that key). A SAS tied to a stored access policy (si parameter) reads its permissions/expiry from a container-level policy that can be modified or deleted at any time to instantly revoke access — without rotating the account key. Exam scenario: "You need the ability to revoke a single client's access without affecting other SAS tokens." Answer: stored access policy.

User Delegation SAS — the Azure AD-backed option

A User Delegation SAS is signed with a user delegation key, which is itself obtained using Azure AD credentials (GetUserDelegationKeyAsync) rather than the storage account key. This means:

  • The caller must have an Azure RBAC role (e.g., Storage Blob Data Contributor) that includes the Microsoft.Storage/storageAccounts/blobServices/generateUserDelegationKey/action permission.
  • The resulting SAS is bound to that Azure AD principal's permissions — it cannot exceed what the caller's RBAC role allows.
  • Maximum validity is 7 days from the time the key was requested (the account-key-based SAS types have no such Azure-imposed ceiling, only whatever expiry you set).
  • Works for Blob storage only — not available for Queue, Table, or File.

② Exam Gotchas
⚠
Rotating the account key invalidates every ad hoc SAS signed with it — including ones you meant to keep
Because ad hoc Service/Account SAS tokens are signed with the storage account key, regenerating that key (e.g., during a security rotation) immediately invalidates all SAS tokens signed with it, not just the one you intended to revoke. This is why production systems should prefer stored access policies (targeted revocation) or User Delegation SAS (no account key involved at all). Exam scenario: "After rotating the primary key, several unrelated client integrations broke. Why?" Answer: they were all using SAS tokens signed with the primary key.
⚠
A SAS token does not appear in Azure AD sign-in logs
SAS access is anonymous at the identity layer — Azure Monitor / Storage Analytics logs will show the request, but there is no Azure AD principal to audit in sign-in logs (except for User Delegation SAS, where the key generation event is logged, not each subsequent use of the resulting token). If the exam scenario requires per-user audit trails of every access, plain SAS is the wrong answer — Azure AD RBAC direct or User Delegation SAS (for the issuance step only) get you closer, but only Azure AD-authenticated calls without SAS give a full per-request identity trail.
⚠
Account SAS can grant account-level operations that Service SAS cannot
A Service SAS is scoped to operations within one service (e.g., reading blobs in a container). An Account SAS can additionally grant account-level operations such as List Containers, Get/Set Service Properties, and can span multiple services (blob + queue + table + file) in one token. If a scenario needs a client to enumerate containers in an account, a Service SAS cannot do it — only an Account SAS (with the sp=l at account scope) or Azure AD RBAC can.

⑤ Wrong-Answer Traps
⚠
Trap: "Use an Account SAS whenever you need to share a single blob"
Account SAS is the broadest, most privileged SAS type — using it to share one blob violates least privilege. A Service SAS (or better, a User Delegation SAS) scoped to that container/blob is the correct answer whenever access needs to be narrow. Reach for Account SAS only when the scenario genuinely needs account-level operations or access across multiple storage services in one token.
⚠
Trap: "SAS tokens are tied to a user identity, so you can audit exactly who accessed a blob"
False for Service/Account SAS — anyone holding the URL can use it; there is no per-use identity check. Only a User Delegation SAS ties the issuance of the token to an Azure AD principal (visible in that principal's audit trail), but subsequent uses of the resulting SAS URL are still anonymous bearer access, same as any other SAS. If the requirement is "audit every individual access by user," the answer is Azure AD RBAC direct access (no SAS at all), not any SAS variant.
⚠
Trap: "A stored access policy is required for every SAS token"
Stored access policies are optional — most SAS tokens in practice are ad hoc, with permissions and expiry baked directly into the token. Stored access policies exist specifically to solve the revocation problem; they are not a mandatory prerequisite for generating a SAS. The exam trap usually reverses this: it implies ad hoc SAS "doesn't work" without a policy, which is false.
⑥ Practice Q&A
1
A partner organisation needs read-only access to a single blob container for 3 months. Their access must be revocable at any time without affecting other integrations that also use SAS tokens signed with the same storage account. What should you configure?
▼
✓ A Service SAS tied to a stored access policy on that container
A stored access policy lets you revoke this one integration's access by deleting/modifying the policy — without rotating the account key and breaking every other ad hoc SAS token.
  • Create the policy on the container with the desired permissions and a ~90-day expiry.
  • Generate the Service SAS referencing the policy's si identifier — the token itself carries no embedded expiry/permissions.
  • Wrong: an ad hoc Service SAS would require an account key rotation to revoke early, breaking unrelated tokens.
  • Wrong: Account SAS is broader scope than needed — violates least privilege for a single-container scenario.
2
A web app needs to generate short-lived download links for blobs on behalf of signed-in users, without ever storing or handling the storage account key in application code. Which SAS approach fits, and what is required of the app's identity?
▼
✓ User Delegation SAS — the app's managed identity needs an RBAC role with generateUserDelegationKey permission
User Delegation SAS is signed with a key derived from Azure AD credentials, so the account key never needs to be present in app code or config.
  • The app (typically via Managed Identity) calls GetUserDelegationKeyAsync, which requires an RBAC role such as Storage Blob Data Contributor.
  • The resulting SAS can be no broader than the caller's own RBAC permissions.
  • Max validity from the time the key is requested is 7 days — plan short expiries (e.g., a few hours) for download links.
  • Wrong: Account/Service SAS would require embedding or retrieving the storage account key, which the requirement explicitly disallows.
3
After a suspected credential leak, an administrator regenerates the primary storage account key. Immediately afterward, several client integrations that were working the day before start failing with 403 errors. What is the most likely cause?
▼
✓ Those integrations were using ad hoc SAS tokens signed with the primary key, which are now invalid
Any Service/Account SAS signed with the primary key becomes invalid the instant that key is regenerated — regardless of the token's own se expiry.
  • This is expected behaviour, not a bug — key rotation invalidates every signature dependent on that key.
  • To avoid blast radius on future rotations: use the secondary key to sign some tokens (rotate keys independently), use stored access policies, or move to User Delegation SAS / Azure AD RBAC entirely.
  • User Delegation SAS tokens are unaffected by account key rotation since they don't depend on the account key at all.
4
A client application needs to list all containers in a storage account and upload files to any of them. Which SAS type, at minimum, supports this?
▼
✓ Account SAS
Listing containers is an account-level operation, not scoped to a single container.
  • A Service SAS is scoped to a single container/blob and cannot enumerate other containers.
  • An Account SAS with sp=l (list) at the account resource type and write permission across containers covers this.
  • User Delegation SAS is Blob-only and still scoped similarly to Service SAS semantics per resource — it does not grant account-wide list either.
  • Alternative: Azure AD RBAC role like Storage Blob Data Contributor scoped at the account level achieves the same without any SAS.
5
A compliance review requires that you be able to identify exactly which individual accessed a specific blob at a specific time. Your current design uses Service SAS tokens distributed by email. What is the correct remediation?
▼
✓ Replace SAS-based access with Azure AD RBAC direct access so each request is authenticated as a specific principal
SAS tokens (including User Delegation SAS at the point of use) are anonymous bearer credentials — the storage service cannot distinguish which human is presenting the token.
  • Only Azure AD-authenticated requests (users or managed identities calling with RBAC roles) produce per-principal entries in resource logs.
  • Switching to Azure AD RBAC means each user authenticates with their own identity — no shareable token exists at all.
  • User Delegation SAS only logs the key generation event under the requesting identity — every subsequent download using that URL is still anonymous.

04 Microsoft Graph

① Key Concepts

What Microsoft Graph Is

Microsoft Graph is the unified REST API and SDK surface for Microsoft 365, Azure AD, and related services (Users, Groups, Mail, Calendar, Teams, SharePoint, Intune, security signals). AZ-204 tests it as the standard way an app reads/writes Azure AD data and Microsoft 365 resources — always through the same OAuth 2.0 mechanics covered in Module 1, using either delegated or application permissions.

Calling Graph — the Auth Handshake

Every Graph call requires a bearer access token with aud = https://graph.microsoft.com, obtained via the flows from Module 1 and MSAL methods from Module 2. There is no separate "Graph login" — it is the same Microsoft Identity Platform token flow, just requesting Graph scopes/roles.

Throttling and Retry

Graph enforces per-app, per-tenant rate limits. Exceeding them returns HTTP 429 (Too Many Requests) with a Retry-After header specifying seconds to wait.

  • Clients must honour Retry-After and use exponential backoff — never retry immediately in a tight loop.
  • The Microsoft Graph SDKs include a built-in retry handler that does this automatically.
  • Batching multiple requests into one HTTP call ($batch) reduces the number of round trips and helps avoid throttling for bulk operations.

💡
Always paginate with @odata.nextLink — never assume one page has everything
Graph list endpoints (e.g., /users, /groups/{id}/members) page results by default (typically 100–999 depending on the resource). A response is only complete when there is no @odata.nextLink. Code that reads value[] once and stops will silently miss data in any tenant large enough to paginate — a classic exam "what's wrong with this code" trap.
② Exam Gotchas
⚠
/beta endpoints are not supported for production workloads
The /beta version of Graph exposes preview functionality that can change or disappear without notice and carries no SLA. Using it in production code is a common "what's wrong with this design" trap — the fix is to use /v1.0 and, if a needed feature is beta-only, wait for GA or find a v1.0 equivalent rather than shipping against beta.
⚠
429 without backoff causes cascading failures, not just a single failed call
If a client retries immediately after a 429 instead of respecting Retry-After, it worsens the throttling condition for itself and potentially other apps in the same tenant. The exam may describe a scenario where "calls started failing intermittently after scaling out the client" — root cause is usually naive retry-without-backoff amplifying throttling, not a Graph outage.
⚠
Application permissions on Graph can be far broader than intended — least privilege matters
User.ReadWrite.All as an application permission lets the app modify every user in the tenant, unrestricted by any delegated user context. Exam scenario: "An app only needs to update profile photos for users in the Marketing group, but was granted User.ReadWrite.All (application)." The better design uses Azure AD Administrative Units or a narrower delegated-permission-based approach with the app acting on behalf of a scoped admin, rather than a blanket application permission.

⑤ Wrong-Answer Traps
⚠
Trap: "Poll the Graph API every few seconds to detect new emails as fast as possible"
Polling wastes API quota, increases throttling risk, and adds latency between the event and detection. The correct pattern for near-real-time notification of changes is a change notification subscription (webhook) — Graph pushes a notification to your endpoint when the resource changes. Polling is only acceptable for low-frequency, low-volume scenarios where webhook infrastructure isn't justified.
⚠
Trap: "Use /beta because it has the newest features and will eventually become v1.0 with the same shape"
Beta endpoints can change shape or be removed entirely before (or instead of) reaching v1.0 — there's no guarantee of a stable migration path. Production code must target /v1.0; if a required capability only exists in beta, treat that as a blocker to raise with stakeholders, not a reason to ship against beta.
⚠
Trap: "Grant User.ReadWrite.All (application) since it's simpler than figuring out delegated scopes"
Broad application permissions are an over-privileged shortcut, not a simplification the exam rewards. When a scenario describes a need scoped to specific users or a specific action, the expected answer favours delegated permissions (scoped to what the signed-in user can already do) or the narrowest application permission that satisfies the requirement — not the broadest one available.
⑥ Practice Q&A
1
An application needs to be notified within seconds whenever a new email arrives in a shared mailbox, without polling. What Microsoft Graph feature should it use?
▼
✓ Create a change notification (webhook) subscription on the mailbox's messages resource
Change notifications push events to a registered endpoint as they happen, avoiding polling latency and quota waste.
  • POST to /subscriptions with resource set to the mail folder, changeType: "created", and a callback notificationUrl.
  • Subscriptions expire and must be renewed with a PATCH request before expirationDateTime.
  • The app must validate the clientState value on incoming notifications to confirm authenticity.
  • Wrong: polling on a timer introduces latency and burns API quota, and is more likely to trigger throttling under load.
2
Code that calls GET /users and only reads the first `value` array in the response is missing users in a large tenant. What is the bug?
▼
✓ The code does not follow @odata.nextLink to retrieve subsequent pages
Graph list endpoints are paginated by default — a single response only contains a page of results (commonly up to a few hundred, resource-dependent).
  • The response includes an @odata.nextLink URL when more results exist; absence of that field means the last page was reached.
  • The fix is to loop, requesting @odata.nextLink repeatedly until it's absent — or use the SDK's PageIterator, which does this automatically.
  • This bug is invisible in small test tenants (everything fits on one page) and only surfaces at production scale — a classic AZ-204 "why does this work in dev but not prod" trap.
3
A background service calling Microsoft Graph starts receiving HTTP 429 responses under load. The current implementation retries immediately in a tight loop. What should change?
▼
✓ Honour the Retry-After header and use exponential backoff between retries
HTTP 429 means the app has exceeded Graph's throttling limits; immediate retries make it worse.
  • The response includes a Retry-After header specifying the minimum wait in seconds before retrying.
  • Exponential backoff (with jitter) should be applied on repeated 429s beyond the first.
  • The Microsoft Graph SDKs include a default retry handler that implements this — prefer it over hand-rolled retry logic.
  • Batching related calls via $batch can also reduce the total request count and lower throttling risk.
4
An app must let a signed-in user forward one of their own calendar meetings via Graph. Which permission model is appropriate?
▼
✓ Delegated permission (e.g., Calendars.ReadWrite) — the app acts as the signed-in user
Because the action is scoped to the user's own data and performed in a user-present context, delegated permissions are correct and least-privileged.
  • The resulting token carries scp with Calendars.ReadWrite, and Graph enforces that the user can only act on their own calendar (or others they already have delegate/share access to).
  • Application permission (Calendars.ReadWrite as a role) would let the app act on any user's calendar — unnecessarily broad for this scenario and a red flag on the exam if chosen for a single-user action.
  • Use application permissions only for admin/unattended scenarios spanning multiple mailboxes.
5
A developer is building a feature against a Graph API that only exists under /beta and plans to ship it to production next week. What is the risk, and what should the team do?
▼
✓ Beta endpoints have no SLA and can change or be removed without notice — avoid shipping production dependencies on /beta
/beta is explicitly a preview surface, not a staging area guaranteed to promote unchanged to /v1.0.
  • Shape changes, field removals, or full endpoint retirement can happen with little or no warning.
  • The team should check whether an equivalent v1.0 endpoint exists, delay the feature until GA, or explicitly accept and monitor the risk if there's no alternative — but never treat beta as production-stable by default.
  • This is a frequent "spot the bad practice" style question on the exam.
