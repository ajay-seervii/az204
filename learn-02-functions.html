① Key Concepts

Hosting Plans Comparison

Plan Scale Behavior Cold Start Timeout Default / Max Max Instances VNET
Consumption Scale to zero; scale controller manages scale-out automatically Yes — can be seconds on first invocation 5 min / 10 min 200 (HTTP), 100 (non-HTTP) No (outbound only via NAT gateway workaround)
Flex Consumption Per-instance concurrency configurable; supports minimum instances; fast scale-out Reduced — minimum instances eliminate cold start for baseline load 30 min / unlimited (configurable) Configurable; high scale possible Yes — VNET integration built-in
Premium (Elastic) Pre-warmed instances; no scale-to-zero; elastic scale-out None — pre-warmed workers always ready 30 min / unlimited 100 Yes — full VNET integration
Dedicated (App Service) Manual or App Service autoscale rules; no automatic event-driven scale None — always on (requires Always On setting enabled) 30 min / unlimited Per App Service plan tier Yes — same as App Service
Consumption Instance Resources
Each Consumption plan instance is allocated 1.5 GB RAM and 1 vCPU. You cannot configure these. If your function needs more memory or CPU, move to Premium or Dedicated.

Durable Functions — Function Roles

Function Type Trigger Attribute Role Key Constraint
Orchestrator [OrchestrationTrigger] Defines the workflow logic; calls activity functions and waits for results Must be deterministic — no I/O, no DateTime.Now, no random numbers directly
Activity [ActivityTrigger] Performs the actual work — I/O, database calls, HTTP calls, computations Can be non-deterministic; called by orchestrators via CallActivityAsync
Entity [EntityTrigger] Manages stateful entities (Durable Entities / Actor model) State is persisted and can be read/written by orchestrators and clients
Client [DurableClient] Starts, queries, terminates, or sends events to orchestration instances Entry point — triggered by HTTP, queue, timer, etc. to kick off an orchestration

Durable Functions — Patterns

Pattern Description Key API / Behavior
Function Chaining Execute a sequence of functions in order, passing output of one to the next await context.CallActivityAsync() called sequentially; simple linear flow
Fan-out / Fan-in Execute multiple functions in parallel, then wait for all to complete before continuing Task.WhenAll(tasks) with multiple CallActivityAsync calls; parallelism via tasks list
Async HTTP API Client kicks off long-running operation, polls a status endpoint until complete Built-in HTTP management URLs; orchestrator exposes check-status URL automatically
Monitor Flexible recurring process in a workflow; polls until a condition is true (replaces fixed timers) context.CreateTimer() with retry loop; durable timer waits without consuming compute
Human Interaction Workflow pauses waiting for external event (approval, user input) with a timeout fallback context.WaitForExternalEvent() combined with context.CreateTimer() via Task.WhenAny()
Aggregator Accumulate event data over time into a stateful entity; replaces stateless event consumers Uses Durable Entities; events are routed to a single entity that maintains running state
Why Fan-out / Fan-in Uses Task.WhenAll
In fan-out, you build a list of Task<T> objects by calling CallActivityAsync without awaiting each one, then await Task.WhenAll(tasks). Awaiting inside the loop would serialize the calls into chaining, not parallelism.
② Exam Gotchas
Orchestrator Determinism — The Most Common Durable Functions Trap
Orchestrator functions replay from the beginning every time they are resumed after an await. If your orchestrator calls DateTime.Now, Guid.NewGuid(), or makes a direct HTTP call, the replayed value will differ from the original, causing non-deterministic behavior and potential workflow corruption. Always delegate non-deterministic operations to Activity functions, and use context.CurrentUtcDateTime instead of DateTime.UtcNow inside orchestrators.
Changing Orchestrator Code While Instances Are Running
If you deploy a new version of an orchestrator function while existing instances are mid-execution, those instances will fail on replay because the history no longer matches the new code path. The exam may describe this scenario as "orchestration instances failing after a deployment" — the correct mitigation is versioning strategies: use side-by-side deployment slots, wait for existing instances to complete, or implement versioning logic inside the orchestrator to handle both old and new code paths.
Dedicated Plan — "Always On" Is Not Automatic
In a Dedicated (App Service) plan, Azure Functions will time out and unload if the app is idle — just like a regular App Service web app. You must enable the "Always On" setting in the App Service configuration to prevent this. Without Always On, timer-triggered functions on a Dedicated plan may silently miss scheduled executions because the host is not running.

// Durable Functions — Orchestrator (C# isolated worker)
[Function("OrderOrchestrator")]
public static async Task<string> RunOrchestrator(
    [OrchestrationTrigger] TaskOrchestrationContext context)
{
    // CORRECT: use context.CurrentUtcDateTime, NOT DateTime.UtcNow
    var deadline = context.CurrentUtcDateTime.AddMinutes(30);

    // Fan-out: kick off parallel activity calls
    var tasks = new List<Task<string>>();
    foreach (var item in orderItems)
    {
        tasks.Add(context.CallActivityAsync<string>("ProcessItem", item));
    }
    // Fan-in: wait for all parallel tasks
    string[] results = await Task.WhenAll(tasks);

    // Human interaction: wait for external approval with timeout
    using var cts = new CancellationTokenSource();
    var approvalTask = context.WaitForExternalEvent<bool>("ApprovalReceived");
    var timeoutTask = context.CreateTimer(deadline, cts.Token);
    var winner = await Task.WhenAny(approvalTask, timeoutTask);
    if (winner == approvalTask) { cts.Cancel(); /* approved */ }
    else { /* timed out — escalate */ }

    return string.Join(",", results);
}
// Durable Functions — Activity (can do real I/O)
[Function("ProcessItem")]
public static async Task<string> ProcessItem(
    [ActivityTrigger] string item,
    FunctionContext executionContext)
{
    // Activities CAN call external services, use DateTime.UtcNow, etc.
    var result = await callExternalService(item);
    return result;
}

// Durable Functions — Client (starts orchestration from HTTP trigger)
[Function("StartOrchestration")]
public static async Task<HttpResponseData> HttpStart(
    [HttpTrigger(AuthorizationLevel.Anonymous, "post")] HttpRequestData req,
    [DurableClient] DurableTaskClient client,
    FunctionContext executionContext)
{
    string instanceId = await client.ScheduleNewOrchestrationInstanceAsync("OrderOrchestrator");
    // Returns built-in status polling URLs to the caller
    return await client.CreateCheckStatusResponseAsync(req, instanceId);
}
⑤ Wrong-Answer Traps
Trap: "Use Consumption plan for a time-critical scheduled function that must fire exactly on schedule"
Wrong. Consumption plan functions can experience cold starts — the host must spin up from zero before the timer trigger fires, which can delay execution by several seconds or more. For functions that must execute precisely on schedule (financial jobs, SLA-bound operations), use the Premium plan with pre-warmed instances, or the Dedicated plan with Always On enabled.
Trap: "Durable orchestrators can call external HTTP APIs directly to avoid an extra function call"
Wrong. Making direct HTTP calls (or any I/O) inside an orchestrator violates the determinism requirement. When the orchestrator replays, the HTTP call would fire again, potentially causing duplicate operations or returning a different result than the original run. Always wrap external calls in an Activity function and invoke it via CallActivityAsync.
Trap: "The Scale Controller manages scaling for all Azure Functions hosting plans"
Wrong. The Scale Controller is specific to Consumption and Premium (Elastic) plans, where it monitors event sources (queue depth, event hub offset lag, etc.) and dynamically adds or removes worker instances. Functions on a Dedicated (App Service) plan are governed by standard App Service autoscale rules — the Scale Controller plays no role there.
⑥ Practice Q&A
1
Your company runs a financial batch job as an Azure Function on the Consumption plan. The job has been running for 9 minutes and is approaching completion, but it occasionally takes 11 minutes. What is the most likely outcome, and what should you do?
▼
✓ Correct Answer: The function will be forcibly terminated at 10 minutes; migrate to Premium or Dedicated plan
Explanation: The Consumption plan enforces a hard maximum timeout of 10 minutes (600 seconds). A function that sometimes takes 11 minutes will be killed before completion. The default is 5 minutes but can be extended to 10 minutes in host.json — there is no way to exceed 10 minutes on Consumption.
  • Migrating to Premium gives unlimited timeout with pre-warmed instances (no cold start) — ideal for this scenario
  • Dedicated also supports unlimited timeout but requires Always On to be enabled
  • Do not attempt to split the job into chained 9-minute functions — that adds complexity and still risks timing issues
  • The correct host.json setting to extend Consumption timeout: "functionTimeout": "00:10:00"
2
A Durable orchestrator function calls DateTime.UtcNow to set a deadline for a human approval step. A developer reviews the code and flags it as a bug. Why is this a problem?
▼
✓ Correct Answer: Orchestrators replay from history — DateTime.UtcNow returns a different value on replay, causing non-deterministic behavior
Explanation: Durable Functions orchestrators save their execution history to Azure Storage. When an orchestrator resumes after an await, it replays the entire function from the beginning to reconstruct its in-memory state. During replay, DateTime.UtcNow returns the current wall-clock time — which is different from the original execution — making the deadline inconsistent.
  • The fix: use context.CurrentUtcDateTime (or context.CurrentUtcDateTime in the .NET SDK), which returns the deterministic, history-stamped time from the first execution
  • Same applies to Guid.NewGuid() — use context.NewGuid() instead
  • Same applies to any external HTTP or I/O call — delegate to Activity functions
  • The rule: if a value changes between runs of the same code, it cannot appear in an orchestrator
3
You need to process a large number of product catalog items in parallel, then send a summary notification only after all items are processed. Which Durable Functions pattern should you use, and what is the key API call?
▼
✓ Correct Answer: Fan-out / Fan-in pattern using Task.WhenAll()
Explanation: Fan-out / Fan-in parallelizes multiple activity calls and waits for all of them to complete before proceeding.
  • Build a List<Task<T>> by calling context.CallActivityAsync<T>() for each item without awaiting — this fires them all in parallel
  • Then await Task.WhenAll(tasks) — the orchestrator suspends until every task completes
  • Common mistake: awaiting inside the loop turns parallel fan-out into sequential chaining
  • After Task.WhenAll returns, you have all results and can call a final activity to send the summary
  • Fan-in is the "gather" half — collecting all outputs before the next step
4
An Azure Function needs to connect to resources inside a virtual network and must have no cold start. The function may run for up to 45 minutes. Which hosting plan should you choose?
▼
✓ Correct Answer: Premium (Elastic) plan
Explanation: This scenario has three requirements that eliminate other plans:
  • VNET integration — eliminates Consumption plan (no native VNET integration)
  • No cold start — eliminates Consumption plan; Premium provides pre-warmed worker instances
  • 45-minute runtime — eliminates Consumption plan (max 10 min); Premium supports unlimited timeout
  • Dedicated plan also supports VNET and unlimited timeout, but does NOT guarantee no cold start — it depends on Always On and App Service plan tier behavior; Premium is the explicit "no cold start" answer
  • Flex Consumption also supports VNET and can minimize cold starts via minimum instances, but Premium is the canonical "no cold start" plan in exam questions
5
A team deploys an updated Durable orchestrator function while 50 orchestration instances are mid-execution. Shortly after deployment, those instances begin failing with replay errors. What is the root cause?
▼
✓ Correct Answer: The orchestrator history no longer matches the new code path, causing replay failures
Explanation: When an orchestrator resumes, it replays its history to reconstruct state. If the new version of the code has different logic (added/removed activity calls, changed branching), the replay path diverges from the recorded history — the runtime throws a non-determinism exception.
  • Mitigation 1: Wait for all running instances to complete before deploying new code (drain approach)
  • Mitigation 2: Use deployment slots — route new orchestrations to the new slot while the old slot drains existing instances
  • Mitigation 3: Implement versioning logic inside the orchestrator (check a version parameter and branch accordingly)
  • This is a known operational challenge with Durable Functions — the exam tests whether you understand the replay mechanism as the root cause
6
You need a workflow that pauses and waits for a manager to approve an expense report. If no approval arrives within 48 hours, the request should be automatically escalated. Which Durable Functions pattern handles this, and what APIs are involved?
▼
✓ Correct Answer: Human Interaction pattern using WaitForExternalEvent and CreateTimer with Task.WhenAny
Explanation: The Human Interaction pattern is designed exactly for this: pause a workflow waiting for a person, with a timeout fallback.
  • context.WaitForExternalEvent<bool>("ApprovalReceived") — creates a task that resolves when an external system raises the named event
  • context.CreateTimer(deadline, cancellationToken) — creates a durable timer (no compute consumed while waiting)
  • await Task.WhenAny(approvalTask, timeoutTask) — whichever resolves first wins
  • If the timer wins, escalate; if the approval task wins, cancel the timer via the CancellationTokenSource
  • The external event is raised by a Client function calling client.RaiseEventAsync(instanceId, "ApprovalReceived", true)
  • Key exam detail: Durable timers use Azure Storage, not thread sleep — the orchestrator host can recycle with zero resource waste during the wait
7
A function on the Consumption plan processes non-HTTP queue messages. The team observes that during peak load, the function only scales to 80 instances and seems to cap there. Is this expected behavior?
▼
✓ Correct Answer: Yes — non-HTTP triggers on Consumption are limited to a maximum of 100 instances
Explanation: Azure Functions Consumption plan differentiates between HTTP and non-HTTP triggered functions for maximum scale-out:
  • HTTP triggers: can scale to 200 instances maximum
  • Non-HTTP triggers (Queue, Service Bus, Event Hubs, Timer, etc.): maximum of 100 instances
  • 80 instances during peak is below the 100 limit — this is normal; the Scale Controller adds instances adaptively based on queue depth and processing rate
  • If 100 instances are genuinely insufficient, move to the Premium plan (also 100 max, but with configurable per-instance concurrency) or Dedicated plan
  • Do not confuse the 200-instance HTTP limit with the 100-instance non-HTTP limit — the exam uses both numbers as distractors
8
Which Durable Functions pattern would you use to implement a price monitoring service that checks a stock price every 5 minutes and stops only when the price exceeds $150?
▼
✓ Correct Answer: Monitor pattern
Explanation: The Monitor pattern replaces rigid timer triggers with a flexible, stateful polling loop that can vary its interval and terminate based on a condition.
  • The orchestrator enters a loop: call an activity to check the price, evaluate the result, then call context.CreateTimer() to wait 5 minutes before checking again
  • The loop exits when the price exceeds $150 — no external signal needed
  • Contrast with a Timer trigger: that fires on a fixed schedule regardless of state, and has no built-in way to stop itself based on a condition without external coordination
  • Contrast with Human Interaction: that waits for an external event, not a polled condition
  • The Monitor pattern is resource-efficient: the durable timer suspends the orchestrator, consuming no compute between checks
  • Watch for: using context.CurrentUtcDateTime.AddMinutes(5) for the timer, NOT DateTime.UtcNow.AddMinutes(5)

Module 2 Develop Azure Functions

① Key Concepts

Triggers and Bindings

Every Azure Function has exactly one trigger. Triggers cause the function to run and provide input data. Input bindings read additional data. Output bindings write data. A function can have multiple input and output bindings.

Binding Type Direction Purpose Example
Trigger In (causes invocation) Defines what causes the function to run; exactly one per function HTTP request, Queue message, Timer schedule, Blob creation, Event Hub event
Input binding In (data only) Reads data from an external source when the function runs; does not cause invocation Read a Cosmos DB document by ID, read a Blob by name, read a Table entity
Output binding Out (write data) Writes data to an external service after function logic completes Write to a Queue, write a Blob, insert a Cosmos DB document, send an HTTP response

Binding Expressions

Binding expressions resolve values from trigger metadata at runtime using {paramName} syntax. Common expression sources include trigger input (e.g., {name} from a queue message), app settings (%AppSettingName%), and system values like {rand-guid} or {DateTime}.

Expression Resolved From Example Use Case
{queueTrigger} Queue message content (if plain string) Use queue message value as Blob path for input binding
{rand-guid} Generated GUID at runtime Output blob name: output/{rand-guid}.json ensures unique file names
{DateTime} Current UTC datetime Output blob path: logs/{DateTime}.log
%MyAppSetting% App setting value (not binding connection) Inject config values into binding path expressions
{blobTrigger} Full path of the triggering blob Log or route based on triggering blob's full path

In-Process vs. Isolated Worker Model

Aspect In-Process (.NET) Isolated Worker (.NET)
Process model Runs in the same process as the Functions host Runs in a separate worker process; communicates via gRPC
Binding configuration function.json file per function Attribute-based on method parameters; no function.json needed
Supported .NET versions .NET 6 and below only (limited to host's runtime) .NET 6, 7, 8, Framework 4.8 — decoupled from host
Startup configuration FunctionsStartup class HostBuilder with ConfigureFunctionsWorkerDefaults()
Microsoft's direction Legacy; no new feature investment Recommended for all new .NET Functions development

Configuration Files

File Purpose Deployed to Azure?
host.json Runtime-wide settings: logging, extension bundles, concurrency, retry policies, timeout Yes — deployed with function app
local.settings.json Local development configuration: connection strings, app settings, runtime version. Also configures local runtime behavior. No — local dev only; typically in .gitignore; secrets stay local
function.json Binding configuration per function (in-process model only); defines trigger, input, output binding types and their properties Yes (in-process); auto-generated in isolated worker model
Connection Property = App Setting Name, Not the Value
In function.json (and in binding attributes), the "connection" property must contain the name of an app setting that holds the connection string — not the connection string itself. For example: "connection": "MyStorageAccount" tells the runtime to look up the app setting named MyStorageAccount and use its value as the connection string. Writing the actual connection string in the "connection" field will cause the binding to fail.

Retry Policies

Policy Behavior Configuration
Fixed delay Waits a fixed interval between each retry attempt "retryStrategy": "fixedDelay", "delayInterval": "00:00:05"
Exponential backoff Doubles the wait interval after each retry, up to a maximum; reduces thundering herd on dependent services "retryStrategy": "exponentialBackoff", "minimumInterval", "maximumInterval"

Maximum retry count for both strategies is 10 retries. Set "maxRetryCount": -1 for unlimited retries (use with caution).

② Exam Gotchas
The "connection" Property Holds an App Setting Name — Not the Connection String
This is one of the most common exam traps for binding configuration. A question shows function.json with something like: "connection": "DefaultEndpointsProtocol=https;AccountName=..." — a real connection string inline in the binding configuration. This is incorrect. The runtime looks up an app setting by the value in "connection". The app setting itself holds the connection string. If you put the literal connection string in "connection", Azure cannot resolve it and the binding fails. The correct pattern: store MyStorageConnection=DefaultEndpointsProtocol=https;...; in App Settings, then use "connection": "MyStorageConnection" in the binding.
local.settings.json Is Never Deployed to Azure
Developers often assume that local.settings.json is deployed with func azure functionapp publish. It is not. This file is intentionally excluded from deployment — it is for local development only and is listed in .gitignore by default. Any connection strings or app settings in local.settings.json must be separately configured as Application Settings in the Azure portal, ARM template, or via Azure CLI. Forgetting this causes "works locally, fails in Azure" bugs — a scenario the exam specifically tests.
HTTP Trigger Timeout Is Capped at 230 Seconds by Azure Load Balancer
Even on Premium or Dedicated plans where Functions support unlimited timeout, HTTP-triggered functions face a hard 230-second (~3.8 minute) timeout enforced by the Azure load balancer upstream. The load balancer will close the connection after 230 seconds regardless of the Functions runtime timeout setting. For long-running HTTP-initiated work, use the Async HTTP API (Durable Functions) pattern: return a 202 Accepted with a polling URL immediately, process the work asynchronously, and let the client poll for completion.

⑤ Wrong-Answer Traps
Trap: "Put the connection string directly in the 'connection' field of function.json to simplify configuration"
Wrong. The "connection" property in function.json or binding attributes is resolved as an application setting name. The runtime calls Environment.GetEnvironmentVariable("YourConnectionName") and uses the result as the actual connection string. Placing a raw connection string directly in the "connection" field means the runtime tries to look up an environment variable with that entire connection string as its name — which does not exist — and the binding fails at startup.
Trap: "Configure the function's local connection strings in local.settings.json and they will be available after deployment"
Wrong. local.settings.json is a local development tool only. The Azure Functions Core Tools explicitly excludes it from the func azure functionapp publish deployment package. You must separately configure all required Application Settings in Azure — through the Azure portal, Azure CLI (az functionapp config appsettings set), ARM template, or Bicep. A function that works locally but fails in Azure almost always has this root cause.
Trap: "For a function that takes 10 minutes via HTTP trigger, set the host.json timeout to 00:10:00 on the Premium plan"
Wrong — or incomplete at best. Even with a generous functionTimeout in host.json, the HTTP trigger is hard-capped at 230 seconds by the Azure infrastructure load balancer. No host.json setting overrides this. The correct architectural response is to use the Async HTTP API (Durable Functions) pattern: the HTTP trigger immediately returns 202 Accepted with a polling URL, the long-running logic executes in an orchestrator/activity, and the client polls for completion. The exam will offer increasing the host.json timeout as a plausible distractor.
⑥ Practice Q&A
1
A developer writes the following in function.json: "connection": "DefaultEndpointsProtocol=https;AccountName=myaccount;AccountKey=abc123==". The function fails to start with a binding error. What is wrong, and how should it be fixed?
▼
✓ Correct Answer: The "connection" property must contain an app setting NAME, not the connection string value itself
Explanation: The runtime interprets "connection" as the name of an environment variable (App Setting). It calls Environment.GetEnvironmentVariable("DefaultEndpointsProtocol=https;...") — which returns null because no such env var exists.
  • Fix step 1: Add an App Setting named MyStorageConn with the value DefaultEndpointsProtocol=https;AccountName=myaccount;AccountKey=abc123==
  • Fix step 2: Change function.json to "connection": "MyStorageConn"
  • In local dev: add "MyStorageConn": "DefaultEndpointsProtocol=..." to the Values section of local.settings.json
  • This pattern applies to ALL bindings: Storage, Service Bus, Cosmos DB, Event Hubs — the connection property is always an app setting name
2
A function runs perfectly locally but fails immediately after deployment to Azure with a "connection string not found" error. The developer confirms the code is identical. What is the most likely cause?
▼
✓ Correct Answer: The connection strings defined in local.settings.json were not deployed; they must be added as Application Settings in Azure
Explanation: local.settings.json is deliberately excluded from deployment. This is by design — it often contains secrets that should not be in source control or deployment packages.
  • The developer must recreate each setting from local.settings.json as an Application Setting in the Azure Function App (Configuration blade in portal)
  • Via CLI: az functionapp config appsettings set --name <appname> --resource-group <rg> --settings MyConn="value"
  • For sensitive values, store in Azure Key Vault and reference via Key Vault reference syntax: @Microsoft.KeyVault(SecretUri=...)
  • The func azure functionapp publish command publishes code, not settings — settings must be managed separately
3
You are building a new Azure Functions project in .NET 8. Should you use the in-process model or the isolated worker model, and why?
▼
✓ Correct Answer: Isolated worker model — it supports .NET 8, is decoupled from the host runtime, and is Microsoft's recommended approach for all new development
Explanation: The in-process model is limited to .NET versions that the Functions host itself targets — as of Functions v4, that is .NET 6. It cannot run .NET 8 in-process.
  • Isolated worker model runs .NET 8 (or later) in a separate process, communicating with the host via gRPC — no .NET version coupling
  • Isolated worker also supports .NET Framework 4.8 (for legacy migration scenarios)
  • Microsoft has announced the in-process model will be retired — no new features are being added
  • Trade-off: isolated worker has slightly higher cold start overhead and per-call latency due to gRPC marshaling — negligible for most workloads
  • Startup: use HostBuilder with ConfigureFunctionsWorkerDefaults() — same DI model as ASP.NET Core

An output binding writes a blob. The developer wants each output blob to have a unique name derived from a GUID generated at runtime. How should the blob path be configured in function.json?
▼
✓ Correct Answer: Use the {rand-guid} binding expression: "path": "output/{rand-guid}.json"
Explanation: Binding expressions allow dynamic values in binding configuration without code changes.
  • {rand-guid} is a built-in binding expression that generates a new GUID for each function invocation
  • Configured in function.json: "path": "output/{rand-guid}.json"
  • In attribute-based (isolated): [BlobOutput("output/{rand-guid}.json", Connection = "...")]
  • Other useful expressions: {DateTime} for timestamp-based names, {queueTrigger} to use queue message content as part of the path
  • Do not generate the GUID in code and try to pass it to the binding — binding paths are evaluated by the runtime before the function runs; code-generated values belong in the output object's path property if using SDK-style bindings

A developer wants the Azure Functions runtime to automatically retry a queue-triggered function on transient errors, waiting progressively longer between attempts. Which configuration in host.json achieves this?
▼
✓ Correct Answer: Set retry strategy to "exponentialBackoff" in host.json with minimumInterval and maximumInterval
Explanation: Azure Functions supports host-level retry policies in host.json:
  • Fixed delay: waits the same interval every time — good for predictable transient faults
  • Exponential backoff: doubles the interval after each retry up to maximumInterval — good for reducing load on a recovering downstream service (avoids thundering herd)
  • Config: "strategy": "exponentialBackoff", "maxRetryCount": 5, "minimumInterval": "00:00:02", "maximumInterval": "00:02:00"
  • Maximum supported retries: 10 (set "maxRetryCount": -1 for unlimited)
  • Note: queue trigger has its own maxDequeueCount (default 5) — after that many dequeues without success, the message is moved to the poison queue regardless of the host retry policy

A user submits a request via HTTP to an Azure Function that triggers a video transcoding job expected to take 20 minutes. The function is on the Premium plan. After 4 minutes, the user receives a timeout error. What is happening and what is the correct architectural fix?
▼
✓ Correct Answer: The 230-second Azure load balancer timeout is closing the HTTP connection; use the Durable Functions Async HTTP API pattern
Explanation: The Azure infrastructure load balancer enforces a 230-second (~3.8 minute) idle/request timeout on HTTP connections, independent of the Functions runtime timeout setting. Moving to Premium or increasing functionTimeout in host.json does not change this.
  • Fix: Implement the Async HTTP API pattern using Durable Functions
  • Step 1: HTTP trigger receives request, starts a Durable orchestration, immediately returns 202 Accepted with a statusQueryGetUri polling URL (built-in, returned by client.CreateCheckStatusResponseAsync())
  • Step 2: Orchestration calls activity functions to perform the transcoding asynchronously
  • Step 3: Client polls the status URL every N seconds until it receives 200 OK with the result
  • This is the canonical exam answer for any HTTP-initiated long-running operation — always Durable Async HTTP API

What is the purpose of the "host.json" file vs. "local.settings.json" in an Azure Functions project, and which one is deployed to Azure?
▼
✓ Correct Answer: host.json controls runtime-wide behavior and IS deployed; local.settings.json holds local dev config and is NOT deployed
Explanation:
  • host.json: controls the Functions host runtime globally — timeout, logging, concurrency, retry policies, extension settings. Checked into source control and deployed with the app. Same file used locally and in Azure.
  • local.settings.json: provides local equivalents of Azure App Settings and connection strings. Only read by the local Functions host (func start). Not deployed. Listed in .gitignore by default. The AzureWebJobsStorage entry here points to a local emulator (e.g., Azurite) or a dev storage account — it must be separately configured as an App Setting in Azure.
  • A common exam distractor: "To change the logging level in production, update local.settings.json" — wrong. Update host.json (which is deployed) or use Application Insights configuration in the portal.

A function uses a Queue trigger input binding and a Blob Storage output binding. The developer wants the output blob's name to match the content of the queue message. How should the blob path be configured?
▼
✓ Correct Answer: Use the {queueTrigger} binding expression in the blob output path
Explanation: The {queueTrigger} binding expression resolves to the string content of the incoming queue message at runtime.
  • Example function.json blob output path: "path": "results/{queueTrigger}.json" — if the queue message is order-12345, the blob will be named results/order-12345.json
  • This avoids any custom code to construct the blob path — the binding engine handles the resolution
  • In attribute syntax: [BlobOutput("results/{queueTrigger}.json", Connection = "...")]
  • Binding expressions can be chained: "results/{queueTrigger}/{DateTime}.json" combines queue content and timestamp
  • Important: the queue message must be a plain string for {queueTrigger} to work as the full value; for JSON messages, use trigger metadata properties like {queueTrigger.orderId}
