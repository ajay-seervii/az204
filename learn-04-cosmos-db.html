Topic 04

01 Explore Azure Cosmos DB

① Key Concepts

API Options

Cosmos DB is a globally distributed, multi-model NoSQL database. The API you choose at container creation is permanent — it cannot be changed later. Each API exposes a different wire protocol and data model.

API Data Model Query Language Best For
Core (SQL / NoSQL) JSON documents SQL-like syntax New apps, native Cosmos, richest SDK support
MongoDB BSON documents MongoDB wire protocol Migrating existing MongoDB apps with minimal code change
Cassandra Wide-column (rows + columns) CQL (Cassandra Query Language) Migrating Cassandra workloads; time-series, IoT
Gremlin Graph (vertices + edges) Gremlin traversal language Social graphs, recommendation engines, fraud detection
Table Key-value pairs OData / REST Migrating Azure Table Storage apps; globally distributed key-value
💡
Core (SQL) API = exam default
When the exam says "Cosmos DB" without specifying an API, assume Core (SQL) API. It supports the richest set of SDK operations, server-side JS (stored procs/triggers/UDFs), change feed, and has the most exam coverage. All code examples in exam questions use the Core API SDK unless stated otherwise.

Consistency Levels

Cosmos DB offers five consistency levels. You set the default at the account level and can optionally weaken it per request (but never strengthen beyond the account default). Stronger consistency = higher RU cost and higher latency.

Level (weakest → strongest) Guarantee Reads Reflect Use Case
Eventual Lowest latency, no ordering guarantee Any replica at any point in time Social likes/counts where stale data is acceptable
Consistent Prefix Reads never see out-of-order writes Writes in order, but possibly lagged Photo captions, comment ordering
Session Read-your-own-writes within a session token All writes from the current session Shopping cart, user profile — DEFAULT and most common
Bounded Staleness Lag bounded by K versions or T seconds Writes within the configured staleness window Global apps needing near-real-time sync with low latency
Strong Linearizable reads — always reads latest committed write Most recent write globally Financial transactions, inventory systems requiring absolute accuracy
💡
Strong consistency restriction
Strong consistency is only available in a single-region account OR a multi-region account where all regions are configured for write operations with the same region set. With multi-region writes (multi-master) enabled, Strong consistency is not supported — you must use Bounded Staleness or weaker. The exam will present scenarios to test whether you know this restriction.

Partitioning Architecture

Understanding the two-layer partition model is essential for both correctness and performance questions on the exam.

Concept Description Key Fact
Logical Partition All items sharing the same partition key value Max size: 20 GB. Cannot grow beyond this.
Physical Partition Internal server-side shard managed by Cosmos DB Can host multiple logical partitions. Scales automatically.
Partition Key Property used to route items and queries Immutable after container creation. Choose carefully.
Hot Partition One partition receiving disproportionate traffic Results in HTTP 429 (TooManyRequests) on that partition while others are idle
💡
Good partition key characteristics
A good partition key has high cardinality (many distinct values), produces even distribution of both storage and throughput, and is commonly used in queries so most queries are single-partition (cheaper). Examples: userId, deviceId, orderId. Bad examples: status (only 3-4 values), country (uneven distribution if most users are in one country), createdDate truncated to day (time-series hotspot).
② Exam Gotchas
⚠️
Gotcha: Partition key cannot be changed after container creation
This is the single most impactful and irreversible architectural decision when creating a Cosmos DB container. Once a container is created with a given partition key, it cannot be modified. If you chose poorly, you must create a new container and migrate data. The exam will present scenarios where a team is experiencing hot partition issues and ask what they should have done differently — the answer always involves selecting a better partition key, not adding more RU/s.
⚠️
Gotcha: "Global distribution is automatic" — this is FALSE
Adding regions to a Cosmos DB account is a manual action in the Azure Portal, CLI, or ARM template. The data is automatically replicated after you add a region, but the region must be added explicitly. Automatic failover (promoting a read region to write region on failure) is also a setting you must enable — it is not on by default. Do not confuse "data is automatically replicated once a region is added" with "regions are automatically provisioned."
⚠️
Gotcha: Session consistency only guarantees reads within the same session token
Session consistency provides read-your-own-writes only within the same session, scoped by the session token. If a different client (or a new client instance without the session token) reads data, it may not see the latest write. Exam scenarios will describe a user who just submitted a form and sees stale data on page reload — this is only a problem if the session token was not passed correctly to the new request, or a different consistency level is in use. Verify that the scenario is using Session consistency and the token is being threaded through.
③ Key Numbers & Limits
Metric Value Exam Note
Max item size 2 MB Items larger than 2 MB cannot be stored; design data model accordingly
Max logical partition size 20 GB Exceeding this causes a write error; requires better partition key choice
Point read cost (1 KB item) 1 RU Cheapest read type; use ReadItemAsync whenever you have id + partition key
Max stored procedure execution time 5 seconds Transactions must complete within this window or they time out and roll back
Default TTL Disabled Must enable TTL at container level first (DefaultTimeToLive), then set on items
Bounded Staleness (K) Configurable version lag Min K=1 version. For multi-region, min is 100,000 versions or 300 seconds
Bounded Staleness (T) Configurable time lag Min 5 seconds for single region; 300 seconds minimum for multi-region
Max regions per account Depends on region availability No hard cap; all Azure regions supported. Cost scales linearly with regions added
④ CLI / SDK Patterns
# Create a Cosmos DB account with Session consistency (default) in two regions
az cosmosdb create \
  --name mycosmosaccount \
  --resource-group myRG \
  --default-consistency-level Session \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=false \
  --locations regionName=westus failoverPriority=1 isZoneRedundant=false

# Enable automatic failover (not on by default!)
az cosmosdb update \
  --name mycosmosaccount \
  --resource-group myRG \
  --enable-automatic-failover true

# Create a Core (SQL) database
az cosmosdb sql database create \
  --account-name mycosmosaccount \
  --resource-group myRG \
  --name myDatabase

# Create a container with partition key /userId and provisioned 400 RU/s
az cosmosdb sql container create \
  --account-name mycosmosaccount \
  --resource-group myRG \
  --database-name myDatabase \
  --name myContainer \
  --partition-key-path /userId \        # immutable — choose carefully
  --throughput 400                      # min 400 RU/s for dedicated throughput

# Enable TTL at container level (required before items can use TTL)
az cosmosdb sql container update \
  --account-name mycosmosaccount \
  --resource-group myRG \
  --database-name myDatabase \
  --name myContainer \
  --ttl -1                              # -1 = enabled but no default expiry; items set their own TTL
                                        # positive integer = default TTL in seconds for all items
💡
TTL value meanings
DefaultTimeToLive = null means TTL is disabled (items never expire). DefaultTimeToLive = -1 means TTL is enabled but no default — individual items must set their own _ts-relative TTL. DefaultTimeToLive = N (positive integer) means items expire after N seconds by default unless overridden at item level.
⑤ Wrong-Answer Traps
🚨
Trap: "You can change the partition key by updating the container"
This is completely false. The partition key is set at container creation and is immutable. The exam may offer "Update the container's partition key to /orderId" as a distractor. The only correct remediation for a bad partition key is to create a new container with the correct partition key and migrate the data using the Cosmos DB data migration tool, Azure Data Factory, or a custom script.
🚨
Trap: "Multi-region writes support Strong consistency"
Strong consistency requires a single write region. When multi-region writes (multi-master) is enabled, the strongest supported consistency level is Bounded Staleness. The exam will describe a scenario requiring both multi-master and Strong consistency — this combination is not possible. You must choose one or the other.
🚨
Trap: "Adding more RU/s fixes a hot partition problem"
Throwing more throughput at a hot partition is a temporary workaround at best and does not fix the underlying problem. The correct exam answer for a hot partition scenario is always to re-design the partition key to achieve better distribution. More RUs increase cost proportionally but the hot partition will still hit its per-partition limit once a physical partition fills up. Also note: RU/s scale across physical partitions equally — you cannot allocate more RUs to a specific partition.
⑥ Practice Q&A
1
A development team is building a new application on Cosmos DB. They want to read their own writes immediately after committing them, but do not require all global replicas to be up-to-date before a read succeeds. Which consistency level should they configure as the account default?
▼
✓ Correct Answer: Session
Session consistency is the default and most commonly used level. It guarantees that within a single client session (identified by a session token), reads always reflect the writes made in that session — this is read-your-own-writes. Replicas in other regions may still be lagging.
  • Eventual — no ordering or recency guarantee; reads may see older data even within the same session
  • Consistent Prefix — guarantees ordering but not currency; would not guarantee reading your own writes
  • Bounded Staleness — ensures global consistency within a lag window; stronger than needed and more expensive
  • Strong — reads always return the latest committed write globally; overkill and not compatible with multi-region writes
2
An e-commerce platform stores orders in Cosmos DB using status as the partition key. After launch, the team observes HTTP 429 errors concentrated on the "pending" partition while all other partitions are idle. What is the root cause and the correct fix?
▼
✓ Correct Answer: Hot partition due to low-cardinality partition key — redesign the partition key
Root cause: status has very low cardinality (only a few values: pending, shipped, delivered, cancelled). The vast majority of new orders land in the "pending" partition, creating a hot partition that exhausts its share of the provisioned RUs.
  • HTTP 429 (TooManyRequests) is the Cosmos DB response when a partition exhausts its RU allocation
  • The fix is to redesign the container with a high-cardinality key such as orderId or a synthetic key combining customerId + month
  • Increasing RU/s is a cost-inefficient workaround; the hot partition will still saturate relative to others
  • The partition key cannot be changed in place — a new container must be created and data migrated
3
A company runs a Cosmos DB account with multi-region writes enabled across three regions. A junior developer attempts to set the account's default consistency level to Strong. What will happen?
▼
✓ Correct Answer: The operation will fail — Strong consistency is not compatible with multi-region writes
Strong consistency requires a single write region. When multi-region writes (multi-master) is enabled, the Azure portal and SDK both prevent setting Strong consistency and will return a validation error.
  • With multi-region writes, the strongest available consistency is Bounded Staleness
  • This is a hard platform constraint, not a configuration recommendation
  • To use Strong consistency, the team must disable multi-region writes and designate a single write region
  • The exam will offer this exact scenario — do not select "Strong" when multi-master is enabled
4
A Cosmos DB container was created using the MongoDB API. Six months later, the team wants to switch to the Core (SQL) API to use stored procedures and server-side JavaScript. What is the correct approach?
▼
✓ Correct Answer: Create a new Core (SQL) API container and migrate the data
The API is selected at container (and account) creation and is permanent. There is no in-place migration path between Cosmos DB APIs.
  • The team must provision a new Cosmos DB account with the Core (SQL) API
  • Data must be migrated using Azure Data Factory, the Cosmos DB Spark connector, or a custom migration script
  • This is an expensive and disruptive process — reinforcing why API selection is a critical upfront architectural decision
  • The exam will not offer "change the API setting on the existing account" as a valid answer — it does not exist
5
A container has TTL set to -1 at the container level. An item is written with "ttl": 300 in its JSON body. What happens to that item?
▼
✓ Correct Answer: The item expires and is deleted 300 seconds after its last modification timestamp (_ts)
Container TTL = -1 means TTL is enabled but there is no container-level default. Items that do not specify a ttl property will never expire. Items that set "ttl": N will expire N seconds after their _ts (last modified timestamp).
  • TTL expiry is background garbage collection — items may persist slightly beyond the exact TTL window
  • If container TTL is null, item-level TTL is ignored entirely — this is a common exam trap
  • Setting "ttl": -1 on an item when the container has a positive default TTL overrides it, making that specific item never expire
  • TTL does not consume RUs when items are deleted by the system
6
A logical partition in a Cosmos DB container has grown to 19.8 GB. A developer is concerned about future writes. What is the correct statement about this situation?
▼
✓ Correct Answer: The partition is approaching the 20 GB limit; writes to this partition will start failing once the limit is exceeded
The maximum size of a single logical partition is 20 GB. This is a hard limit that cannot be increased by adding more RUs or storage.
  • Once the limit is exceeded, write operations targeting that partition key value will return an error
  • Cosmos DB does not automatically split a logical partition — only physical partitions split automatically
  • The correct remediation is to change the partition key strategy to distribute data more finely (e.g., append a suffix or use a composite key)
  • This scenario is common in time-series data where all data for a given sensor or tenant accumulates under one partition key value
7
A team is comparing Azure Table Storage and the Cosmos DB Table API. Which of the following is a correct reason to migrate from Azure Table Storage to Cosmos DB Table API?
▼
✓ Correct Answer: Cosmos DB Table API provides global distribution, lower latency SLAs, and automatic indexing while remaining wire-protocol compatible with Azure Table Storage SDKs
The Cosmos DB Table API is a superset of Azure Table Storage — it uses the same PartitionKey/RowKey model and is compatible with existing Azure Table Storage SDKs with only a connection string change.
  • Benefits: single-digit millisecond latency at the 99th percentile, global distribution, automatic secondary indexing on all properties
  • Azure Table Storage does not provide global distribution or guaranteed low-latency SLAs at the same level
  • Migration is low-friction — existing code using Microsoft.Azure.Cosmos.Table SDK works with both; switch by changing the connection string
  • Cosmos DB Table API is not cheaper for low-throughput workloads — Azure Table Storage may be more cost-effective for infrequently accessed data
8
Bounded Staleness is configured with K=100,000 versions. In a multi-region Cosmos DB account, what is the minimum allowed value for the time-based staleness (T) parameter?
▼
✓ Correct Answer: 300 seconds (5 minutes)
For multi-region accounts, Bounded Staleness has minimum thresholds: K must be at least 100,000 versions, and T must be at least 300 seconds.
  • For single-region accounts, the minimums are much lower: K=1 version, T=5 seconds
  • These minimums exist because replication across regions introduces irreducible latency
  • Attempting to set T below 300 seconds on a multi-region account will be rejected by the platform
  • Bounded Staleness is the strongest consistency available when multi-region writes are enabled

02 Work with Azure Cosmos DB

① Key Concepts

SDK v3 Object Hierarchy

The Cosmos DB .NET SDK v3 follows a strict hierarchy. CosmosClient is thread-safe and should be a singleton — creating one per request is a common and costly mistake.

Object Created Via Represents Key Methods
CosmosClient new CosmosClient(endpoint, key) Connection to the account; singleton GetDatabase(), CreateDatabaseIfNotExistsAsync()
Database client.GetDatabase("dbName") Cosmos DB database GetContainer(), CreateContainerIfNotExistsAsync()
Container db.GetContainer("cName") Collection / container of items CreateItemAsync(), ReadItemAsync(), GetItemQueryIterator()
Item operations Methods on Container CRUD on individual documents ReadItemAsync, UpsertItemAsync, PatchItemAsync, DeleteItemAsync

Point Read vs Query — Cost Implications

Operation Method Requires RU Cost
Point Read ReadItemAsync(id, partitionKey) Both item id AND partition key value 1 RU per 1 KB — cheapest possible read
Single-partition query GetItemQueryIterator with partition key filter Query that includes partition key in WHERE clause Higher than point read; depends on result set size and index usage
Cross-partition query GetItemQueryIterator without partition key filter Fan-out across all physical partitions Significantly higher; scales with number of partitions and data scanned
💡
Always use ReadItemAsync when you have both id and partition key
If your code has both the item's id and the partition key value, always use ReadItemAsync instead of a query. It is the cheapest operation in Cosmos DB — a 1 KB item costs exactly 1 RU. A query that returns the same single item will cost more because it goes through the query engine.

Throughput Modes

Mode How It Works Min RU/s Multi-Region Best For
Manual Provisioned Fixed RU/s you set; you pay regardless of utilization 400 RU/s per container (or 100 RU/s shared database) Yes Predictable, steady workloads
Autoscale Scales automatically between 10% and 100% of max RU/s you set 10% of configured max (e.g., max=1000 → min=100) Yes Variable or unpredictable workloads; dev/test environments
Serverless Pay-per-request; no reserved capacity No minimum No — single region only Sporadic, low-volume workloads; new apps still being developed

Change Feed

The change feed captures all inserts and updates to a container, ordered by modification time within each partition. It does not capture deletes by default.

Model Mechanism Use Case
Push model (Processor) ChangeFeedProcessor — lease container tracks position; workers pull and process Event-driven processing, Azure Functions trigger, real-time pipelines
Pull model (Iterator) GetChangeFeedIterator — client controls when to poll and saves its own checkpoint Batch jobs, custom retry logic, scenarios needing manual checkpoint control
💡
Tracking deletes in change feed: soft-delete pattern
Since change feed does not capture hard deletes, the standard pattern is: add an isDeleted: true flag to documents instead of deleting them, and set a short TTL on those documents. The change feed will emit the update (setting isDeleted = true

Server-Side Programming (Stored Procedures, Triggers, UDFs)

Feature Language Scope ACID Key Limitation
Stored Procedures JavaScript Single logical partition only Yes — within the partition 5-second execution limit; no cross-partition transactions
Pre-triggers JavaScript Item operation (before write) Participates in item write transaction Must be explicitly invoked per request; not automatic
Post-triggers JavaScript Item operation (after write) Participates in item write transaction Must be explicitly invoked per request; not automatic
UDFs JavaScript Query expressions N/A (read-only) Can only be used in SELECT; cannot modify data

Partial Document Updates: PatchItemAsync

PatchItemAsync implements JSON Patch (RFC 6902) semantics. It updates specific fields without replacing the entire document, reducing payload size and RU cost compared to a full ReplaceItemAsync.

② Exam Gotchas
⚠️
Gotcha: Change feed does NOT capture deletes
This is one of the most frequently tested Cosmos DB facts. The change feed only records inserts and updates. A hard delete is invisible to change feed consumers. If a scenario asks how to detect when a document is deleted and downstream systems need to react, the answer is always the soft-delete pattern: mark the item with isDeleted: true (which generates a change feed event), then let TTL clean up the item. Any answer that claims "change feed captures deletes" is wrong.
⚠️
Gotcha: Stored procedures are NOT ACID across containers or partitions
Stored procedures provide ACID transactions, but only within a single logical partition of a single container. All operations in a stored procedure must target the same partition key value. If you call a stored procedure with items from partition key "A" and try to modify an item with partition key "B", the operation will fail. Cross-container transactions are also not supported. This is a hard architectural constraint, not a configuration option.
⚠️
Gotcha: CosmosClient must be a singleton — recreating it per request causes connection pool exhaustion
CosmosClient is designed to be instantiated once and reused for the lifetime of the application. Each instance maintains its own internal HTTP connection pool and caches endpoint routing information. Creating a new CosmosClient per HTTP request is a critical performance anti-pattern that causes port exhaustion, excessive TCP connection overhead, and dramatically inflated latency. In Azure Functions, register it as a singleton in DI or use a static field. The exam will present code samples where a client is created inside a method body and ask you to identify the problem.

Metric Value Exam Note
Serverless max storage 1 TB per container Serverless is also single-region only — both constraints together are exam-tested
Provisioned min RU/s per container 400 RU/s Shared database throughput allows containers to share from a 100 RU/s minimum
Autoscale min RU/s 10% of configured max If max is set to 4,000 RU/s, the min is 400 RU/s — you always pay at least 10%

④ CLI / SDK Patterns
// CosmosClient as a singleton — register in DI (ASP.NET Core / Azure Functions)
// Program.cs / Startup.cs
builder.Services.AddSingleton(sp =>
    new CosmosClient(
        builder.Configuration["CosmosDb:Endpoint"],
        builder.Configuration["CosmosDb:Key"],
        new CosmosClientOptions
        {
            ApplicationRegion = Regions.EastUS,   // preferred read region
            SerializerOptions = new CosmosSerializationOptions
            {
                PropertyNamingPolicy = CosmosPropertyNamingPolicy.CamelCase
            }
        }
    )
);

// --- CRUD operations ---
Container container = client.GetDatabase("myDb").GetContainer("myContainer");

// Point read — cheapest (1 RU/KB), requires BOTH id and partition key value
ItemResponse<MyItem> response = await container.ReadItemAsync<MyItem>(
    id: "item-123",
    partitionKey: new PartitionKey("user-456")  // must be the exact partition key value
);
Console.WriteLine($"RU cost: {response.RequestCharge}");

// Create item — partition key value must match the field in the item
await container.CreateItemAsync(
    item: new MyItem { Id = "item-124", UserId = "user-456", Name = "Widget" },
    partitionKey: new PartitionKey("user-456")
);

// Upsert (insert or replace)
await container.UpsertItemAsync(myItem, new PartitionKey("user-456"));

// Partial update with PatchItemAsync (JSON Patch — no full document replacement needed)
List<PatchOperation> ops = new()
{
    PatchOperation.Replace("/status", "shipped"),       // update existing field
    PatchOperation.Add("/shippedDate", DateTime.UtcNow), // add new field
    PatchOperation.Increment("/retryCount", 1)           // atomic increment
};
await container.PatchItemAsync<MyItem>("item-123", new PartitionKey("user-456"), ops);

// Query — single partition (cheaper; include partition key in filter)
QueryDefinition query = new QueryDefinition(
    "SELECT * FROM c WHERE c.userId = @uid AND c.status = @s"
).WithParameter("@uid", "user-456")
 .WithParameter("@s", "pending");

FeedIterator<MyItem> iter = container.GetItemQueryIterator<MyItem>(
    query,
    requestOptions: new QueryRequestOptions
    {
        PartitionKey = new PartitionKey("user-456"),  // enables single-partition query
        MaxItemCount = 50
    }
);
while (iter.HasMoreResults)
{
    FeedResponse<MyItem> page = await iter.ReadNextAsync();
    foreach (var item in page) { /* process */ }
}

// Change feed processor — push model (recommended for event-driven scenarios)
// Requires a separate "lease" container to track processing state

var processor = container
    .GetChangeFeedProcessorBuilder<MyItem>(
        processorName: "myProcessor",
        onChangesDelegate: async (changes, cancellationToken) =>
        {
            foreach (var item in changes)
            {
                // Note: deletes are NOT visible here
                // Only inserts and updates appear
                Console.WriteLine($"Changed item: {item.Id}");
            }
        })
    .WithInstanceName(Environment.MachineName)
    .WithLeaseContainer(leaseContainer)  // dedicated lease container required
    .WithStartTime(DateTime.MinValue.ToUniversalTime()) // process from beginning
    .Build();

await processor.StartAsync();

// Soft-delete pattern to track deletes via change feed
// Step 1: instead of deleting, mark isDeleted and set TTL
var patch = new[]
{
    PatchOperation.Set("/isDeleted", true),
    PatchOperation.Set("/ttl", 3600) // expire in 1 hour after soft-delete
};
await container.PatchItemAsync<MyItem>(id, partitionKey, patch);
// Step 2: change feed consumer sees isDeleted=true and processes the logical delete
// Step 3: after TTL expires, Cosmos DB physically removes the item (silent, no change feed event)

// Stored procedure — ACID within a single partition only
// Registered in Cosmos DB, called from client code

// Client-side invocation (JavaScript stored proc targeting partition key "user-456")
var sprocResponse = await container.Scripts.ExecuteStoredProcedureAsync<string>(
    storedProcedureId: "bulkCreate",
    partitionKey: new PartitionKey("user-456"),  // ALL operations in the sproc must use this partition
    parameters: new dynamic[] { itemsArray }
);

⑤ Wrong-Answer Traps
🚨
Trap: "Use GetItemQueryIterator when you have the id and partition key to get a single item"
Any time you have both the item's id and the partition key value, you must use ReadItemAsync, not a query. A query that filters by WHERE c.id = 'x' still goes through the query engine and costs more RUs than a point read. The exam will show code that uses a query for a single known item and ask you to identify the inefficiency — the answer is always to replace it with ReadItemAsync.
🚨
Trap: "ReplaceItemAsync and PatchItemAsync are equivalent for updating a single field"
They are not equivalent in cost or behavior. ReplaceItemAsync requires sending the entire document payload, replacing everything (including fields you did not intend to change — a race condition risk if another write happened between read and replace). PatchItemAsync sends only the operations to apply, is more bandwidth-efficient, costs fewer RUs for large documents, and is atomic at the server without a read-modify-write cycle. For updating one or two fields on a large document, PatchItemAsync is the correct choice.
🚨
Trap: "Serverless Cosmos DB can be deployed to multiple regions for high availability"
Serverless mode is single-region only. It does not support global distribution or multi-region replication. If a scenario requires global distribution, high availability across regions, or multi-region writes, Serverless is immediately disqualified regardless of cost or throughput considerations. The correct mode is either Manual Provisioned or Autoscale, both of which support multi-region configurations.
⑥ Practice Q&A
1
A developer writes the following code in an Azure Function that processes HTTP requests:

public async Task<IActionResult> Run(HttpRequest req) { var client = new CosmosClient(endpoint, key); var item = await client.GetDatabase("db").GetContainer("c").ReadItemAsync<MyItem>(id, pk); return new OkObjectResult(item.Resource); }

What is the critical problem with this code?
▼
✓ Correct Answer: CosmosClient is instantiated per request instead of being a singleton
CosmosClient is heavyweight — it maintains connection pools, caches endpoint routing information, and establishes TCP connections. Creating a new instance per request causes:
  • TCP port exhaustion under load (each instance opens new sockets)
  • Significant latency from connection establishment on every request
  • Excessive memory pressure from non-disposed connection pools
  • The fix is to register CosmosClient as a singleton in DI or use a static readonly field at the class level
  • In Azure Functions, use builder.Services.AddSingleton<CosmosClient>(...) in Program.cs
2
A Cosmos DB change feed consumer processes events and needs to detect when documents are deleted. The developer adds logic to handle items where the change type is "Delete". After deployment, no delete events are ever received. What is wrong?
▼
✓ Correct Answer: Cosmos DB change feed does not emit events for deletes — only inserts and updates
Hard deletes are invisible to the change feed. The change feed processor will never emit a "delete" event regardless of how the consumer is configured.
  • The canonical solution is the soft-delete pattern: instead of deleting, set a flag "isDeleted": true and optionally set a TTL on the item
  • This generates an update event in the change feed, which the consumer processes as a logical delete
  • After the TTL expires, Cosmos DB physically removes the item — this removal also does not appear in the change feed
  • This limitation applies to all change feed consumption models (both push and pull)
3
An application must update only the status and updatedAt fields of a 500 KB document stored in Cosmos DB. A developer proposes using ReplaceItemAsync. A senior engineer suggests PatchItemAsync instead. What is the strongest argument for using PatchItemAsync?
▼
✓ Correct Answer: PatchItemAsync sends only the patch operations, avoiding a full document read and replace, reducing RU cost and eliminating the read-modify-write race condition
ReplaceItemAsync requires sending the entire 500 KB document, which is both expensive in bandwidth and RU cost. It also requires a prior read to get the current document state before replacing it, introducing a time window where concurrent writes could overwrite each other.
  • PatchItemAsync sends only the JSON Patch operations (e.g., Replace /status, Set /updatedAt), which is a tiny payload
  • No prior read is required — the patch is applied atomically on the server
  • This eliminates the read-modify-write race condition entirely
  • RU cost is dramatically lower for large documents where only a few fields change
  • Supports operations: Add, Remove, Replace, Set, Increment, Move, Copy, Test
4
A startup expects very uneven traffic — quiet during weekdays and heavy bursts on weekends. The application is global and must serve users in North America and Europe. Which Cosmos DB throughput mode is most appropriate?
▼
✓ Correct Answer: Autoscale provisioned throughput
Autoscale automatically scales RU/s between 10% and 100% of the configured maximum, paying only for what is consumed (billed at the peak RU/s within each hour). This handles variable traffic without over-provisioning.
  • Serverless is eliminated because it is single-region only — the requirement for North America and Europe means multi-region is needed
  • Manual provisioned would require setting RUs at the weekend peak and paying that rate 24/7, making weekday periods expensive and wasteful
  • Autoscale works with multi-region deployments, satisfying both the global distribution and cost-efficiency requirements
  • Set the max RU/s to the expected weekend peak; the system will scale down automatically during quiet periods

A stored procedure in Cosmos DB is designed to process a batch of 1,000 documents as a single transaction. During testing, the procedure consistently times out. What is the maximum execution window and what should the developer do?
▼
✓ Correct Answer: Stored procedures have a 5-second maximum execution window; the developer should implement continuation tokens to process in smaller batches
All stored procedures must complete within 5 seconds or the entire transaction is rolled back — no partial commits are preserved.
  • The standard pattern for processing large batches is to implement a continuation token within the stored procedure: process as many items as time allows, return a continuation token, and have the client call the procedure again with the token to resume from where it left off
  • Each invocation is its own ACID transaction; the batch is split into sequential transactional chunks
  • Alternatively, if atomicity across all 1,000 documents is not required, process them individually or in smaller groups from the client side
  • There is no way to extend the 5-second limit — it is a hard platform constraint
6
A Cosmos DB change feed processor is running on three worker nodes. A developer notices that some change events are being processed twice. What component is most likely misconfigured?
▼
✓ Correct Answer: The lease container — each processor instance must use a unique instance name and share the same lease container
The lease container is the coordination mechanism for the change feed processor. It tracks which partitions have been processed and by which worker.
  • If multiple processor instances share the same lease container but use the same instance name, they compete for the same leases and may process events multiple times
  • Each processor instance must be initialized with a unique WithInstanceName() value (e.g., machine name, pod name)
  • All instances of the same logical processor group must point to the same lease container — this is how work is distributed among them
  • If each instance uses a different lease container, they each process all partitions independently, resulting in duplicate processing

An application reads a document, modifies it locally, and then calls ReplaceItemAsync to write it back. Another client modifies the same document between the read and the write. The second write silently overwrites the first client's changes. How should the developer prevent this?
▼
✓ Correct Answer: Use optimistic concurrency control with ETags via the IfMatchEtag request option
Every Cosmos DB item has an _etag property that changes every time the item is modified. By passing the original ETag in the IfMatchEtag request option, the write will only succeed if the item has not been modified since it was read.
  • If the ETag no longer matches (another client modified the item), Cosmos DB returns HTTP 412 Precondition Failed
  • The client should catch this error, re-read the latest version, reapply their changes, and retry the write
  • This is the standard optimistic concurrency pattern — no locks are held, so performance is not degraded for non-conflicting operations
  • PatchItemAsync with the FilterPredicate option can also implement conditional updates without a separate read step

A developer is choosing between Autoscale at max 10,000 RU/s and Manual Provisioned at 5,000 RU/s for a Cosmos DB container. The workload is stable with brief daily peaks that reach 9,000 RU/s for about one hour. Which option minimizes cost while meeting throughput requirements?
▼
✓ Correct Answer: Autoscale at max 10,000 RU/s is likely more cost-effective given the brief peak pattern
With Autoscale, you are billed for the highest RU/s consumed within each hour, scaled to 10% increments. During the 23 off-peak hours, if the workload uses ~1,000 RU/s, Autoscale charges for ~1,000 RU/s (10% of 10,000). Manual at 5,000 RU/s charges the full 5,000 RU/s 24 hours a day.
  • Manual Provisioned at 5,000 cannot handle the 9,000 RU/s peak — this would cause 429 throttling, making it not a valid option at all
  • Manual at 10,000 RU/s would cost significantly more than Autoscale during the 23 off-peak hours
  • Autoscale pricing: you pay for the peak RU/s consumed per hour, meaning off-peak hours are cheap
  • Rule of thumb: Autoscale is cost-effective when utilization is below ~60% of max for most of the day; Manual Provisioned is cheaper for steady high-utilization workloads
