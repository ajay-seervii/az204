M1 Explore API Management

① Key Concepts

APIM Architecture: Three Core Components

Azure API Management sits between API consumers and your backend services. Every request passes through the Gateway, which applies policies, enforces authentication, throttles traffic, and forwards calls to the backend. The Management Plane is the administrative API used by the Azure Portal, CLI, ARM templates, and Bicep — it lets you define APIs, configure products, write policies, and manage users. The Developer Portal is an auto-generated, customizable website where API consumers browse documentation, test endpoints interactively, and self-subscribe to products.

Gateway is the only component that handles live traffic
Changes made in the Management Plane (new policies, new API versions) do not take effect until the Gateway picks them up. In the Consumption tier this is near-instant; in dedicated tiers there can be a short propagation delay. The Developer Portal is for documentation and subscription management only — it never proxies API calls.

Products and Subscriptions

A Product is a logical bundle of one or more APIs. To call an API via APIM, a consumer must subscribe to a product that contains that API (unless the product is set to "Open" with no subscription required). Each Subscription generates two 512-bit keys (primary + secondary) that the caller passes in the Ocp-Apim-Subscription-Key HTTP header (or as a subscription-key query parameter). Subscriptions can be scoped to a product, a single API, or all APIs (global scope). Rate-limit and quota policies operate against the subscription key by default.

Policy Scopes and Execution Order

Policies are XML documents applied at four scopes, evaluated from broadest to most specific. The <base/> element explicitly imports the parent scope's policy block at that insertion point. If <base/> is omitted, the parent policy does not execute for that scope.

Scope LevelWhere ConfiguredApplies To
GlobalAll APIs section in APIM instanceEvery request through the gateway
ProductPer-product policy editorAll APIs in that product
APIPer-API policy editorAll operations in that API
OperationPer-operation policy editorOne specific HTTP operation

Policy Sections

SectionWhen It RunsTypical Uses
<inbound>After request received, before forwarding to backendAuth (validate-jwt), rate limiting, header injection, URL rewriting, caching lookup
<backend>Just before the backend call is madeSet backend URL dynamically, load balancing between backends
<outbound>After backend response received, before returning to callerTransform response, filter fields, add response headers, cache-store
<on-error>When an exception occurs in any sectionCustom error responses, logging, retry logic

Named Values

Named Values are a global key-value store in APIM used to avoid hard-coding secrets or configuration strings directly in policy XML. A Named Value is referenced in policy using double-curly-brace syntax: {{my-named-value}}. Values can be plain text, secrets (stored encrypted), or references to Azure Key Vault secrets. Using Key Vault-backed Named Values is the exam-preferred approach for storing API keys, connection strings, and signing certificates.

Named Values vs. hard-coded secrets in policy XML
On the exam, any scenario asking how to store a backend API key securely without embedding it in policy XML should be answered with Named Values backed by Key Vault. APIM uses its system-assigned Managed Identity to retrieve the Key Vault secret, so no credentials are stored in APIM itself.

② Exam Gotchas
rate-limit vs. rate-limit-by-key — shared key = shared limit
rate-limit applies its counter per subscription key. If two callers (e.g., two mobile app instances) both use the same subscription key, they collectively exhaust the same rate-limit counter — there is no per-IP or per-caller isolation. To independently limit each caller's IP address regardless of which subscription key they use, you must use rate-limit-by-key with counter-key="@(context.Request.IpAddress)". The exam frequently presents scenarios where "each unique IP should get 100 calls/minute" — the only correct policy is rate-limit-by-key, not rate-limit.
Omitting <base/> silently skips parent scope policies
When you write a policy at the Operation scope, the default template includes <base/> in each section, which instructs APIM to execute the parent (API-level) policy at that point. If you delete <base/> from the <inbound> section of an operation policy, the API-level inbound policy — including any validate-jwt or rate-limit configured there — is silently skipped for that operation. This is a common trap: an operation appears to bypass authentication because someone removed <base/> while "cleaning up" the policy XML. The exam may present this as a security scenario where one operation is unprotected.
validate-jwt validates but does NOT authorize — require-claim is separate
validate-jwt checks the token's signature against the provided JWKS/certificate and verifies standard claims (expiry, issuer, audience). It does NOT check whether the token contains a specific role or scope claim. If a valid token for a read-only user calls an admin endpoint, validate-jwt alone will let it through. To enforce role-based authorization, you must add a <required-claims> element inside validate-jwt, or follow up with a choose/when policy that inspects context.Request.Headers["Authorization"] claims. The exam presents scenarios where "only users with the admin role should access this operation" — the answer is validate-jwt with required-claims.

④ CLI / SDK Patterns

Creating an APIM Instance (Azure CLI)

# Create a resource group
az group create --name rg-apim-demo --location eastus

# Create APIM instance — publisher-email and publisher-name are REQUIRED
# --sku-name: Consumption | Developer | Basic | Standard | Premium
az apim create \
  --name my-apim-instance \          # must be globally unique
  --resource-group rg-apim-demo \
  --publisher-email admin@contoso.com \  # required: who receives alerts
  --publisher-name "Contoso API Team" \  # required: shown in Developer Portal
  --sku-name Developer \             # Developer = no SLA, good for dev/test
  --location eastus

Import an API from OpenAPI Specification

# Import a REST API using an OpenAPI (Swagger) spec URL
az apim api import \
  --resource-group rg-apim-demo \
  --service-name my-apim-instance \
  --api-id orders-api \              # unique within the APIM instance
  --path /orders \                   # URL path prefix in the gateway
  --specification-format OpenApiJson \
  --specification-url https://contoso.com/api/swagger.json \
  --display-name "Orders API"

Applying a Policy via ARM / Bicep

// Bicep: set an inbound rate-limit policy at the API level
resource apiPolicy 'Microsoft.ApiManagement/service/apis/policies@2022-08-01' = {
  name: '${apim.name}/${api.name}/policy'
  properties: {
    format: 'xml'        // 'xml' for inline XML; 'xml-link' for external URL
    value: '''
      <policies>
        <inbound>
          <base />
          <rate-limit calls="100" renewal-period="60" />
          <validate-jwt header-name="Authorization" failed-validation-httpcode="401">
            <openid-config url="https://login.microsoftonline.com/{tenant}/.well-known/openid-configuration" />
            <required-claims>
              <claim name="roles" match="any">
                <value>API.Read</value>
              </claim>
            </required-claims>
          </validate-jwt>
        </inbound>
        <backend><base /></backend>
        <outbound><base /></outbound>
        <on-error><base /></on-error>
      </policies>
    '''
  }
}

Named Values — Create via CLI

# Create a plain Named Value (referenced as {{backend-api-key}} in policy)
az apim nv create \
  --resource-group rg-apim-demo \
  --service-name my-apim-instance \
  --named-value-id backend-api-key \
  --display-name "Backend API Key" \
  --value "supersecretvalue123" \
  --secret true                      # --secret true encrypts it at rest

# Create a Key Vault-backed Named Value (preferred for exam scenarios)
az apim nv create \
  --resource-group rg-apim-demo \
  --service-name my-apim-instance \
  --named-value-id kv-backend-key \
  --display-name "KV Backend Key" \
  --key-vault-secret-identifier https://my-vault.vault.azure.net/secrets/BackendKey \
  --secret true                      # always true for KV-backed values

Policy Expressions: C# Inline Syntax

<!-- Single-expression: @(expression) returns a value -->
<set-header name="X-Caller-IP" exists-action="override">
  <value>@(context.Request.IpAddress)</value>
</set-header>

<!-- Multi-statement block: @{...} returns via return statement -->
<rate-limit-by-key
  calls="50"
  renewal-period="60"
  counter-key="@{
    var userId = context.Request.Headers.GetValueOrDefault("X-User-Id", "");
    return string.IsNullOrEmpty(userId)
      ? context.Request.IpAddress   // fall back to IP if no user ID
      : userId;
  }" />

<!-- Accessing JWT claims in policy expressions -->
<set-header name="X-User-Email" exists-action="override">
  <value>@(context.Request.Headers["Authorization"]
      .Split(' ')[1]
      .AsJwt()?.Claims["email"].FirstOrDefault()?.Value ?? "unknown")</value>
</set-header>

Calling APIM with Subscription Key (C# SDK / HttpClient)

using var client = new HttpClient();

// Method 1: Header (preferred — not visible in server logs)
client.DefaultRequestHeaders.Add("Ocp-Apim-Subscription-Key", subscriptionKey);

// Method 2: Query string (fallback — visible in URL/logs)
var url = $"https://my-apim.azure-api.net/orders?subscription-key={subscriptionKey}";

// Trace header — only works if subscription has tracing enabled in APIM portal
client.DefaultRequestHeaders.Add("Ocp-Apim-Trace", "true");

var response = await client.GetAsync("https://my-apim.azure-api.net/orders");
// Trace output is in the Ocp-Apim-Trace-Location response header
⑤ Wrong-Answer Traps
Trap: "Use Consumption tier for VNET-integrated backend access"
The Consumption tier has zero VNET support. It cannot reach backends deployed in private virtual networks. Questions that describe a backend App Service deployed in a VNET (internal VNET integration) and ask which APIM tier supports this should be answered with Premium (for production with SLA) or Developer (for testing only, no SLA). Choosing Consumption tier here is a plausible-sounding wrong answer because Consumption is "modern" and "serverless."
Trap: "mock-response still calls the backend for fallback if there is no mock"
mock-response always terminates the pipeline immediately — it never falls through to the backend. When the policy is present and active, the backend service is never called, regardless of whether the mock data fully covers all scenarios. This is by design: it is used during development to simulate responses before the backend exists. If you need conditional mock behavior (mock only for certain inputs), you must wrap it in a <choose><when condition="..."> block. The exam may imply mock-response is a "soft fallback" — it is not.
Trap: "quota and rate-limit are the same thing — just different time windows"
They are semantically different. rate-limit enforces a burst rate (calls per short window, typically seconds or minutes) and resets automatically each window. quota enforces a total allocation over a longer renewal period (days, weeks, months) and is used for billing/entitlement: "your plan includes 10,000 calls per month." They are complementary — production APIM deployments often use both together: rate-limit to prevent bursts, quota to enforce the total monthly allowance. The exam distinguishes them: "prevent a single client from overwhelming the API in a burst" = rate-limit; "enforce monthly API consumption limits per subscription plan" = quota.
⑥ Practice Q&A
1
You need to limit each unique IP address to 100 requests per minute through your APIM gateway, regardless of which subscription key the caller uses. Which policy and configuration achieves this?
▼
✓ rate-limit-by-key with counter-key="@(context.Request.IpAddress)"
Why not rate-limit?
  • rate-limit applies its counter per subscription key. Two callers from different IPs using the same subscription key share one counter.
  • rate-limit-by-key lets you specify any computed key as the counter discriminator, including context.Request.IpAddress, JWT claims, or custom headers.
  • Configuration: <rate-limit-by-key calls="100" renewal-period="60" counter-key="@(context.Request.IpAddress)" />
  • Place this in the <inbound> section, ideally at Global or Product scope so it applies to all APIs uniformly.
2
An API operation's policy XML has <inbound><rate-limit calls="10" renewal-period="60"/></inbound> with no <base/> tag. The parent API-level policy has validate-jwt in its inbound section. What happens when a caller sends a request to this operation without a JWT token?
▼
✓ The request is processed without JWT validation — the API-level validate-jwt is skipped
Why the operation bypasses JWT validation:
  • The <base/> element is the mechanism by which a child scope inherits and executes its parent scope's policy block.
  • When <base/> is absent from the operation's <inbound> section, the runtime executes only the operation-level inbound policies — the parent's validate-jwt is never invoked.
  • The caller will be rate-limited (10 calls/min) but their token is never verified. A request with an expired, forged, or missing JWT will succeed.
  • Fix: add <base/> before or after the rate-limit depending on desired order. Typically <base/> comes first so authentication runs before rate limiting.
3
Your APIM gateway is deployed in the Consumption tier. You want to cache GET /products responses for 5 minutes to reduce backend load. Which approach is valid?
▼
✓ Configure an external Azure Cache for Redis and use cache-lookup / cache-store policies pointing to it
Consumption tier cache constraint:
  • The Consumption tier has no built-in gateway cache. The cache-lookup and cache-store policies require either the built-in cache (unavailable in Consumption) or an explicitly configured external Redis cache.
  • To enable caching in Consumption, go to APIM > External cache > Add, and add your Azure Cache for Redis connection string.
  • Once configured, cache-lookup/cache-store policies work normally — APIM uses Redis as the backing store.
  • Tiers with built-in cache (Developer, Basic, Standard, Premium) do not require this external configuration step.
4
A developer is building a new API backend that is not yet complete. You need to return sample response data from the APIM gateway immediately so the frontend team can continue development. Which policy provides this and what is its behavior?
▼
✓ mock-response — returns a static response and never calls the backend
mock-response behavior:
  • mock-response is placed in the <inbound> section. When the gateway reaches it, the request pipeline is terminated immediately and a synthetic response is returned to the caller.
  • The backend service receives zero requests — it is not called at all. This is by design and is the key point distinguishing it from a fallback or circuit breaker.
  • The mock data can come from the API definition (examples in OpenAPI spec) or be specified inline in the policy XML via <status-code>, <headers>, and <body> elements.
  • To mock only specific scenarios, wrap it in <choose><when condition="..."><mock-response/></when><otherwise><!-- let through --></otherwise></choose>.
5
A backend API key must be injected into all outbound requests as a header named X-Internal-Key. You want to store this key securely without embedding it as plaintext in the APIM policy XML. What is the correct approach?
▼
✓ Store the key in a Key Vault-backed Named Value; reference it as {{named-value-id}} in the set-header policy
Named Values + Key Vault integration:
  • In the APIM portal, create a Named Value with type "Key Vault" and point it to your Key Vault secret URI. APIM uses its system-assigned Managed Identity to retrieve the secret — no credentials stored in APIM.
  • In the policy XML: <set-header name="X-Internal-Key" exists-action="override"><value>{{backend-api-key}}</value></set-header>
  • The {{backend-api-key}} placeholder is resolved at policy evaluation time, not at deploy time — so rotating the Key Vault secret is reflected automatically after the cache refresh interval (default 4 hours; can trigger refresh manually).
  • Place this policy in the <inbound> or <backend> section so the header is added before the backend call is made.
6
Your organization requires that the APIM gateway be deployed inside a private virtual network so that the Developer Portal and gateway are only accessible from within the corporate network. Which tier and network mode supports this?
▼
✓ Premium tier with Internal VNET mode
APIM VNET modes:
  • External VNET: APIM is injected into a VNET for backend connectivity, but the gateway and Developer Portal remain accessible from the public internet. Available in Developer and Premium tiers.
  • Internal VNET: Both the gateway endpoint and Developer Portal are only accessible from within the VNET (or via VPN/ExpressRoute). Available in Developer and Premium tiers only.
  • For production (SLA required), Internal VNET = Premium tier is the only valid answer. Developer tier supports Internal VNET but carries no SLA.
  • Basic and Standard tiers have no VNET support in any form. Consumption tier also has no VNET support.
7
A validate-jwt policy is configured in the inbound section of an API. A caller sends a request with a valid, unexpired JWT signed by the correct issuer — but the token does not contain the required "admin" role claim. What does APIM return?
▼
✓ 401 Unauthorized — if required-claims is configured; 200 forwarded to backend if required-claims is absent
validate-jwt claim enforcement behavior:
  • validate-jwt by default only verifies the token's signature, expiry (exp), issuer (iss), and audience (aud). It does NOT reject requests based on missing role/scope claims unless you explicitly add <required-claims>.
  • If <required-claims><claim name="roles" match="any"><value>admin</value></claim></required-claims> is present in the policy, a token without the "admin" role claim returns 401 with the body specified by failed-validation-error-message.
  • This is one of the most common exam traps: a token being "valid" (signature OK, not expired) does not mean it is "authorized" (has required claims). The two checks are independent and require separate policy configuration.
  • The failed-validation-httpcode attribute controls the status code returned on failure — it defaults to 401 but can be set to 403 for semantic correctness when the identity is known but lacks permission.
8
You have a subscription key that is approaching compromise. You need to rotate the key without any downtime for callers. What is the correct procedure using APIM's built-in key management?
▼
✓ Update callers to use the secondary key, then regenerate the primary key; update callers back to primary when ready
Zero-downtime key rotation procedure:
  • Every APIM subscription has two keys: primary and secondary. Both keys are active simultaneously — APIM accepts either key for the same subscription.
  • Step 1: Distribute the secondary key to all active callers and confirm they switch to it (no downtime — both keys work during this transition).
  • Step 2: Regenerate the primary key in the APIM portal or via CLI. The old primary key is now invalid. Since callers are using secondary, there is no service interruption.
  • Step 3 (optional): Distribute the new primary key to callers, switch them back, then optionally regenerate secondary to produce a fresh backup key.
  • The exam may ask "how do you revoke a compromised key without downtime" — the two-key system is the answer. Never regenerate the key that callers are currently using.
