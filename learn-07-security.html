Topic 07

01 Azure Key Vault

① Key Concepts

Object Types

Key Vault stores three distinct object types. Each has its own permissions, versioning, and lifecycle management.

Object TypeWhat It StoresKey Details
Secrets Arbitrary strings — connection strings, passwords, API keys Max 25 KB per value. Versioned automatically on update. Can set activation/expiry dates.
Keys RSA (2048/3072/4096) or EC (P-256/P-384/P-521) cryptographic keys Software-protected (Standard SKU) or HSM-backed (Premium SKU). HSM keys never leave the hardware; only operations are performed inside the HSM.
Certificates X.509 certificates with auto-renewal support Can be self-signed or issued by integrated CAs (DigiCert, GlobalSign). The associated private key is stored as a Key object. Import via PFX or PEM.

Access Models: Vault Access Policy vs Azure RBAC

FeatureVault Access Policy (legacy)Azure RBAC (recommended)
Granularity Per object-type (all secrets, all keys, all certificates) — cannot restrict to a single secret Per individual object (e.g., only secret named "db-password")
Assignment scope Vault-level only Management group, subscription, resource group, vault, or individual secret/key/cert
Audit trail Vault diagnostic logs Azure Activity Log + diagnostic logs
Max principals 1024 access policies per vault Azure RBAC limits (2000 role assignments per subscription)
Inheritance No — policy must be set on each vault Yes — role assignments inherit down the scope hierarchy

Key Operations (for Key objects)

OperationPurposeCommon Use Case
Encrypt / Decrypt Use the vault key to encrypt or decrypt data Small payloads; note: for large data, use Wrap/Unwrap with a data-encryption key instead
Sign / Verify Generate or verify a cryptographic signature Code signing, JWT signing, document integrity
Wrap / Unwrap Encrypt/decrypt another key (key encryption key pattern) Envelope encryption: generate a random AES key locally, wrap it with the KV key, store the wrapped key alongside the ciphertext

Soft Delete and Purge Protection

FeatureBehaviourDefault
Soft Delete Deleted objects move to a "deleted" state and are retained for the retention period. Can be recovered with az keyvault secret recover. Enabled by default on all new vaults (cannot be disabled on new vaults). Retention: 7–90 days, default 90.
Purge Protection Prevents permanent deletion (purging) of soft-deleted objects and the vault itself until the retention period expires. Even vault owners and the service cannot purge. Disabled by default. Once enabled, it CANNOT be disabled — this is permanent and irreversible.
Exam tip: Purge protection is required for CMK with Azure Storage and Disk Encryption
When you use a customer-managed key (CMK) in Key Vault to encrypt Azure Storage accounts or Azure Managed Disks, Azure requires purge protection to be enabled on the vault. If you try to configure CMK encryption without purge protection, the operation will fail. This is a common scenario in exam questions.
② Exam Gotchas
Key Vault Contributor does NOT grant secret read access
Key Vault Contributor is a management-plane role. It lets you create vaults, configure access policies, and view vault properties. It does not allow reading, writing, or listing secrets, keys, or certificates. To read a secret you need either: (a) Key Vault Secrets User (RBAC model) — provides Get + List on secrets only, or (b) the Get permission in a Vault Access Policy. Exam questions frequently offer "Key Vault Contributor" as a tempting wrong answer when asking how to give an app access to secrets.
HSM-backed keys require the Premium SKU — not Standard
Software-protected keys (stored encrypted in Azure storage) are available on both Standard and Premium SKUs. HSM-backed keys require Premium SKU. When you run az keyvault key create --protection HSM against a Standard SKU vault, the call will succeed but Azure silently downgrades the key to software protection — the --protection flag is ignored. To truly guarantee HSM protection, you must create a Premium SKU vault first. BYOK (Bring Your Own Key) also requires Premium.
Access policy and RBAC models cannot be mixed for the same scope
A vault is configured in either Vault Access Policy mode or Azure RBAC mode, set via the --enable-rbac-authorization flag. When RBAC mode is active, vault access policies are ignored entirely — assigning a vault access policy has no effect. When you migrate a vault from access policy to RBAC, all existing access policies stop working immediately. Exam questions may describe both an RBAC role assignment and an access policy on the same vault and ask which takes effect.

④ CLI / SDK Patterns
# Create a vault with RBAC authorization model (recommended)
az keyvault create \
  --name myVault \
  --resource-group myRG \
  --location eastus \
  --sku Standard \                          # Standard or Premium (Premium required for HSM)
  --enable-rbac-authorization true \        # Use Azure RBAC instead of access policies
  --enable-purge-protection true \          # WARNING: irreversible once set
  --retention-days 90                       # Soft delete retention (7-90)

# Set a secret
az keyvault secret set \
  --vault-name myVault \
  --name "DbConnectionString" \
  --value "Server=myserver;..."

# Read a secret (requires Key Vault Secrets User role or Get access policy)
az keyvault secret show \
  --vault-name myVault \
  --name "DbConnectionString"

# Create an HSM-backed key (Premium SKU vault required)
az keyvault key create \
  --vault-name myVaultPremium \
  --name "MasterKey" \
  --protection HSM \                        # Silently ignored on Standard SKU!
  --kty RSA-HSM \                           # Key type: RSA-HSM or EC-HSM for HSM-backed
  --size 2048

# Import a certificate (PFX format)
az keyvault certificate import \
  --vault-name myVault \
  --name "MyCert" \
  --file ./mycert.pfx \
  --password "pfxpassword"

# Recover a soft-deleted secret
az keyvault secret recover \
  --vault-name myVault \
  --name "DbConnectionString"

# Purge a soft-deleted secret (only works if purge protection is NOT enabled)
az keyvault secret purge \
  --vault-name myVault \
  --name "DbConnectionString"

⑤ Wrong-Answer Traps
Trap: "Enable purge protection, then disable it to urgently delete a secret"
FALSE. Purge protection is a one-way door. Once enabled on a vault, it cannot be disabled by anyone — including subscription owners and Microsoft support. You must wait for the full soft-delete retention period (up to 90 days) to expire before the vault or its objects can be permanently deleted. There is no override, no support ticket, and no ARM property that undoes it.
Trap: "Assign Key Vault Contributor to the app's service principal so it can read secrets at runtime"
FALSE. Key Vault Contributor grants management-plane access only (create/update/delete vaults, manage policies). It has zero data-plane permissions. At runtime, to read a secret value, the identity needs Key Vault Secrets User (RBAC mode) or the Get permission on secrets in an access policy. Giving Contributor to an application service principal is both insecure and non-functional for secret retrieval.
Trap: "Use the Encrypt key operation to protect a 50 MB file with a vault key"
FALSE — and impractical. The Encrypt operation on a Key Vault key is designed for small payloads (limited to the key size minus padding overhead; for RSA-2048 with OAEP that is about 214 bytes of plaintext). For large data, use envelope encryption: generate a random symmetric key (AES-256) locally, encrypt the file with it, then use Key Vault's Wrap operation to encrypt the symmetric key. Store the wrapped key alongside the ciphertext.
⑥ Practice Q&A
1
A developer is assigned the Key Vault Contributor role on a vault. They run az keyvault secret show and receive a 403 Forbidden error. What is the most likely cause?
▼
✓ Key Vault Contributor is management-plane only
Answer: Key Vault Contributor does not grant any data-plane permissions. The developer can manage vault configuration (create vaults, set access policies, view vault metadata) but cannot read, write, or list secrets, keys, or certificates.

  • To read secret values, the developer needs Key Vault Secrets User (Azure RBAC mode) or the Get permission in a Vault Access Policy.
  • To read and list, they need Key Vault Secrets User (Get + List) or both permissions in the access policy.
  • This is the single most common Key Vault exam trap — do not confuse management-plane roles with data-plane access.
2
Your organization requires that deleted Key Vault secrets cannot be permanently destroyed for at least 30 days after deletion, even by global administrators. Which combination of vault properties must be configured?
▼
✓ Soft Delete enabled + Purge Protection enabled
Answer: Both soft delete (set retention days to 30 or more) and purge protection must be enabled.

  • Soft delete alone retains objects in a "deleted" state, but an administrator with the correct permissions can still purge (permanently delete) them immediately.
  • Purge protection prevents the purge operation from executing until the retention period expires — no one can bypass this, including subscription owners.
  • Set --retention-days 30 (or higher) to ensure the 30-day minimum.
  • Once purge protection is enabled, it cannot be disabled. The vault must be explicitly created with this setting in mind.
3
A vault uses the Azure RBAC authorization model. An operator creates a Vault Access Policy granting a service principal Get and List on secrets. The service principal still receives 403 errors when calling GetSecret. Why?
▼
✓ RBAC mode overrides access policies entirely
Answer: When a vault has --enable-rbac-authorization true, vault access policies are completely ignored. The two models do not coexist — RBAC mode takes exclusive control of data-plane authorization.

  • Creating an access policy on an RBAC-enabled vault will not produce an error, but the policy has no effect.
  • The fix is to assign an Azure RBAC role instead: Key Vault Secrets User for read access, or Key Vault Secrets Officer for read/write.
  • The role assignment can be scoped to the vault or down to an individual secret URI.
4
You need to create a key that is guaranteed to never exist in software — all cryptographic operations must occur inside hardware. What is required?
▼
✓ Premium SKU vault + --kty RSA-HSM or EC-HSM
Answer: A Premium SKU Key Vault is required for HSM-backed keys. On a Standard SKU vault, specifying --protection HSM is silently ignored and the key is created as software-protected.

  • Use --kty RSA-HSM or --kty EC-HSM when creating the key to explicitly request HSM protection.
  • HSM-backed keys support the BYOK (Bring Your Own Key) workflow — generate the key material in your own HSM, wrap it, and import it so it is only ever unwrapped inside Azure's HSM.
  • With HSM keys, key.managed will be null (or false) and key.keyType will show RSA-HSM in the response.
  • HSM keys have a lower transaction limit (1000/10s vs 2000/10s for software keys).
5
A certificate stored in Key Vault is approaching its expiration date. You want Azure to automatically renew it with no manual steps. Which certificate issuer type supports fully automated renewal via Key Vault's built-in integration?
▼
✓ Integrated CAs: DigiCert or GlobalSign
Answer: Key Vault supports fully automated certificate lifecycle management (issuance and renewal without any manual step) only when using DigiCert or GlobalSign as the certificate authority through the integrated CA partnership.

  • For self-signed certificates, Key Vault can generate and auto-renew them automatically using its own self-signed issuer.
  • For other third-party CAs, Key Vault generates a CSR that must be manually submitted to the CA — this is a semi-automated flow, not fully automated.
  • Renewal can be triggered at a configurable percentage of the certificate lifetime (e.g., 80%) or a fixed number of days before expiry.
  • Renewal events fire to Event Grid, allowing downstream automation (e.g., update an App Service binding).
6
An application needs to encrypt 1 GB of data using a key stored in Key Vault. The security requirement is that the key never leaves Key Vault. Which approach is correct?
▼
✓ Envelope encryption — Wrap/Unwrap pattern
Answer: Use envelope encryption: generate a random AES-256 key locally (or in memory), encrypt the 1 GB payload with AES locally, then call Key Vault's WrapKey operation to encrypt the AES key using the vault key. Store the wrapped key alongside the ciphertext.

  • The vault key (key encryption key, or KEK) never leaves the vault — only the WrapKey and UnwrapKey operations are called against it.
  • The Encrypt/Decrypt operations on vault keys are limited to tiny payloads (the key size minus padding, e.g., ~214 bytes for RSA-2048 with OAEP).
  • To decrypt: call UnwrapKey to recover the AES key, then decrypt the payload locally.
  • This is exactly how Azure Storage Service Encryption with CMK works internally.
7
After deleting a secret from a Key Vault that has soft delete with a 90-day retention period and purge protection enabled, a compliance audit requires the secret to be recoverable for 90 days. An administrator wants to release the secret name for reuse immediately. What can they do?
▼
✓ Nothing — the name is locked until the retention period expires
Answer: With purge protection enabled, the deleted secret name cannot be purged or reused until the 90-day retention period expires. There is no workaround.

  • Soft-deleted secrets occupy their name in the vault's deleted state — you cannot create a new secret with the same name while the deleted one is in retention.
  • Purge protection prevents the purge operation (which would free the name) from executing before retention expires.
  • No one — not the vault owner, not a subscription admin, not Microsoft Support — can bypass purge protection.
  • Best practice: design secret names to include a version or date suffix (e.g., db-password-v2) if you anticipate needing to cycle names quickly.
8
You want to grant a team of developers read-only access to all secrets in a specific Key Vault (RBAC mode) but ensure they cannot access keys or certificates. Which built-in role should you assign?
▼
✓ Key Vault Secrets User
Answer: Key Vault Secrets User grants get and list on secrets only. It has no permissions for keys or certificates.

  • Key Vault Secrets Officer — get, list, set, delete, recover, backup, restore on secrets. Too broad for read-only.
  • Key Vault Reader — management-plane read (vault metadata, policies) but NO data-plane access to secret values.
  • Key Vault Administrator — full data-plane access to all object types. Never assign this to developers.
  • Assign the role at the vault scope to cover all secrets, or narrow to an individual secret resource ID for per-secret access control.

02 Managed Identities

① Key Concepts

System-Assigned vs User-Assigned Identity

PropertySystem-AssignedUser-Assigned
Lifecycle Tied to the Azure resource. Automatically deleted when the resource is deleted. Independent. Persists until explicitly deleted. Survives resource deletion.
Sharing One identity per resource. Cannot be shared across resources. Can be assigned to multiple resources simultaneously (e.g., 50 VMs all sharing one identity).
Service Principal Created automatically in AAD when enabled. Deleted automatically with the resource. Created when the user-assigned identity is created. Must be manually deleted.
Role assignments Must be re-created each time the resource is recreated. Role assignments persist on the user-assigned identity across resource recreations.
Best for Workloads with unique permissions; single resource accessing its own resources. Workloads requiring pre-authorization before the resource exists; shared identity across resource sets.

How Managed Identity Works — Token Acquisition Flow

  1. Enable managed identity on the Azure resource (App Service, VM, Function App, etc.)
  2. Grant RBAC roles or access policies to the identity's service principal on target resources (e.g., Key Vault Secrets User on a vault).
  3. Application code calls the Instance Metadata Service (IMDS) endpoint to retrieve a bearer token. The IMDS endpoint is always http://169.254.169.254 — a link-local address only reachable from inside the Azure resource.
  4. IMDS returns a signed JWT access token valid for the target resource (e.g., https://vault.azure.net).
  5. The application presents the token as a Bearer Authorization header when calling the Azure resource's REST API.
IMDS Token Request DetailValue
Endpoint URLhttp://169.254.169.254/metadata/identity/oauth2/token
Required headerMetadata: true
Required query paramsapi-version=2018-02-01 and resource=<audience>
Example resource for Key Vaulthttps://vault.azure.net
Example resource for Storagehttps://storage.azure.com
Token cachingIMDS caches the token and refreshes it automatically before expiry. Avoid implementing your own caching on top — call IMDS directly per request and let it handle caching.

DefaultAzureCredential — Credential Chain Order

DefaultAzureCredential (from the Azure Identity SDK) tries credential providers in this exact order, stopping at the first that succeeds:

PriorityCredential TypeActivated By
1EnvironmentCredentialAZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_CLIENT_SECRET or AZURE_CLIENT_CERTIFICATE_PATH env vars
2WorkloadIdentityCredentialAZURE_FEDERATED_TOKEN_FILE env var (AKS Workload Identity / Azure AD Workload Identity)
3ManagedIdentityCredentialRunning inside an Azure service with an assigned managed identity (App Service, VM, AKS pod, etc.)
4SharedTokenCacheCredentialAzure tools (Visual Studio, VS Code) signed in via shared token cache on the developer machine
5VisualStudioCredentialVisual Studio "Tools > Azure Service Authentication" sign-in
6VisualStudioCodeCredentialVS Code Azure Account extension sign-in
7AzureCliCredentialaz login on the developer machine
8AzurePowerShellCredentialConnect-AzAccount in PowerShell on the developer machine
9InteractiveBrowserCredentialDisabled by default; enables interactive browser pop-up as last resort
Production behavior: EnvironmentCredential takes priority over ManagedIdentityCredential
If you set AZURE_CLIENT_ID, AZURE_TENANT_ID, and AZURE_CLIENT_SECRET as environment variables in an App Service that also has a managed identity, DefaultAzureCredential will use the service principal from the env vars — NOT the managed identity. This is a common misconfiguration. In production, either use managed identity (and set no env vars) or service principal env vars (and disable MI). Do not mix.
② Exam Gotchas
Owner role does NOT grant access to Key Vault secrets
Assigning the Owner role to a managed identity does not allow it to read Key Vault secrets or keys. Owner is a management-plane role — it can manage resources, assign roles to others, and configure vault properties. Data-plane access (reading secret values) requires an explicit data-plane role: Key Vault Secrets User (RBAC) or a vault access policy with Get permission. Exam questions phrase this as: "The function app has Owner on the vault but gets 403 on GetSecret — what must you add?"
System-assigned identity's role assignments do NOT survive resource deletion
When a resource with a system-assigned managed identity is deleted, the service principal is automatically deleted. If you recreate the resource, a brand-new service principal is created with a different object ID. All previous role assignments are now orphaned — they pointed to the old deleted principal and have no effect. You must reassign every RBAC role to the new identity. This is a significant operational hazard and an exam scenario: "After redeploying the App Service, secrets access stopped working."
IMDS endpoint is only reachable from within an Azure resource
http://169.254.169.254 is a link-local address that is only routable from inside an Azure VM, App Service, Container Instance, or similar compute. Code that calls IMDS will work in Azure and fail silently (connection refused or timeout) on a local developer machine. DefaultAzureCredential handles this gracefully by falling through to CLI/VS credentials locally. However, if you call the IMDS endpoint directly in code without using the SDK, you must implement fallback logic yourself — the exam may ask about this.

⑤ Wrong-Answer Traps
Trap: "Set AZURE_CLIENT_ID to the managed identity's client ID to ensure it is used in production"
FALSE — this backfires. Setting AZURE_CLIENT_ID alone triggers EnvironmentCredential to attempt a service principal login, which requires AZURE_CLIENT_SECRET or AZURE_CLIENT_CERTIFICATE_PATH as well. Without those, EnvironmentCredential fails, and DefaultAzureCredential will try the next credential in the chain. However, some SDK versions may throw an error instead of falling through. The safe pattern is: do NOT set any AZURE_* env vars in production when you want managed identity. Let DefaultAzureCredential discover the MI automatically via IMDS.
Trap: "A function app can only have one managed identity — system-assigned or user-assigned, not both"
FALSE. Azure resources support having both a system-assigned MI and one or more user-assigned MIs simultaneously. The types are not mutually exclusive. A common pattern: use system-assigned MI for the app's own resource-specific access (e.g., its own Key Vault), and a shared user-assigned MI for cross-cutting concerns (e.g., a centralized logging storage account shared by many services).
Trap: "Use principalId and clientId interchangeably when assigning RBAC roles"
FALSE. These are different identifiers with different uses. principalId (also called the object ID) is what you pass to az role assignment create --assignee — it is the AAD object ID of the service principal. clientId (application ID) is what you pass to SDK options like ManagedIdentityClientId when specifying which user-assigned MI to use for token acquisition. Confusing them causes role assignments to fail silently or the SDK to authenticate with the wrong identity.
⑥ Practice Q&A
1
An App Service uses a system-assigned managed identity to access Key Vault. After the dev team deletes and recreates the App Service in a new deployment, the application receives 403 errors on Key Vault. No code changes were made. What is the root cause?
▼
✓ System-assigned MI service principal changed — RBAC roles must be reassigned
Answer: When the App Service was deleted, its system-assigned managed identity's service principal was automatically deleted from AAD. The new App Service has a new service principal with a different object ID. The Key Vault RBAC role assignment (Key Vault Secrets User) pointed to the old, now-deleted object ID — it no longer applies to anything.

  • The fix is to run az role assignment create again, using the new App Service's principalId from az webapp identity show.
  • To avoid this in future deployments, switch to a user-assigned managed identity, which persists independently of the App Service lifecycle.
  • With user-assigned MI, role assignments are made once to the MI object and survive resource recreations.
2
Your application uses DefaultAzureCredential. It works correctly in production (using managed identity) but fails in local development with the error "EnvironmentCredential: Missing AZURE_TENANT_ID". What is the most likely configuration issue?
▼
✓ AZURE_CLIENT_ID is set locally without the other required env vars
Answer: The environment has AZURE_CLIENT_ID set (perhaps as a leftover configuration) but not AZURE_TENANT_ID or AZURE_CLIENT_SECRET. DefaultAzureCredential sees AZURE_CLIENT_ID and activates EnvironmentCredential, which then fails because the required tenant ID and secret are missing instead of falling through to AzureCliCredential.

  • Fix: remove the partial AZURE_CLIENT_ID env var so EnvironmentCredential is not triggered.
  • Alternatively, set all three env vars for a service principal: AZURE_CLIENT_ID, AZURE_TENANT_ID, and AZURE_CLIENT_SECRET.
  • Or simply run az login and let AzureCliCredential (priority 7) handle local auth.
3
You need to retrieve an access token for Azure Key Vault from within a Python Azure Function using the managed identity. The IMDS URL is http://169.254.169.254/metadata/identity/oauth2/token. Which parameter specifies the target Azure service?
▼
✓ resource=https://vault.azure.net
Answer: The resource query parameter specifies the audience (the Azure service) for which the token is requested. For Key Vault, the value is https://vault.azure.net.

  • Full URL: http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://vault.azure.net
  • The Metadata: true header is required — requests without it are rejected with a 400.
  • Other common resource values: https://storage.azure.com (Storage), https://management.azure.com (ARM), https://graph.microsoft.com (Graph API).
  • In practice, use the Azure Identity SDK (DefaultAzureCredential) instead of calling IMDS directly — it handles token caching, refresh, and credential fallback.
4
Fifty Azure VMs all need access to the same shared Azure Storage account. Which managed identity approach minimizes operational overhead for role assignment management?
▼
✓ Single user-assigned MI assigned to all 50 VMs
Answer: Create one user-assigned managed identity, assign it the Storage Blob Data Reader (or appropriate) role on the storage account once, then attach that same MI to all 50 VMs.

  • With 50 system-assigned MIs, you would need to create 50 separate role assignments — one per VM. Any new VM added to the fleet requires a new role assignment.
  • With a user-assigned MI, one role assignment covers all current and future VMs that get the MI attached.
  • The VMs specify the user-assigned MI's client ID in SDK code to avoid ambiguity (in case they also have a system-assigned MI).
  • The user-assigned MI is an independent ARM resource and must be explicitly created and deleted separately from the VMs.
5
A developer calls the IMDS endpoint directly (without SDK) and does not include any specific header. The call returns HTTP 400. What is missing?
▼
✓ The required Metadata: true header
Answer: The IMDS endpoint requires the HTTP header Metadata: true on every request. This header was designed as a security measure to prevent DNS rebinding attacks — a browser's CORS preflight mechanism would not send custom headers to an arbitrary IP, protecting the endpoint from cross-site request forgery.

  • Without the header: HTTP 400 Bad Request.
  • From outside an Azure resource (local machine): connection timeout or refused — IMDS is only reachable from within Azure.
  • Also required: api-version=2018-02-01 as a query parameter and the resource query parameter specifying the target audience.
6
An App Service has both a system-assigned MI and a user-assigned MI (clientId: abc123). The application code uses new DefaultAzureCredential() with no options. Which identity is used for token acquisition in Azure?
▼
✓ System-assigned MI — it is the default when no clientId is specified
Answer: When multiple managed identities are attached to a resource and no ManagedIdentityClientId is specified, the SDK defaults to the system-assigned MI. If only user-assigned MIs are present, the SDK uses the first one, which may be non-deterministic.

  • To explicitly use the user-assigned MI, pass ManagedIdentityClientId = "abc123" to DefaultAzureCredentialOptions.
  • Alternatively, use new ManagedIdentityCredential("abc123") directly to bypass the credential chain entirely.
  • Always be explicit when multiple MIs are present — relying on implicit defaults leads to hard-to-diagnose permission errors.
7
A function app's managed identity is granted the Owner role on the resource group containing the Key Vault. After deployment, GetSecret calls return 403. What additional step is required?
▼
✓ Assign a Key Vault data-plane role (Key Vault Secrets User) separately
Answer: Owner is a management-plane role. Even scoped to the resource group or the vault itself, it does not grant data-plane access to Key Vault objects (secrets, keys, certificates).

  • Management plane: create/delete vaults, configure access policies, manage RBAC assignments. Owner covers this.
  • Data plane: read/write/delete secret values, perform key operations. Owner does NOT cover this.
  • The fix: assign Key Vault Secrets User on the vault (or a specific secret) to the function app's MI. This is separate from the Owner assignment and both can coexist.
  • If the vault uses Vault Access Policy model, Owner does not automatically grant access policy permissions either — you must create an explicit access policy entry.

03 Azure App Configuration

① Key Concepts

Core Capabilities

Azure App Configuration is a managed service for centralized application settings and feature flags. It decouples configuration from code, enabling dynamic updates without redeployment.

CapabilityDescriptionKey Detail
Key-Value Store Store application settings as key-value pairs with optional labels (e.g., Environment=Production) Keys are hierarchical by convention (e.g., AppSettings:ConnectionString). Labels allow environment-specific or app-specific variants of the same key.
Feature Flags Manage feature toggles centrally using the feature management library Stored as special key-value pairs prefixed .appconfig.featureflag/. Supports filters: time-window, percentage rollout, targeting (user/group).
Key Vault References Store a reference to a Key Vault secret URI; the SDK resolves it to the actual secret value at runtime App Config stores the URI, NOT the secret value. The app needs access to BOTH App Config AND Key Vault to retrieve the value.
Snapshots Point-in-time immutable capture of the configuration store's key-value pairs Useful for rollback. Retained for 30 days after the snapshot expires. Cannot be modified after creation.
Dynamic Refresh Application polls for config changes at a configurable interval using a sentinel key pattern The app monitors one "sentinel" key for changes. When the sentinel changes, it reloads the full configuration. This avoids polling every key individually.

Labels — Multi-Environment Configuration

Labels are a key differentiator of App Configuration. A single key can have multiple values differentiated by label:

Key Vault References vs Direct Secrets

Sentinel key pattern for efficient dynamic refresh
Instead of polling every configuration key for changes (expensive), add a single "sentinel" key (e.g., Sentinel with value 1) to your App Config store. Configure the SDK to watch only this key. When you update configuration, increment the sentinel value. The SDK detects the sentinel change and reloads all keys. This minimizes API calls and gives you explicit control over when a refresh is triggered.
② Exam Gotchas
App Configuration does NOT store secret values — dual access is required
A Key Vault Reference in App Configuration stores only the URI pointing to a Key Vault secret. When the application reads that configuration key, the App Config SDK fetches the URI from App Config and then makes a separate call to Key Vault to retrieve the actual value. This means the application needs both: access to the App Configuration store (App Configuration Data Reader role or connection string) AND access to Key Vault (Key Vault Secrets User role or access policy). Removing Key Vault access while keeping App Config access will cause runtime failures when resolving KV references.
Dynamic config refresh is NOT automatic — it requires a sentinel key and polling interval
App Configuration does not push configuration changes to applications, and the SDK does not check for changes on every request (that would be prohibitively expensive). You must explicitly configure: (1) a refresh interval via options.SetRefreshInterval(TimeSpan), (2) a sentinel key for the SDK to watch, and (3) call refresher.TryRefreshAsync() in a middleware or background service. A configuration change in the portal does NOT propagate to running applications until the next polling cycle triggers a refresh.
Feature flags in App Config are managed by a separate feature management library
Reading feature flags is NOT done with the standard key-value API. Feature flags use the Microsoft.FeatureManagement library. In .NET, you inject IFeatureManager and call await featureManager.IsEnabledAsync("MyFeature"). The flag value stored in App Configuration (.appconfig.featureflag/MyFeature) is a JSON object with conditions and filters — it is not a simple boolean. Do not attempt to read feature flags as ordinary key-value pairs.

# Create an App Configuration store

  --name myAppConfig \

// C# .NET: Add App Configuration provider in Program.cs
using Azure.Identity;

var builder = WebApplication.CreateBuilder(args);

builder.Configuration.AddAzureAppConfiguration(options =>
{
    options
        .Connect(
            new Uri("https://myAppConfig.azconfig.io"),
            new DefaultAzureCredential()              // Uses MI in production
        )
        .Select("AppSettings:*", "production")        // Keys matching pattern, label=production
        .ConfigureKeyVault(kv =>
        {
            kv.SetCredential(new DefaultAzureCredential()); // For resolving KV references
        })
        .ConfigureRefresh(refresh =>
        {
            refresh
                .Register("Sentinel", refreshAll: true)   // Watch sentinel key
                .SetRefreshInterval(TimeSpan.FromMinutes(5)); // Poll every 5 minutes
        });
});

// Register the refresher service so IConfigurationRefresherProvider is injectable
builder.Services.AddAzureAppConfiguration();

var app = builder.Build();

// Add middleware to trigger refresh on incoming requests
app.UseAzureAppConfiguration();      // Calls TryRefreshAsync() on each request (after interval)
// C# .NET: Feature flags with IFeatureManager
using Microsoft.FeatureManagement;

// Register feature management (reads flags from App Configuration)
builder.Services.AddFeatureManagement();

// In a controller or service:
public class DashboardController : ControllerBase
{
    private readonly IFeatureManager _featureManager;

    public DashboardController(IFeatureManager featureManager)
    {
        _featureManager = featureManager;
    }

    public async Task<IActionResult> GetDashboard()
    {
        if (await _featureManager.IsEnabledAsync("BetaDashboard"))
        {
            return Ok(new BetaDashboardViewModel());
        }
        return Ok(new StandardDashboardViewModel());
    }
}
⑤ Wrong-Answer Traps
Trap: "Using a Key Vault Reference in App Config means the app only needs App Config access"
FALSE. A Key Vault Reference is exactly that — a reference. The secret value lives in Key Vault, and the App Config SDK makes a live call to Key Vault to retrieve it at runtime. The application's identity must have explicit data-plane access to Key Vault (Key Vault Secrets User role or Get access policy) in addition to the App Configuration Data Reader role on the App Config store. Both access grants are required; neither alone is sufficient.
Trap: "Updating a value in App Configuration immediately refreshes it in all running applications"
FALSE. App Configuration has no push mechanism to notify running applications. Configuration changes only reach a running app when: (1) the configured refresh interval has elapsed since the last check, (2) a request arrives (triggering the UseAzureAppConfiguration middleware), and (3) the sentinel key has changed. If an application is idle (no incoming requests), configuration changes will not propagate until the next request after the refresh interval expires. For time-sensitive updates, use a background refresh timer instead of the request-based middleware.
Trap: "Feature flags can be read using the standard IConfiguration key-value API"
FALSE. Feature flags are stored in App Configuration as structured JSON objects under the key prefix .appconfig.featureflag/. Reading them as a plain string via IConfiguration["FeatureName"] returns the raw JSON — not a boolean. To properly evaluate feature flags including conditional activation rules (time windows, percentage rollouts, user targeting), you must use the Microsoft.FeatureManagement library and inject IFeatureManager. The IsEnabledAsync method evaluates all filters configured on the flag.
⑥ Practice Q&A
1
An application is reading a Key Vault Reference from Azure App Configuration. The App Config SDK resolves the key to the actual secret value at runtime. The application identity has App Configuration Data Reader on the App Config store but receives a 403 error. What additional permission is required?
▼
✓ Key Vault Secrets User role on the referenced Key Vault
Answer: A Key Vault Reference stores only the KV secret URI in App Configuration. When the SDK resolves it, it makes a live API call to Key Vault using the application's own credential. The application needs Key Vault Secrets User (RBAC mode) or Get in an access policy on the referenced vault.

  • App Configuration Data Reader grants access to the App Config store only — it does not delegate Key Vault access.
  • Both permissions are required simultaneously: App Config access to get the URI, Key Vault access to get the value.
  • The credential passed to ConfigureKeyVault(kv => kv.SetCredential(...)) is used for KV calls. In production, this is typically the same DefaultAzureCredential that resolves to the managed identity.
2
A team updates a configuration value in App Configuration at 14:00. The running application (refresh interval set to 30 minutes, sentinel key configured) last checked at 13:45. At what earliest time will the application pick up the new value?
▼
✓ At 14:15 — after the 30-minute interval from last check expires
Answer: The SDK checked at 13:45 and set the next check time to 14:15 (13:45 + 30 minutes). The config change at 14:00 is not visible until the next refresh cycle at 14:15, when the next incoming request triggers the middleware to poll App Configuration, detects the sentinel key has changed, and reloads all keys.

  • The refresh interval timer starts from the last successful refresh, not from the last change.
  • If the application receives no requests between 14:15 and later, the update may be delayed further (request-driven middleware only fires on incoming traffic).
  • For time-critical config updates, use IConfigurationRefresher in a IHostedService background timer instead of the request-based middleware.
3
You want to gradually roll out a new UI feature to 20% of users, increasing to 100% over two weeks, using Azure App Configuration. Which feature flag filter type is most appropriate?
▼
✓ Percentage rollout filter (Microsoft.Percentage)
Answer: The Microsoft.Percentage built-in filter enables a feature for a configurable percentage of requests. Set it to 20 initially, then increase the percentage in App Configuration without redeployment.

  • Microsoft.TimeWindow — enables a feature only between specified start and end times. Useful for scheduled maintenance windows or timed promotions, not gradual rollouts.
  • Microsoft.Targeting — enables a feature for specific users, groups, or with a percentage fallback. More precise than a pure percentage rollout — you can target named groups and set a default percentage for everyone else.
  • Percentage filter decisions are stateless per request — the same user may see the feature on one request and not another. For consistent per-user experiences, use the Targeting filter with user claims.
4
You need to capture the exact state of your production App Configuration at the end of the quarter for audit and rollback purposes. What App Configuration feature should you use?
▼
✓ App Configuration Snapshot
Answer: An App Configuration Snapshot is an immutable, point-in-time capture of the key-value pairs matching specified filters. Once created, a snapshot cannot be modified.

  • Snapshots can be created with key and label filters to capture only a subset of the store (e.g., all keys with the production label).
  • Retained for 30 days after their configured expiry (or until manually deleted). Can also be "archived" for indefinite retention.
  • Applications can load configuration from a specific snapshot by passing the snapshot name to the SDK, enabling consistent rollback to a known-good configuration state.
  • Up to 200 snapshots per Standard tier store; 10 on Free tier.
5
Your .NET application uses AddAzureAppConfiguration. You have the Sentinel key registered for refresh. After updating the database connection string and incrementing the Sentinel in the portal, the running application continues to use the old connection string even after 10 minutes. The refresh interval is set to 5 minutes. What is the most likely cause?
▼
✓ app.UseAzureAppConfiguration() middleware is missing or the app has received no requests
Answer: The refresh mechanism in AddAzureAppConfiguration is triggered by the UseAzureAppConfiguration() middleware, which fires TryRefreshAsync() on each incoming HTTP request. If the middleware is not registered, or if the application has received no requests since the interval expired, the refresh never occurs.

  • Verify app.UseAzureAppConfiguration() is called in the middleware pipeline in Program.cs.
  • For worker services or applications without HTTP requests, use IConfigurationRefresher.TryRefreshAsync() in a BackgroundService timer loop instead.
  • Also verify that builder.Services.AddAzureAppConfiguration() is called to register the refresher in DI — without this, the middleware cannot resolve the refresher.
6
Two applications (App A and App B) share the same App Configuration store. They need different values for the key Api:BaseUrl. What is the recommended approach using App Configuration's built-in features?
▼
✓ Use Labels — same key with different labels per application
Answer: Create the key Api:BaseUrl twice — once with the label app-a and once with label app-b. Each application selects its label in the SDK configuration:

  • App A: .Select("*", "app-a") — loads all keys with the app-a label.
  • App B: .Select("*", "app-b") — loads all keys with the app-b label.
  • Unlabeled keys serve as defaults — app reads unlabeled first, then overlays the labeled values. This allows shared defaults with per-app overrides.
  • An alternative pattern is to key-prefix by app name (e.g., AppA:Api:BaseUrl), but labels are the idiomatic App Configuration approach as they keep key names clean and consistent.
7
A developer wants to enable a feature flag named NewCheckout only between 09:00 and 17:00 UTC on weekdays, automatically disabling itself outside those hours without any code deployment. Which feature flag filter achieves this?
▼
✓ Microsoft.TimeWindow filter
Answer: The Microsoft.TimeWindow built-in filter enables a feature flag only between a configured start and end time (ISO 8601 datetime with timezone offset). The evaluation happens in the IFeatureManager.IsEnabledAsync call at runtime — no code change or deployment needed.

  • Configure: Start: "Mon, 01 Jan 2024 09:00:00 +00:00" and End: "Mon, 01 Jan 2024 17:00:00 +00:00" with a recurring weekly pattern.
  • Note: The basic TimeWindow filter does not natively support recurring weekly schedules — you would need to update the window each week or implement a custom filter for recurrence.
  • Microsoft.Percentage — for random percentage enablement. Microsoft.Targeting — for user/group targeting. Neither handles time-based activation.
8
An App Configuration store is in the Free tier. A team needs geo-replication to a secondary Azure region for high availability. What must they do?
▼
✓ Upgrade the store to the Standard tier
Answer: Geo-replication (replica regions) is only available on the Standard tier. The Free tier does not support it.

  • Free tier limits: 10,000 key-values, 1 MB storage, 1,000 requests/day, no geo-replication, no private endpoints, no SLA.
  • Standard tier: 10,000 key-values included (then pay-per-key), 1 GB storage, geo-replication to up to 10 replica regions, private endpoint support, 99.9% SLA.
  • After upgrading and adding a replica, both the primary and replica regions are read-write. Client SDKs can be configured with multiple endpoints for automatic failover.
  • Key Vault References are always resolved from Key Vault directly — geo-replication of App Config does not affect Key Vault HA requirements separately.
