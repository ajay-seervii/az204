1 Azure Message Queues: Storage Queue vs Service Bus

① Key Concepts

Storage Queue

Azure Storage Queue is the simplest queuing mechanism in Azure. It is REST-based, lives inside a Storage Account, and is designed for high-volume, loosely coupled workloads where advanced messaging features are not required. Messages are stored as plain text (typically Base64-encoded for binary content).

  • Max message size: 64 KB per message
  • Max TTL (Time-to-Live): 7 days (default; configurable up to 7 days)
  • Max queue storage: 500 TB — essentially unlimited for most workloads
  • Ordering: Best-effort FIFO only — no guarantees
  • Poison messages: No built-in dead-letter queue. You must check DequeueCount manually and move messages to a separate "poison" queue in your own code
  • Topics/Subscriptions: Not supported
  • Sessions: Not supported
  • Transactions: Not supported across operations
  • Visibility timeout: When a message is dequeued, it becomes invisible to other consumers for the visibility timeout period (default 30 seconds, max 7 days). If not deleted within that window, it reappears.

Service Bus Queue

Azure Service Bus is an enterprise message broker supporting advanced patterns: sessions, transactions, dead-lettering, topics, and subscriptions. It comes in two tiers — Standard and Premium — with significantly different message size limits.

  • Max message size: 256 KB (Standard tier), 100 MB (Premium tier)
  • Max TTL: Configurable; default is 14 days, maximum is effectively unlimited (TimeSpan.Max)
  • Max queue size: 1 GB to 80 GB depending on tier and configuration
  • Dead-letter queue (DLQ): Built-in. Messages exceeding max delivery count (default 10) or that expire are automatically moved to <queue-name>/$DeadLetterQueue
  • Sessions: Enables FIFO processing per session ID. Must be enabled at queue/subscription creation — cannot be toggled on an existing entity
  • Transactions: Atomic send/receive across multiple Service Bus entities within the same namespace
  • Topics + Subscriptions: Pub/sub pattern. One topic, multiple subscriptions with SQL-like or correlation filter rules
  • Lock duration: Default 60 seconds. Message is locked for the receiver; must call CompleteAsync() or AbandonAsync() before the lock expires, or the message re-enqueues
  • Receive modes: Peek-Lock (default, safe) and Receive-and-Delete (destructive, faster)
  • Max delivery count: Default 10. After 10 unsuccessful deliveries, message goes to DLQ automatically
💡
Peek-Lock is the safe default
In Peek-Lock mode the message is not removed from the queue when received. It is merely invisible (locked) while you process it. You must call CompleteMessageAsync() to permanently delete it. If your processing code crashes, the lock expires and the message becomes visible again for retry — giving you at-least-once delivery semantics. Receive-and-Delete removes the message immediately on receipt; if your code then crashes, the message is lost.

💡
Topic Subscription Filters
Service Bus subscriptions support two filter types: SQL filters (arbitrary property expressions, e.g. sys.Label = 'order' AND Priority > 3) and Correlation filters (match specific properties like CorrelationId, Label, To, ReplyTo — faster and cheaper to evaluate). A third type, True/False filters, either passes all messages or none. The default filter on a new subscription is TrueFilter (pass all).
② Exam Gotchas
🚨
Gotcha: Peek-Lock Does NOT Delete the Message
In Peek-Lock mode, receiving a message does not remove it. The message is locked (invisible to other consumers) for the duration of lockDuration (default 60 seconds). If your code receives the message but then throws an exception before calling CompleteMessageAsync(), the lock will eventually expire and the message will reappear in the queue — incrementing the delivery count. After 10 delivery attempts (default maxDeliveryCount), the message is automatically moved to the dead-letter queue. This is by design for at-least-once delivery. If you process a message successfully but forget to call CompleteAsync(), you will process the same message 10 times before it is dead-lettered.
🚨
Gotcha: Storage Queue Has NO Built-in Dead-Letter Queue
This is one of the highest-frequency traps on the exam. Azure Storage Queue does not have a dead-letter queue. If a message fails processing repeatedly, it simply continues to reappear after the visibility timeout expires — forever, until its TTL (max 7 days) elapses. To implement poison message handling with Storage Queue you must: (1) read the DequeueCount property on the retrieved message, (2) compare it against your threshold (e.g., 5), (3) if exceeded, explicitly enqueue it into a separate queue you name (e.g., myqueue-poison), and (4) delete it from the original queue. This is entirely manual. Contrast this with Service Bus which automatically moves messages to <queue>/$DeadLetterQueue after maxDeliveryCount is exhausted.
🚨
Gotcha: Sessions Cannot Be Enabled on an Existing Queue
Service Bus sessions must be enabled at queue or subscription creation time. You cannot enable sessions on an existing entity — not through the portal, not through ARM, not through the SDK. The property requiresSession is immutable after creation. If a question asks how to add session support to an existing Standard-tier queue, the correct answer is: delete the queue and recreate it with sessions enabled, then replay any lost messages. This is also why you should design session requirements upfront. Additionally, session-enabled queues/subscriptions require session-aware receivers (ServiceBusSessionReceiver or ServiceBusSessionProcessor), not regular receivers.

④ CLI / SDK Patterns

Storage Queue — Create and Use (Azure SDK for .NET)

// Create QueueClient — requires Storage Account connection string
var queueClient = new QueueClient(connectionString, "myqueue");
await queueClient.CreateIfNotExistsAsync();

// Enqueue a message (plain text or Base64-encoded binary)
await queueClient.SendMessageAsync("Hello from Storage Queue");

// Dequeue with visibility timeout (message becomes invisible for 30s)
QueueMessage[] messages = await queueClient.ReceiveMessagesAsync(
    maxMessages: 5,
    visibilityTimeout: TimeSpan.FromSeconds(30)  // default is 30s
);

foreach (var msg in messages)
{
    // Check DequeueCount for poison message detection
    if (msg.DequeueCount > 5)
    {
        // Manual poison message handling — enqueue to poison queue
        var poisonClient = new QueueClient(connectionString, "myqueue-poison");
        await poisonClient.CreateIfNotExistsAsync();
        await poisonClient.SendMessageAsync(msg.MessageText);
        await queueClient.DeleteMessageAsync(msg.MessageId, msg.PopReceipt);
        continue;
    }

    // Process message
    Console.WriteLine(msg.MessageText);

    // Delete the message explicitly after processing
    await queueClient.DeleteMessageAsync(msg.MessageId, msg.PopReceipt);
}

Service Bus — Send and Receive (Peek-Lock)

// Create ServiceBusClient — use connection string or DefaultAzureCredential
await using var client = new ServiceBusClient(connectionString);

// SEND — create sender for the queue
await using ServiceBusSender sender = client.CreateSender("myqueue");

var message = new ServiceBusMessage("Order placed for item 42")
{
    MessageId = Guid.NewGuid().ToString(),  // used for duplicate detection
    SessionId = "customer-001",             // required if sessions are enabled
    TimeToLive = TimeSpan.FromDays(2)
};
await sender.SendMessageAsync(message);

// RECEIVE — Peek-Lock mode (default)
await using ServiceBusReceiver receiver = client.CreateReceiver("myqueue",
    new ServiceBusReceiverOptions
    {
        ReceiveMode = ServiceBusReceiveMode.PeekLock  // safe default
        // ServiceBusReceiveMode.ReceiveAndDelete = destructive, no retry
    });

ServiceBusReceivedMessage received = await receiver.ReceiveMessageAsync(
    maxWaitTime: TimeSpan.FromSeconds(30));

if (received != null)
{
    try
    {
        // Process the message
        Console.WriteLine(received.Body.ToString());

        // Complete = permanently delete from queue
        await receiver.CompleteMessageAsync(received);
    }
    catch
    {
        // Abandon = release lock immediately, message re-enqueues
        await receiver.AbandonMessageAsync(received);
        // Or: DeadLetterMessageAsync() to force it to DLQ
    }
}

Service Bus — Processor (Recommended for Production)

// ServiceBusProcessor handles lock renewal automatically
await using var processor = client.CreateProcessor("myqueue",
    new ServiceBusProcessorOptions
    {
        MaxConcurrentCalls = 4,
        AutoCompleteMessages = false  // set false to control completion manually
    });

// Register handlers
processor.ProcessMessageAsync += async args =>
{
    string body = args.Message.Body.ToString();
    Console.WriteLine($"Processing: {body}");

    // Must call CompleteMessageAsync when AutoCompleteMessages = false
    await args.CompleteMessageAsync(args.Message);
};

processor.ProcessErrorAsync += args =>
{
    Console.Error.WriteLine(args.Exception.Message);
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
// ... keep alive ...
await processor.StopProcessingAsync();

Service Bus — Session-Aware Processing

// Session queues require ServiceBusSessionProcessor
await using var sessionProcessor = client.CreateSessionProcessor("myqueue-sessions",
    new ServiceBusSessionProcessorOptions
    {
        MaxConcurrentSessions = 8,        // how many sessions processed in parallel
        MaxConcurrentCallsPerSession = 1  // messages within one session are sequential
    });

sessionProcessor.ProcessMessageAsync += async args =>
{
    // args.SessionId is the session this message belongs to
    Console.WriteLine($"Session: {args.SessionId}, Body: {args.Message.Body}");

    // Store session state (e.g., order accumulator across messages)
    await args.SetSessionStateAsync(BinaryData.FromString("processed"));

    await args.CompleteMessageAsync(args.Message);
};

sessionProcessor.ProcessErrorAsync += args =>
{
    Console.Error.WriteLine(args.Exception);
    return Task.CompletedTask;
};

await sessionProcessor.StartProcessingAsync();

CLI — Create Service Bus Namespace and Queue

# Create Service Bus namespace (Standard tier)
az servicebus namespace create \
  --name myns \
  --resource-group myRG \
  --sku Standard        # Premium for 100 MB messages / VNet integration

# Create queue with sessions enabled (immutable after creation)
az servicebus queue create \
  --name myqueue \
  --namespace-name myns \
  --resource-group myRG \
  --enable-session true \            # cannot change after creation
  --max-delivery-count 5 \           # default is 10
  --lock-duration PT2M \             # ISO 8601: 2 minutes (default PT1M = 60s)
  --default-message-time-to-live P7D # ISO 8601: 7 days

# Create topic + subscription with SQL filter
az servicebus topic create --name mytopic --namespace-name myns --resource-group myRG

az servicebus topic subscription create \
  --name highpriority \
  --topic-name mytopic \
  --namespace-name myns \
  --resource-group myRG

az servicebus topic subscription rule create \
  --name PriorityRule \
  --subscription-name highpriority \
  --topic-name mytopic \
  --namespace-name myns \
  --resource-group myRG \
  --filter-sql-expression "Priority = 1"   # SQL filter on message properties
⑤ Wrong-Answer Traps
⚠️
Trap: "Service Bus guarantees strict FIFO ordering across all messages"
Service Bus guarantees FIFO ordering only within a session. Without sessions enabled, the queue delivers messages in roughly FIFO order but makes no formal guarantee — concurrent receivers, network retries, and redelivery after lock expiry can all disrupt ordering. The exam will present a scenario requiring strict ordered processing and list "use Service Bus Queue" as a distractor. The correct answer is always "use Service Bus Queue with sessions enabled, assigning the same SessionId to all related messages." Without sessions, even Service Bus is best-effort FIFO.
⚠️
Trap: "Storage Queue automatically moves failed messages to a dead-letter queue"
This is false. Storage Queue has no dead-letter queue, period. The queue only has a visibility timeout mechanism — if a message is not deleted within the visibility timeout, it reappears. If your code consistently fails to process it, the message keeps reappearing until its TTL (max 7 days) elapses and it is automatically deleted by the service. There is no automatic movement to any other queue. You must implement poison message handling yourself using DequeueCount. Answers referencing /$DeadLetterQueue or any automatic DLQ behavior for Storage Queue are wrong.
⚠️
Trap: "You can read the dead-letter queue the same way as the main queue"
The Service Bus dead-letter queue is a sub-queue with a fixed naming convention: <queue-name>/$DeadLetterQueue. To read from it you must explicitly target that path. In the SDK, use ServiceBusReceiver with the sub-queue option: client.CreateReceiver("myqueue", new ServiceBusReceiverOptions { SubQueue = SubQueue.DeadLetter }). Simply creating a receiver for "myqueue" will never see dead-lettered messages — they live in a separate sub-queue and accumulate there until you explicitly process or delete them. The DLQ itself also has a max size and can fill up.
⑥ Practice Q&A
1
Your application needs to send 500 GB of daily telemetry events through a queue. Each message is under 1 KB and no ordered processing or dead-letter handling is needed. Which queue solution minimizes cost and complexity?
▼
✓ Azure Storage Queue
Storage Queue is the correct answer.
  • Storage Queue supports up to 500 TB of storage — far exceeding Service Bus's 1–80 GB maximum queue size.
  • Messages are under 64 KB (the Storage Queue limit), so no size constraint is hit.
  • No ordered delivery or DLQ is required, so Service Bus's advanced features are unnecessary overhead.
  • Storage Queue is significantly cheaper at high volume — priced per operation on a storage account, not per messaging unit.
  • Service Bus would be wrong here because it cannot scale to 500 GB of queue backlog in a single queue.
2
A financial application processes payment messages and requires that all messages for the same customer are processed in strict FIFO order. Multiple concurrent worker processes consume the queue. What must you configure?
▼
✓ Service Bus Queue with sessions enabled, using customer ID as the SessionId
Sessions are required for FIFO ordering in Service Bus.
  • Sessions group related messages under a common SessionId. A single session is processed by one receiver at a time — guaranteeing in-order delivery within that session.
  • Setting SessionId = customerId on each message ensures all messages for a given customer go to the same session and are processed sequentially.
  • Multiple concurrent workers can each handle different sessions in parallel, so throughput is maintained.
  • Sessions must be enabled on the queue at creation (requiresSession = true). They cannot be enabled retroactively.
  • A standard Service Bus queue without sessions does not guarantee FIFO order when multiple concurrent consumers are reading.
3
A Service Bus message processing job takes between 2 and 4 minutes to complete. The queue's lock duration is set to the maximum of 5 minutes. What is the risk and what should you do?
▼
✓ Call RenewMessageLockAsync() periodically during processing
Lock expiry causes unintended redelivery even if processing eventually succeeds.
  • The max configurable lock duration is 5 minutes. If processing takes 4 minutes, you are within the limit — but barely. If any processing step is slow, the lock can expire before CompleteMessageAsync() is called.
  • When a lock expires, the message becomes visible again to other consumers. It will be picked up and processed a second time, incrementing DeliveryCount. This risks double-processing.
  • The correct solution is to call receiver.RenewMessageLockAsync(message) at intervals during long processing — for example, every 60 seconds. This resets the lock timer without acknowledging the message.
  • ServiceBusProcessor automatically renews locks while ProcessMessageAsync is executing, but only if the handler hasn't returned yet. For manual receivers, renewal must be explicit.
  • If you let the lock expire, calling CompleteMessageAsync() afterward will throw ServiceBusException with reason MessageLockLost.
4
An application reads messages from a Service Bus queue. After processing fails three times, the message is found in the dead-letter queue. A new consumer now needs to reprocess those dead-lettered messages. How should the consumer connect?
▼
✓ Create a receiver targeting the SubQueue.DeadLetter sub-queue
Dead-lettered messages live in a separate addressable sub-queue.
  • The DLQ path is <queue-name>/$DeadLetterQueue but in the SDK you address it via the SubQueue.DeadLetter option: client.CreateReceiver("myqueue", new ServiceBusReceiverOptions { SubQueue = SubQueue.DeadLetter }).
  • A receiver for "myqueue" alone will never see DLQ messages — they are in a distinct sub-queue.
  • DLQ messages include system properties like DeadLetterReason and DeadLetterErrorDescription set by Service Bus, which are useful for diagnosing why the message was dead-lettered (e.g., "MaxDeliveryCountExceeded", "TTLExpiredException").
  • To reprocess: receive the message from DLQ, fix the issue, then send it back to the main queue with sender.SendMessageAsync(). There is no built-in "resend to main queue" operation.
5
You have a Service Bus topic with three subscriptions: Orders, Shipping, and Audit. The Orders subscription should only receive messages where the custom property Priority equals 1. The Audit subscription should receive all messages. What filter configuration is correct?
▼
✓ SQL filter "Priority = 1" on Orders subscription; default TrueFilter on Audit subscription
Service Bus subscription filters control which messages each subscriber receives.
  • SQL filters use a subset of SQL-92 syntax applied to message properties and system properties (e.g., Priority = 1, sys.Label = 'order').
  • Correlation filters are more efficient for simple equality matches (matching CorrelationId, Label, To, etc.) and are recommended when SQL filter expressiveness isn't needed.
  • A newly created subscription gets a default TrueFilter which passes all messages. The Audit subscription should keep this default.
  • The Orders subscription should have its default rule replaced: remove the default TrueFilter and add a SQL filter rule with Priority = 1.
  • When a message is published to the topic, Service Bus evaluates it against each subscription's rules. A message CAN be delivered to multiple subscriptions simultaneously — topics are pub/sub, not competing consumer.
  • FalseFilter passes nothing; it's used to disable a subscription temporarily without deleting it.
6
A Storage Queue consumer reads a message but processing fails. The dequeue count on the message is now 6, and your threshold is 5. You have not yet implemented poison message handling. What happens to this message over the next 7 days?
▼
✓ The message reappears repeatedly after each visibility timeout until its TTL (7 days) expires and it is auto-deleted
Storage Queue has no automatic poison message removal — you must handle it yourself.
  • After each failed dequeue, the message becomes invisible for the visibility timeout period, then reappears.
  • Without custom poison message code checking DequeueCount, the message will continue cycling through the queue indefinitely.
  • After 7 days (Storage Queue max TTL), the Azure Storage service automatically deletes expired messages — this is the only automatic removal.
  • During those 7 days, every consumer that polls the queue could dequeue this message, fail, and re-enqueue it. This is resource-wasting and can block legitimate message processing if the queue has a low-concurrency consumer.
  • Correct implementation: check message.DequeueCount >= threshold after each dequeue. If exceeded: enqueue to myqueue-poison, then call DeleteMessageAsync(messageId, popReceipt) to remove from the original queue.
7
A developer uses ServiceBusReceiver in ReceiveAndDelete mode and their processing code throws an exception after receiving but before completing the business logic. What happens to the message?
▼
✓ The message is permanently lost — it was already deleted from the queue at receive time
ReceiveAndDelete provides at-most-once delivery — messages are deleted on receipt, not on completion.
  • In ReceiveAndDelete mode, Service Bus removes the message from the queue as soon as it is delivered to the consumer. There is no lock, no visibility timeout, and no retry mechanism.
  • If the application crashes or throws an exception after receiving, the message is gone. It will never appear in the DLQ because it was never subject to delivery count tracking — it was consumed.
  • Use ReceiveAndDelete only when message loss is acceptable and maximum throughput is the goal (e.g., telemetry ingestion where an occasional dropped message is tolerable).
  • Peek-Lock (the default) is the safe choice for any business-critical workflow: message is held until you explicitly call CompleteMessageAsync(), giving you at-least-once delivery semantics.
  • There is no way to recover a message lost in ReceiveAndDelete mode after an exception.
8
You need to send a single Service Bus message and simultaneously dequeue a message from another queue, in an all-or-nothing atomic operation. Both queues are in the same Service Bus namespace. What feature enables this?
▼
✓ Service Bus Transactions
Service Bus supports atomic transactions across multiple operations within the same namespace.
  • A Service Bus transaction groups multiple operations (send, complete, abandon, dead-letter, defer) into an atomic unit. Either all operations commit or all roll back.
  • Transactions are scoped to a single Service Bus namespace — you cannot span two different namespaces in one transaction.
  • In the SDK: use sender.CreateMessageBatchAsync() or leverage the transaction scope via ServiceBusClient with using var scope = new TransactionScope(TransactionScopeAsyncFlowOption.Enabled).
  • Common scenario: receive a work message, process it, send a result message to another queue, and complete the original message — all atomically. If the send fails, the complete is rolled back and the original message is retried.
  • Storage Queue does not support transactions across operations. This is a key differentiator that the exam tests.
