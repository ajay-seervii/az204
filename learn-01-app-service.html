Topic 01

M1 Explore Azure App Service

① Key Concepts

App Service Plans — The Foundation

An App Service Plan defines the region, number of VM instances, and VM size that all apps on the plan share. The plan is the billing unit — not individual apps. Multiple apps on one plan share the same compute resources.

Tier Category SKUs VM Type Key Capabilities Max Scale-Out
Free F1 Shared 60 min CPU/day, no custom domain, no SSL 1
Shared D1 Shared Custom domain, 240 min CPU/day, no SSL 1
Basic B1, B2, B3 Dedicated Custom domain + SSL, manual scale only, no slots, no autoscale 3
Standard S1, S2, S3 Dedicated Autoscale, 5 deployment slots, daily backups, traffic manager 10
Premium v3 P0v3–P3v3 Dedicated 20 slots, enhanced perf, VNet integration, zone redundancy 30
Isolated v2 I1v2–I6v2 Dedicated (ASE) App Service Environment, private VNet injection, 20 slots 100
Plan vs App distinction
When you scale a plan (e.g., from S1 to S2), every app on that plan gets the larger VM. When you scale out a plan (add instances), all apps distribute load across all instances. There is no per-app instance control within a shared plan.

WebJobs

WebJobs run background tasks within the same App Service plan as your web app — they consume the same CPU/memory resources. Two types:

  • Continuous WebJobs — start immediately and run in a loop; must include an infinite loop in code; can run on a single instance or all instances
  • Triggered WebJobs — run only when explicitly triggered (manually, or via a CRON schedule set in settings.job); always run on a single instance
WebJobs vs Azure Functions
The exam may ask when to prefer WebJobs over Functions. Prefer WebJobs when you need the code to run in the same process as the web app (access to HttpContext, same deployment unit), or when migrating legacy background tasks from an on-premises app. For new greenfield background processing, prefer Azure Functions.

Built-in Authentication (EasyAuth)

EasyAuth is a middleware layer injected before your application code by the App Service platform. It intercepts every incoming HTTP request and handles the entire OAuth/OIDC flow. Key behaviors:

  • Runs as a separate module in the same IIS/Nginx sandbox — zero code changes needed
  • Injects headers (X-MS-CLIENT-PRINCIPAL-NAME, X-MS-TOKEN-*) into requests your code receives
  • Supports: Azure AD (Microsoft), Facebook, Google, Twitter, any OIDC provider
  • Two actions when unauthenticated: Restrict access (HTTP 401/403) or Allow unauthenticated requests (your code decides)
  • Token store at /.auth/me — returns JSON with claims
② Exam Gotchas
Free/Shared Tiers Cannot Use VNet or Custom SSL
Free (F1) and Shared (D1) plans run on shared VMs that Microsoft owns — they are multi-tenant at the VM level. This means you cannot inject them into a VNet, cannot bind a custom SSL certificate, and cannot set IP restrictions. The exam presents scenarios like "you need to restrict inbound traffic to your app — which is the minimum tier?" The answer is Basic (B1), which runs on dedicated VMs and supports IP-based access restrictions and custom SSL via SNI or IP-based binding.
Basic Tier: Dedicated VMs but NO Autoscale and NO Slots
Basic (B1–B3) is the first dedicated-VM tier, so it supports custom domains, SSL, and manual instance count increases. However, Basic does NOT support autoscale rules and has zero deployment slots. The exam exploits this boundary frequently: a scenario will describe an app that needs SSL and the ability to auto-scale — the answer requires Standard (S1+), not Basic. Students who know Basic is dedicated often assume it has all Standard features.
Contributor Role Does NOT Grant Secret Access
The Contributor role on an App Service resource gives full management-plane access (create, configure, delete the resource) but zero data-plane access to secrets stored in Azure Key Vault. A common trap question: "A developer has Contributor on the App Service but cannot read the Key Vault secrets the app uses. What role should you assign?" The answer is Key Vault Secrets User (data plane), assigned on the Key Vault resource. Contributor on the Key Vault itself would also be wrong — that still doesn't grant data-plane access in the new RBAC model.

④ CLI / SDK Patterns
# Create an App Service Plan (Standard tier, Linux, 2 workers)
az appservice plan create \
  --name MyPlan \
  --resource-group MyRG \
  --sku S1 \                    # S1=Standard; B1=Basic; P1v3=Premium v3
  --is-linux \                  # omit for Windows; cannot change after creation
  --number-of-workers 2

# Create a web app — RUNTIME uses pipe separator (NOT colon or slash)
az webapp create \
  --name MyWebApp \
  --resource-group MyRG \
  --plan MyPlan \
  --runtime "DOTNET|8.0"        # common runtimes: "NODE|20-lts", "PYTHON|3.11", "JAVA|17-java17"

# List available runtimes for Linux
az webapp list-runtimes --os-type linux

# Show current plan details
az appservice plan show \
  --name MyPlan \
  --resource-group MyRG \
  --query "{sku:sku.name, workers:sku.capacity}"

# Scale up the plan (change SKU)
az appservice plan update \
  --name MyPlan \
  --resource-group MyRG \
  --sku P1v3                    # upgrade from S1 to Premium v3

# Scale out manually (set fixed instance count)
az appservice plan update \
  --name MyPlan \
  --resource-group MyRG \
  --number-of-workers 5         # sets count; requires Standard+ for >3

# Create a continuous WebJob (must upload as zip)
az webapp webjob continuous add \
  --webjob-name MyWorker \
  --webjob-zip worker.zip \
  --resource-group MyRG \
  --name MyWebApp
⑤ Wrong-Answer Traps
Trap: "Use Free tier for dev/test with custom domain"
The Free (F1) tier does not support custom domains. The minimum tier for a custom domain is Shared (D1). For a custom domain plus SSL, you need Basic (B1). The exam places these options close together to test whether you know the exact tier boundary.
Trap: "Scaling a single app on a shared plan only affects that app"
This is false. When you scale up or out an App Service Plan, all apps on that plan are affected — they all move to the new VM size or gain the new instances. There is no per-app isolation within a single plan. If you want to scale one app independently, it must be on its own plan.
Trap: "EasyAuth requires changes to application code"
EasyAuth is a platform-level middleware — it requires zero application code changes. It intercepts requests before they reach your app and injects claims as HTTP headers. Your app can read those headers if needed, but doesn't have to. If a question asks for the approach with the least code changes for authentication, EasyAuth is the answer over MSAL-based custom authentication.
⑥ Practice Q&A
1
A company runs a .NET web application on App Service Standard S1. They want to add a background process that reads messages from a queue and must restart automatically if it crashes. The background process must share the same deployment pipeline as the web app. What should you use?
▼
✓ Correct Answer: Continuous WebJob
A Continuous WebJob is the right choice here because:
  • It runs within the same App Service as the web app, sharing the same deployment and plan.
  • Continuous WebJobs run as an infinite loop and are automatically restarted by the platform if the process exits unexpectedly.
  • They share the same zip/package deployment, satisfying the "same pipeline" requirement.
  • Azure Functions would be correct for greenfield but requires a separate Functions app and infrastructure.
  • A Triggered WebJob would not restart automatically — it runs once per trigger and stops.
2
Your App Service app on a Basic B2 plan is experiencing traffic spikes. You configure an autoscale rule to add 2 instances when CPU exceeds 70%. After deploying, the rule never triggers despite CPU regularly hitting 85%. What is the most likely reason?
▼
✓ Correct Answer: Basic tier does not support autoscale
The Basic tier (B1–B3) does not support autoscale rules.
  • Autoscale is only available on Standard (S1+), Premium, and Isolated tiers.
  • The Azure portal may allow you to configure the autoscale setting, but it will not take effect on a Basic plan.
  • To resolve: upgrade the plan to at least Standard S1, then the autoscale rule will fire.
  • On Basic, you can only manually set a fixed instance count (up to 3 instances).
3
An application is deployed on App Service Free tier (F1). Users report the app becomes unresponsive for the rest of the day starting around 3 PM. No errors appear in Application Insights. What is the most likely cause?
▼
✓ Correct Answer: The 60-minute daily CPU quota has been exhausted
Free (F1) apps are limited to 60 minutes of CPU time per day.
  • Once the quota is exhausted, the app is throttled — HTTP requests receive a 403 or the app becomes extremely slow; it is not stopped outright.
  • The quota resets at midnight UTC each day, which is why users can access it in the morning.
  • This does not generate application-level errors visible in Application Insights; it is a platform-level throttle.
  • The fix is to upgrade to at least Shared (D1) for 240 min/day, or Basic (B1) for unlimited CPU on a dedicated VM.
4
A developer with the Contributor role on the App Service resource reports they cannot retrieve secrets from the Azure Key Vault that the app references. The app itself retrieves secrets successfully at runtime via Managed Identity. What is the minimum role to grant the developer read access to those secrets?
▼
✓ Correct Answer: Key Vault Secrets User (assigned on the Key Vault)
Key Vault uses a separate data-plane RBAC model.
  • Contributor on the App Service resource is a management-plane role — it controls the App Service, not the Key Vault or its contents.
  • Even Contributor on the Key Vault itself does not grant secret read access in the RBAC permission model.
  • Key Vault Secrets User is the built-in role granting get and list on secrets — this is assigned at the Key Vault scope.
  • Key Vault Secrets Officer would also work but grants create/update/delete — too permissive for read-only.
  • The Managed Identity the app uses has its own role assignment separate from the developer's account.
5
You need to deploy a web application that: (1) requires a custom domain with SSL, (2) must handle traffic spikes automatically, and (3) needs two non-production deployment slots. What is the minimum App Service Plan tier that satisfies all three requirements?
▼
✓ Correct Answer: Standard S1
Mapping each requirement to minimum tier:
  • Custom domain + SSL — minimum is Basic (B1)
  • Autoscale — minimum is Standard (S1)
  • Deployment slots — minimum is Standard (S1); Standard provides up to 5 slots including production, so 4 non-prod slots — two slots is achievable
  • The binding constraint is autoscale + slots, both requiring Standard S1.
  • Basic would satisfy SSL + custom domain but fails on autoscale and has zero slots.
6
You deploy a web app to an App Service using the CLI command: az webapp create --runtime "DOTNET:8.0". The command fails. What is the correct syntax for the runtime flag?
▼
✓ Correct Answer: Use a pipe separator — "DOTNET|8.0"
The App Service CLI runtime format uses a pipe (|) as the separator, not a colon.
  • Correct: --runtime "DOTNET|8.0"
  • Other correct examples: "NODE|20-lts", "PYTHON|3.11", "JAVA|17-java17"
  • You can retrieve the exact valid strings with az webapp list-runtimes --os-type linux
  • This is a classic exam trap — the colon format looks logical but is incorrect and will cause the CLI command to fail with a validation error.
7
Your company requires that all App Service apps be isolated within a private VNet for compliance reasons, with no public internet exposure. Which App Service tier satisfies this requirement?
▼
✓ Correct Answer: Isolated v2 (App Service Environment)
Only the Isolated tier runs inside an App Service Environment (ASE) fully injected into a VNet.
  • Standard and Premium tiers support VNet Integration for outbound traffic, but inbound traffic still arrives over the public internet (though you can add private endpoints).
  • Isolated (I1v2–I6v2) deploys into a dedicated ASE within your VNet — both inbound and outbound are fully private by default.
  • The ASE has its own dedicated inbound IP that is a private IP in your VNet.
  • ASE v3 is the current generation; it eliminates the stamp fee that existed in ASE v2.
  • Scale-out up to 100 instances makes Isolated suitable for large enterprise workloads requiring compliance isolation.

M2 Configure Web App Settings

① Key Concepts

App Settings and Connection Strings

App settings are name-value pairs that App Service injects as environment variables at runtime. For .NET apps they also override matching keys in appsettings.json and web.config. For connection strings, App Service prefixes the values with a type identifier when exposing them as environment variables.

Connection String Type Env Var Prefix Example Env Var
SQLAzure SQLAZURECONNSTR_ SQLAZURECONNSTR_MyDb
SQLServer SQLCONNSTR_ SQLCONNSTR_MyDb
MySQL MYSQLCONNSTR_ MYSQLCONNSTR_MyDb
PostgreSQL POSTGRESQLCONNSTR_ POSTGRESQLCONNSTR_MyDb
Custom CUSTOMCONNSTR_ CUSTOMCONNSTR_MyDb

Slot-Sticky (Deployment Slot) Settings

By default, app settings travel with the code during a swap — they move to the other slot. Marking a setting as slot-sticky (deployment slot setting) causes it to stay bound to the slot it's defined on, regardless of swaps.

When to use slot-sticky settings
Use slot-sticky for environment-specific values: production database connection string, production API keys, feature flags that differ per environment. If DB_CONNECTION_STRING is slot-sticky on production, after a code swap the new code in production still connects to the production database — not the staging database that traveled with the code.

Critical Platform Environment Variables

Environment Variable Value Effect
WEBSITE_RUN_FROM_PACKAGE 1 or URL Mounts zip as a read-only filesystem; fastest cold starts; /home/site/wwwroot becomes read-only
SCM_DO_BUILD_DURING_DEPLOYMENT true Enables Oryx build system — runs npm install, pip install, etc. on the server during deployment
WEBSITE_SWAP_WARMUP_PING_PATH /health (custom path) Custom path the platform pings to warm up the slot before completing a swap
WEBSITE_SWAP_WARMUP_PING_WAIT seconds (default 90) Maximum time to wait for warmup ping to succeed before aborting swap
WEBSITES_ENABLE_APP_SERVICE_STORAGE false For containers: disables persistent /home storage (faster startup, no shared state)
WEBSITE_NODE_DEFAULT_VERSION e.g. ~18 Pins Node.js version on Windows App Service plans
② Exam Gotchas
App Settings Override appsettings.json — the App Service Value Always Wins
If your appsettings.json defines "ConnectionStrings:MyDb": "dev-connection-string" and you also define a MyDb connection string in App Service configuration, the App Service value completely overrides the file. The file-based value is never used at runtime. This is intentional — it allows environment-specific overrides without code changes — but it catches developers who add a new key to appsettings.json and wonder why it isn't taking effect in production (because an identically named App Service setting is shadowing it).
WEBSITE_RUN_FROM_PACKAGE=1 Makes wwwroot Read-Only
Setting WEBSITE_RUN_FROM_PACKAGE=1 mounts the deployment zip as a read-only virtual filesystem at /home/site/wwwroot. Any application code that writes files to that path at runtime (log files, temp caches, generated assets) will fail with access denied. The fix is to write to /home/LogFiles or /tmp instead, or disable run-from-package if write access to wwwroot is genuinely required. The exam presents this as a scenario where a previously-working app suddenly gets file-write failures after a deployment change.
Non-Sticky Settings Travel with Code — Wrong DB After Swap
A classic exam scenario: staging has DB_HOST=staging-db (non-sticky). Production has DB_HOST=prod-db (also non-sticky). After a swap, the code that was in staging is now in production — but it brought its DB_HOST=staging-db setting with it. Production is now pointing at the staging database. The fix: mark DB_HOST as slot-sticky on both slots. Then DB_HOST stays bound to its slot regardless of which code is running there.

④ CLI / SDK Patterns
# Set app settings (multiple key=value pairs in one command)
az webapp config appsettings set \
  --name MyWebApp \
  --resource-group MyRG \
  --settings \
    ENV=production \
    API_URL=https://api.example.com \
    "DB_CONN=Server=myserver;Database=mydb"   # quote values with spaces/semicolons

# Mark a setting as slot-sticky (stays with the slot, not the code)
az webapp config appsettings set \
  --name MyWebApp \
  --resource-group MyRG \
  --slot staging \
  --slot-settings DB_CONN=staging-connection-string   # --slot-settings flag marks sticky

# Set a connection string with type
az webapp config connection-string set \
  --name MyWebApp \
  --resource-group MyRG \
  --connection-string-type SQLAzure \
  --settings MyDb="Server=tcp:myserver.database.windows.net;..."

# Make a connection string slot-sticky
az webapp config connection-string set \
  --name MyWebApp \
  --resource-group MyRG \
  --connection-string-type SQLAzure \
  --slot-settings MyDb="Server=tcp:myserver.database.windows.net;..."  # slot-sticky

# Reference a Key Vault secret as an app setting
az webapp config appsettings set \
  --name MyWebApp \
  --resource-group MyRG \
  --settings \
    "MY_SECRET=@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/MySecret/)"

# Enable run-from-package
az webapp config appsettings set \
  --name MyWebApp \
  --resource-group MyRG \
  --settings WEBSITE_RUN_FROM_PACKAGE=1

# View current app settings
az webapp config appsettings list \
  --name MyWebApp \
  --resource-group MyRG \
  --output table
⑤ Wrong-Answer Traps
Trap: "To keep different DB connections per slot, set them as regular (non-sticky) app settings"
This is the opposite of correct. Non-sticky settings travel with the code on swap, so after a swap, your production slot would have the staging DB connection. You must mark environment-specific settings as slot-sticky so they remain bound to their slot regardless of which deployment is running there.
Trap: "SCM_DO_BUILD_DURING_DEPLOYMENT=true runs a build on every request"
This setting runs the Oryx build system once, during deployment — not on every request. It's used for Node.js, Python, Ruby apps where you want the server to run npm install or pip install as part of the deployment process rather than pre-building locally. It has no effect on runtime request handling.
Trap: "Key Vault references work with any identity type — no special configuration needed"
Key Vault references in App Service require the app to have a system-assigned or user-assigned Managed Identity enabled, and that identity must have the Key Vault Secrets User role (or access policy with get permission) on the Key Vault. Without this, the app setting will show as a failed reference and the environment variable will be empty or contain an error string at runtime.
⑥ Practice Q&A
1
A web app reads its database connection string from appsettings.json using the key ConnectionStrings:MainDb. A junior developer adds a connection string named MainDb of type SQLAzure to App Service configuration pointing to the wrong database. At runtime, which connection string does the app use?
▼
✓ Correct Answer: The App Service connection string (wrong database)
App Service settings always override appsettings.json for the same key.
  • For .NET apps, connection strings set in App Service are injected into the ConnectionStrings section of the configuration provider, overriding values from the file.
  • The App Service value is exposed as environment variable SQLAZURECONNSTR_MainDb, and the .NET configuration provider picks this up with higher priority than the JSON file.
  • To fix: update the App Service connection string value to the correct database, or remove it to fall back to the file.
2
After setting WEBSITE_RUN_FROM_PACKAGE=1 and redeploying, your app crashes with UnauthorizedAccessException when writing to /home/site/wwwroot/cache. What is the cause and fix?
▼
✓ Correct Answer: wwwroot is read-only when run-from-package is enabled; write to /tmp instead
WEBSITE_RUN_FROM_PACKAGE mounts the deployment as a read-only zip archive.
  • When this setting is active, /home/site/wwwroot is a read-only mount — writes fail with access denied.
  • Fix: change the cache directory to /tmp/cache (ephemeral, per-instance) or /home/LogFiles/cache (persistent, shared across instances).
  • If the app genuinely needs to write to wwwroot (e.g., a CMS generating static files), remove WEBSITE_RUN_FROM_PACKAGE and use standard zip deployment instead.
  • The benefit of run-from-package is faster cold starts and atomic deployments — worth keeping if you can eliminate the write requirement.
3
You have a production slot with DB_CONN=prod-db (slot-sticky) and a staging slot with DB_CONN=staging-db (slot-sticky). You deploy new code to staging and perform a slot swap. What does the production slot's DB_CONN value equal after the swap?
▼
✓ Correct Answer: prod-db (unchanged)
Slot-sticky settings remain bound to their slot through any number of swaps.
  • After the swap, the new code (previously in staging) is now running in the production slot.
  • Because DB_CONN is slot-sticky on production, it stays as prod-db — the production slot always has the production DB connection, regardless of which code revision is running.
  • Similarly, the staging slot still has DB_CONN=staging-db — now running the old production code, but pointing at staging DB.
  • This is the correct pattern: slot-sticky ensures environment-specific settings are environment-specific regardless of code swaps.
4
A Python Flask app is deployed via zip deploy. The app fails to start because modules listed in requirements.txt are not installed. No build step is configured. What is the simplest fix using an App Service setting?
▼
✓ Correct Answer: Set SCM_DO_BUILD_DURING_DEPLOYMENT=true
This enables the Oryx build system, which runs pip install -r requirements.txt server-side during deployment.
  • Without this setting, App Service simply unpacks the zip — it does not install Python dependencies.
  • With SCM_DO_BUILD_DURING_DEPLOYMENT=true, the Kudu build pipeline invokes Oryx, which detects the Python app, creates a virtual environment, and installs all packages.
  • This is the recommended approach for Linux App Service apps where you prefer not to include the site-packages directory in your deployment artifact.
  • Alternatively, you can install dependencies locally, include the packages in the zip, and not use Oryx — but this increases zip size significantly.
5
Your App Service app uses a Key Vault reference for an app setting: @Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/ApiKey/). The app setting shows "Key Vault reference is invalid" in the portal. Managed Identity is enabled on the app. What is the most likely missing configuration?
▼
✓ Correct Answer: The Managed Identity has not been granted Key Vault Secrets User role on the Key Vault
Enabling Managed Identity is necessary but not sufficient — the identity must also be authorized on the Key Vault.
  • The App Service uses its Managed Identity to authenticate to Key Vault at app startup to resolve the reference.
  • Without a role assignment (Key Vault Secrets User) or an access policy granting get on the Key Vault, the authentication succeeds but authorization fails — hence "reference is invalid."
  • Check: Azure Portal → Key Vault → Access control (IAM) → verify the App Service identity has Key Vault Secrets User.
  • Also verify the Key Vault is in the same tenant and the firewall isn't blocking the App Service's outbound IP.
6
Which environment variable format correctly exposes a connection string named OrdersDb of type Custom to the application as an environment variable?
▼
✓ Correct Answer: CUSTOMCONNSTR_OrdersDb
App Service prefixes connection string environment variables based on their type.
  • Custom type → CUSTOMCONNSTR_OrdersDb
  • SQLAzure → SQLAZURECONNSTR_OrdersDb
  • SQLServer → SQLCONNSTR_OrdersDb
  • MySQL → MYSQLCONNSTR_OrdersDb
  • Your application reads this via Environment.GetEnvironmentVariable("CUSTOMCONNSTR_OrdersDb") or through the .NET configuration provider as ConnectionStrings:OrdersDb.

M3 Scale Apps in Azure App Service

① Key Concepts

Scale Up vs Scale Out

Operation What Changes How Tier Requirement
Scale Up VM size (CPU, RAM, storage per instance) Change the plan SKU (S1 → S2 → S3) Any tier
Manual Scale Out Number of VM instances (fixed count) Set instance count in plan settings Basic+ (max 3 on Basic, 10 on Standard)
Autoscale Instance count driven by rules/metrics Configure autoscale conditions and rules Standard+ (not available on Basic)

Autoscale Architecture

Autoscale in App Service uses Azure Monitor autoscale — the same engine used across VMs, VMSS, and other resources. Key components:

  • Scale condition — a named set of rules that activates under defined conditions (default, or schedule-based)
  • Scale rule — a single metric threshold that triggers a scale-out or scale-in action
  • Observation window (time grain + time aggregation) — how long the metric must sustain the threshold before triggering
  • Cooldown period — minimum time between consecutive scale actions to prevent thrashing
  • Instance limits — minimum, maximum, and default instance count per scale condition
Autoscale does not trigger instantly on threshold breach
The metric must sustain the threshold for the entire observation window before autoscale fires. A 10-minute window means CPU must be above your threshold for a continuous 10-minute period. A 5-minute spike during a 10-minute window will NOT trigger scale-out. This is by design to prevent over-scaling on transient spikes.

Common Scale Metrics

Metric Source Use Case
CPU Percentage App Service Plan General compute-bound scaling; most common
Memory Percentage App Service Plan Memory-intensive workloads (caching, large datasets)
Http Queue Length App Service Plan Request backlog; scale before CPU is saturated
Data In / Data Out App Service Plan Network-bound apps (file serving, streaming)
Custom metric Application Insights Business metrics (orders/min, queue depth); requires App Insights integration
② Exam Gotchas
The Observation Window Trap — Spike Duration vs Window Duration
Classic exam scenario: "CPU spikes to 85% for 5 minutes. The autoscale rule has a threshold of 75% CPU with a 10-minute observation window. Does autoscale trigger?" Answer: No. The spike must sustain the threshold for the full 10-minute window. A 5-minute spike followed by a return to normal resets the window. The exam will sometimes say "the rule is correctly configured" and ask why it didn't fire — the answer is always the observation window vs spike duration mismatch. To catch shorter spikes, reduce the observation window (minimum: 1 minute).
Scaling the Plan Scales ALL Apps on It
An App Service Plan is a shared compute resource. When you scale out the plan (add instances), every app hosted on that plan runs on all instances — not just the app you configured autoscale for. This means a single bursty app on a shared plan can trigger scale-out that benefits all apps. Conversely, all apps incur the cost of the extra instances. For cost isolation, put high-traffic apps on their own dedicated plan.
Cooldown Prevents Back-to-Back Scale Actions
After a scale-out fires, the cooldown period (default 5 minutes) must elapse before another scale action (in or out) can trigger. The exam presents a scenario: traffic spikes for 3 minutes, drops, spikes again 2 minutes later. If cooldown is 5 minutes and the first scale-out already fired, the second spike won't trigger another scale-out even if CPU is above threshold — the cooldown hasn't expired. The platform logs the cooldown as a "scale action suppressed" event in the Activity Log.

④ CLI / SDK Patterns
# Create an autoscale profile for an App Service Plan
az monitor autoscale create \
  --resource-group MyRG \
  --resource MyPlan \
  --resource-type Microsoft.Web/serverfarms \
  --name MyAutoscale \
  --min-count 2 \              # minimum instance count (never scale below this)
  --max-count 10 \             # maximum instance count (plan limit is 10 for Standard)
  --count 2                    # default instance count when no rules trigger

# Add a scale-OUT rule: add 2 instances when CPU > 70% for 5 minutes
az monitor autoscale rule create \
  --resource-group MyRG \
  --autoscale-name MyAutoscale \
  --condition "Percentage CPU > 70 avg 5m" \  # metric, operator, threshold, aggregation, window
  --scale out 2 \                              # direction + instance delta
  --cooldown 5                                 # minutes before next scale action allowed

# Add a scale-IN rule: remove 1 instance when CPU < 30% for 10 minutes
az monitor autoscale rule create \
  --resource-group MyRG \
  --autoscale-name MyAutoscale \
  --condition "Percentage CPU < 30 avg 10m" \
  --scale in 1 \
  --cooldown 10                 # longer cooldown for scale-in to prevent flapping

# Add a schedule-based profile (weekday business hours)
az monitor autoscale profile create \
  --resource-group MyRG \
  --autoscale-name MyAutoscale \
  --name BusinessHours \
  --timezone "UTC" \
  --start 2024-01-01T08:00:00 \
  --end 2024-01-01T18:00:00 \
  --recurrence day monday tuesday wednesday thursday friday \
  --min-count 4 \               # keep 4 instances during business hours
  --max-count 10 \
  --count 4

# Manual scale (set fixed count on Basic tier)
az appservice plan update \
  --name MyBasicPlan \
  --resource-group MyRG \
  --number-of-workers 3         # max on Basic tier

# View current autoscale settings
az monitor autoscale show \
  --resource-group MyRG \
  --name MyAutoscale
⑤ Wrong-Answer Traps
Trap: "You can configure autoscale for a specific app within a shared plan"
Autoscale operates at the App Service Plan level, not the individual app level. There is no per-app autoscale within a plan — when you configure autoscale, you configure it for the plan, and the scaling affects all apps on that plan equally. To scale a specific app independently, it must be on its own dedicated App Service Plan.
Trap: "Setting max-count to 50 on a Standard S1 plan allows up to 50 instances"
The plan SKU hard limit overrides the autoscale max setting. Standard S1 has a hard ceiling of 10 instances. Even if you configure the autoscale max at 50, the plan will never exceed 10. To get more instances, you must upgrade the plan to Premium v3 (max 30) or Isolated v2 (max 100).
Trap: "A shorter cooldown period improves response to traffic spikes"
A very short cooldown (e.g., 1 minute) allows rapid successive scale-out actions, which sounds better for spikes. However, it also means rapid scale-in once the spike ends, which can cause thrashing — constantly adding and removing instances. The recommendation is to have a longer scale-in cooldown (15–30 minutes) and a shorter scale-out cooldown (5 minutes) to respond quickly to demand but scale in conservatively to avoid repeated churn.
⑥ Practice Q&A
1
An autoscale rule is configured: scale out when CPU > 75%, observation window = 10 minutes, cooldown = 5 minutes. CPU rises to 80% at 2:00 PM and drops to 65% at 2:07 PM. Does autoscale trigger a scale-out?
▼
✓ Correct Answer: No — the threshold was not sustained for the full 10-minute window
The 10-minute observation window requires continuous threshold breach for 10 minutes.
  • CPU was above 75% from 2:00 PM to 2:07 PM — only 7 minutes.
  • When CPU dropped to 65% at 2:07 PM, the window resets. The threshold was not sustained for the full 10 minutes.
  • To catch this spike, reduce the observation window to 5 minutes or less.
  • Alternatively, use a pre-emptive scale rule based on Http Queue Length, which typically rises before CPU saturates.
2
You have two apps on the same Standard S1 plan. App A handles batch processing (high CPU). App B is a low-traffic API. You configure autoscale on the plan to trigger when App A's CPU exceeds 70%. After autoscale fires, what happens to App B?
▼
✓ Correct Answer: App B also runs on the new instances — both apps scale together
Scaling the plan affects all apps hosted on it.
  • When the plan adds instances, both App A and App B run across all instances.
  • App B incurs no cost on its own — it shares the plan — but it does gain extra capacity it doesn't need.
  • For App A's scaling to not affect App B's resource allocation, move App A to its own dedicated plan.
  • Alternatively, App B on a separate plan and App A on the batch plan keeps them independently scaled and billed.
3
A Standard S1 autoscale rule fires and adds instances at 3:00 PM (cooldown = 5 min). CPU spikes again at 3:03 PM above the threshold. What happens?
▼
✓ Correct Answer: The second scale-out is suppressed — cooldown has not elapsed
The cooldown period blocks scale actions until it expires.
  • The first scale-out fired at 3:00 PM. The 5-minute cooldown expires at 3:05 PM.
  • At 3:03 PM (only 3 minutes later), the platform detects the threshold is breached but the cooldown is still active — the scale action is suppressed.
  • If CPU is still above threshold at 3:05 PM (after cooldown), a new scale-out can fire.
  • To handle rapid successive spikes, reduce cooldown to 1–2 minutes (accepting some thrash risk) or increase instance delta (scale out 3 instead of 1 per action).
4
Your app on a Standard S2 plan needs to handle up to 25 concurrent instances during peak load. What should you do?
▼
✓ Correct Answer: Upgrade the plan to Premium v3 (P1v3 or higher)
Standard tier has a hard maximum of 10 instances regardless of autoscale configuration.
  • To reach 25 instances, you must upgrade to Premium v3 (max 30 instances) or Isolated v2 (max 100).
  • Premium v3 P0v3 is the smallest and most cost-effective Premium v3 SKU.
  • You cannot override the plan SKU instance ceiling via autoscale max-count settings.
  • After upgrading the plan, update the autoscale max-count to 25 and the platform will scale to that level.
5
You want your App Service app to scale based on the number of pending orders in a Service Bus queue. Which metric type enables this?
▼
✓ Correct Answer: Custom metric via Application Insights (or Azure Monitor metric from Service Bus)
Azure Monitor autoscale supports scaling on custom metrics from external resources.
  • Service Bus exposes metrics like Active Messages (queue depth) as Azure Monitor metrics — you can target this metric directly in an autoscale rule using the "metric resource" setting.
  • In the autoscale rule, set the metric resource to your Service Bus namespace and metric name to ActiveMessages.
  • You can also emit custom metrics from your app to Application Insights and scale on those values.
  • This pattern decouples scaling from CPU and directly reflects business demand (queue backlog).
6
You need predictable instance counts of 8 during business hours (9–5 weekdays) and 2 outside those hours. Autoscale metric-based rules sometimes keep too many or too few instances at the boundary. What should you add?
▼
✓ Correct Answer: Add schedule-based autoscale profiles for business hours and off-hours
Schedule-based profiles override metric-based rules during specified time windows.
  • Create a "Business Hours" profile: recurrence Monday–Friday, 9:00–17:00, min=8, max=10, default=8.
  • Create an "Off Hours" profile: recurrence all days, 17:00–9:00, min=2, max=4, default=2.
  • The default profile (metric-based) fires at all other times.
  • Schedule profiles guarantee the instance count at profile start — they don't wait for metric thresholds to trigger scaling.
  • This is the recommended pattern for workloads with known diurnal traffic patterns.

M4 Deployment Slots

① Key Concepts

What Are Deployment Slots?

Deployment slots are live, named instances of your web app within the same App Service Plan. Each slot has its own hostname (myapp-staging.azurewebsites.net), its own configuration, and its own running code. Slots share the same plan resources (compute, memory).

Slots are not separate apps
A slot is not billed as an additional app — it runs within your existing plan allocation. However, each slot does consume VM instances from your plan. Running 3 instances on production + 3 instances on staging = 6 instance-hours of billing, even though it's one app with two slots.

The Swap Operation

A slot swap is an atomic exchange of the routing rules between two slots. The platform:

  1. Applies the target slot's (production) configuration to the source slot (staging) instances
  2. Pings the warmup path on the staging instances (WEBSITE_SWAP_WARMUP_PING_PATH)
  3. Once all staging instances respond with HTTP 2xx (or timeout expires), switches the routing
  4. Applies the old production configuration back to what is now the staging slot
Setting Type Behavior During Swap Example
Slot-sticky setting Stays with the slot — does NOT move DB_CONN=prod-db stays in production slot regardless of which code runs there
Non-sticky setting Travels with the code to the new slot FEATURE_FLAG=true set on staging will be in production after swap
Custom domain binding Stays with production slot (not swapped) Production always has myapp.com regardless of swap
TLS/SSL settings Stays with production slot (not swapped) SSL certificate binding remains on production

Traffic Routing (A/B Testing)

You can route a percentage of production traffic to a non-production slot. This is set per-slot in the portal or via CLI. Users assigned to the slot receive a cookie (x-ms-routing-name) that pins them to that slot for subsequent requests. This allows gradual rollout without a full swap.

Auto-Swap

Auto-swap automatically swaps a slot into production when a new deployment is pushed to it. Configured per-slot. Useful for CD pipelines where every push to staging should go live automatically after warmup. Not available on Linux App Service — Linux slots do not support auto-swap.

② Exam Gotchas
After a Swap, the Staging Slot Has the OLD Production Code
This is the most-tested deployment slot concept. After you swap staging → production: staging now contains what was in production, and production contains what was in staging. To rollback a bad deployment, you do another swap — you swap production back to staging. No re-deployment needed. The exam asks: "You discover a bug after swapping staging to production. How do you roll back?" The answer is: perform another swap (production ↔ staging). The old code is still warm in the staging slot, so the rollback is instant.
Minimum Tier for Slots Is Standard — NOT Premium
Many candidates assume deployment slots require Premium tier because slots are an "advanced" feature. The minimum tier is actually Standard (S1). Standard provides up to 5 slots total (4 non-production). Premium v3 provides up to 20 slots. Basic provides zero deployment slots. A question asking for the minimum tier to enable a staging slot has the answer Standard, not Premium.
Non-Sticky Connection String After Swap Causes Wrong Database
Scenario: production has DB=prod-db (non-sticky) and staging has DB=staging-db (non-sticky). After swap, production now has DB=staging-db — it's pointing at the staging database in the production environment. This is a data integrity disaster. The fix is to mark the DB connection string as slot-sticky on both slots. With slot-sticky settings, the production slot always uses DB=prod-db and the staging slot always uses DB=staging-db, regardless of which code revision is active. The exam presents this exact scenario and asks "why is production connecting to the wrong database after a slot swap?"

④ CLI / SDK Patterns
# Create a deployment slot named "staging"
az webapp deployment slot create \
  --name MyWebApp \
  --resource-group MyRG \
  --slot staging               # slot name; hostname = mywebapp-staging.azurewebsites.net

# Deploy code to a specific slot (via zip deploy)
az webapp deployment source config-zip \
  --name MyWebApp \
  --resource-group MyRG \
  --slot staging \             # deploy to staging, not production
  --src ./publish.zip

# Swap staging slot into production
az webapp deployment slot swap \
  --name MyWebApp \
  --resource-group MyRG \
  --slot staging \             # source slot
  --target-slot production     # destination (default is "production" if omitted)

# Preview swap (apply production config to staging without routing traffic)
az webapp deployment slot swap \
  --name MyWebApp \
  --resource-group MyRG \
  --slot staging \
  --action preview             # "preview" | "swap" | "reset"

# Configure traffic routing — send 20% of traffic to staging slot
az webapp traffic-routing set \
  --name MyWebApp \
  --resource-group MyRG \
  --distribution staging=20   # 20% to staging, remaining 80% goes to production automatically

# Clear traffic routing (all traffic back to production)
az webapp traffic-routing clear \
  --name MyWebApp \
  --resource-group MyRG

# Set a slot setting as sticky on the staging slot
az webapp config appsettings set \
  --name MyWebApp \
  --resource-group MyRG \
  --slot staging \
  --slot-settings DB_CONN=staging-connection-string   # sticky — won't swap to production

# Configure auto-swap (Windows only)
az webapp deployment slot auto-swap \
  --name MyWebApp \
  --resource-group MyRG \
  --slot staging \
  --auto-swap-slot production  # swap to production automatically on deployment

# List all slots
az webapp deployment slot list \
  --name MyWebApp \
  --resource-group MyRG \
  --output table
⑤ Wrong-Answer Traps
Trap: "To roll back after a bad swap, redeploy the last good build to production"
Redeployment is slower and unnecessary. The old production code is still running warm in the staging slot immediately after the swap. The correct rollback is to swap again — swap production back to staging. This is instant (no warmup needed for already-warm instances) and restores the previous state atomically. Only redeploy if you have overwritten the staging slot with a third deployment since the original swap.
Trap: "Auto-swap works on both Linux and Windows App Service plans"
Auto-swap is a Windows-only feature. Linux App Service plans do not support auto-swap. If a question asks how to automatically promote a deployment to production on a Linux App Service, the correct approach is to use a CI/CD pipeline (e.g., GitHub Actions, Azure Pipelines) that performs the swap step, not the built-in auto-swap feature.
Trap: "Traffic routing to a slot changes the slot's own configuration"
Traffic routing only affects which slot receives incoming requests — it does not modify any slot's settings, code, or configuration. The routing percentage can be set without any swap taking place. Users are pinned to a slot via cookie, but both slots continue running their respective configurations independently. Routing is purely a load-distribution feature for A/B testing, not a deployment mechanism.
⑥ Practice Q&A
1
Your App Service app on Standard S1 has a production slot and a staging slot. You swap staging to production. Five minutes later you discover a critical bug in the newly deployed code. What is the fastest way to restore the previous production version?
▼
✓ Correct Answer: Perform another swap — swap production back to staging
The previous production code is still running warm in the staging slot.
  • After the initial swap, staging contains the old production code (the good version).
  • Swapping again (production → staging) is near-instant because staging instances are already warm — no container restart or warmup cycle needed.
  • This is faster and safer than a redeployment, which requires a full pipeline run and a fresh warmup.
  • The swap is atomic — traffic is rerouted instantly with no downtime.
2
A team wants to test a new feature with 10% of production users before full rollout, without performing a swap. The new feature is deployed to a "canary" slot. What Azure App Service feature enables this?
▼
✓ Correct Answer: Traffic routing (percentage-based routing to a deployment slot)
App Service supports routing a configurable percentage of traffic to any non-production slot.
  • Set routing for the canary slot to 10% — 90% of users continue to hit production, 10% hit canary.
  • Users are assigned via a session cookie (x-ms-routing-name), ensuring the same user consistently hits the same slot during their session.
  • This is configured in the portal (Deployment slots → Traffic routing) or via CLI: az webapp traffic-routing set --distribution canary=10.
  • When ready to go full production, remove the traffic routing and perform a swap.
3
Your staging slot's warmup path (/health) is set via WEBSITE_SWAP_WARMUP_PING_PATH. During a swap, the staging app takes 2 minutes to warm up, but the swap fails after 90 seconds. What setting should you adjust?
▼
✓ Correct Answer: Increase WEBSITE_SWAP_WARMUP_PING_WAIT beyond 90 seconds
The default swap warmup timeout is 90 seconds — swaps abort if the warmup ping doesn't respond in time.
  • Set WEBSITE_SWAP_WARMUP_PING_WAIT=180 (or higher) to give the app 3 minutes to warm up.
  • This setting is in seconds and should be set on the production slot (it controls the platform's wait behavior during swaps targeting production).
  • Also consider optimizing the app's startup time — loading large caches or initializing connections at startup is common cause of slow warmups.
  • WEBSITE_SWAP_WARMUP_PING_PATH defines what to ping; WEBSITE_SWAP_WARMUP_PING_WAIT defines how long to wait for a response.
4
A developer needs to deploy and test code in a staging environment before promoting to production. The web app runs on App Service Basic B2. They cannot find the deployment slots option in the portal. What is the root cause?
▼
✓ Correct Answer: Basic tier does not support deployment slots — upgrade to Standard S1 minimum
Deployment slots require Standard tier or above.
  • The Basic tier (B1–B3) provides dedicated VMs, custom SSL, and custom domains, but has zero deployment slots.
  • To enable staging slots, upgrade the plan to at least Standard S1.
  • Standard S1 provides 5 slots total (production + 4 staging/test slots).
  • This is a frequently tested boundary: Basic is not sufficient for slots despite being a dedicated-VM tier.
5
Your production slot has SENDGRID_API_KEY marked as slot-sticky. Your staging slot does NOT have this setting defined at all. After a swap (staging → production), what value does SENDGRID_API_KEY have in the new production slot?
▼
✓ Correct Answer: The original production value — slot-sticky settings remain in production
Slot-sticky settings are anchored to the slot, not the code.
  • The production slot's sticky SENDGRID_API_KEY does not move — it stays in production regardless of which code revision is running there.
  • The new production slot (previously staging code) inherits the production slot's sticky settings, including SENDGRID_API_KEY.
  • After the swap, the staging slot (now running old production code) does NOT have SENDGRID_API_KEY — it was only defined on the production slot as sticky, so it doesn't appear on staging.
  • This is intentional — it ensures production credentials are never exposed to the staging environment.
6
You want every push to the staging slot to automatically deploy to production after a successful warmup. The app runs on a Linux App Service plan (Standard S2). Can you use the Auto-swap feature?
▼
✓ Correct Answer: No — Auto-swap is not supported on Linux App Service plans
Auto-swap is a Windows-only App Service feature.
  • Linux App Service plans do not support the Auto-swap slot configuration.
  • The option is not visible in the portal and the CLI command will return an error for Linux plans.
  • Alternative for Linux: configure your CI/CD pipeline (GitHub Actions, Azure DevOps, etc.) to explicitly call az webapp deployment slot swap after a successful deployment to staging.
  • This achieves the same behavior (deploy → warmup → promote) with slightly more pipeline configuration, but is fully supported on Linux.
7
You have a Standard S1 plan with production and three staging slots already created. A new requirement needs a fourth staging slot and also a load testing slot. What is the minimum action to accommodate both?
▼
✓ Correct Answer: Upgrade the plan to Premium v3 (P1v3 or higher)
Standard tier allows only 5 slots total (1 production + 4 non-production).
  • You currently have: 1 production + 3 staging = 4 total slots. Adding 2 more (staging4 + loadtest) requires 6 total slots.
  • Standard S1 cap is 5 — you can add one more, not two.
  • Upgrading to Premium v3 increases the limit to 20 slots — accommodates the fifth and sixth slot and any future needs.
  • The upgrade does not affect existing slots or require redeployment; it changes the plan SKU and available resources.
